# Simulation (`src/simulation/`)

Agent AI, factions, knowledge, hunting, typed-task pipeline, pluralist economy. See root `CLAUDE.md` for cross-cutting `SimulationSet` ordering and tile/Z conventions.

## Game speed (`speed.rs`)

`GameSpeed { current, last_unpaused }` is the player-facing state; `SpeedPreset::{Paused, Normal, Fast (2×), VeryFast (5×)}`. `sync_game_speed_to_virtual_time` (PreUpdate, change-detected) mirrors it onto `Time<Virtual>`. Higher presets make FixedUpdate fire more often per real second — every per-tick decay, cooldown, calendar advance, and `tick % CADENCE == 0` Economy system scales naturally. `SimClock.scale_factor()` carries bucket compensation (`population / bucket_size`) only. `handle_speed_keybinds_system` maps `Space` → pause toggle and `1/2/3` → presets, egui-focus-gated. Per-preset CPU budgets via `SpeedPreset::budget_ms_per_tick()`.

## Background work budgets (`perf.rs`, `world_sim.rs`)

`PerfWorkBudget` centralises main-thread apply caps for background/budgeted systems: chunk data loads 24/tick, camera sprite chunk loads 8/tick, chunk unload despawns 32/tick, graph classify chunks 16/task, tile refreshes 512/tick, faction plan applies 4/tick, settlement plan applies 2/tick. `BackgroundWorkDiagnostics` is the shared debug surface shown in the Debug panel's "Background Work" and expanded "Pathing Debug" sections.

`world_sim_system` no longer scans the full `512 × 256` globe on `tick % 60 == 0`. `WorldSimCursor` walks about one-sixtieth of the globe per FixedUpdate, snapshots unloaded claimed cells plus the loaded-cell mask, and computes food/population/raid intents on `AsyncComputeTaskPool`. `WorldSimTaskState` queues returned cell deltas and raids; the main thread applies a bounded delta batch, validating faction id before writing live `Globe` cells. Stale task results are dropped if `WorldSimCursor.generation` has advanced.

## Agent AI (Goals → HTN → Tasks)

End-to-end through HTN.

- **Goals (`goals.rs`):** high-level objectives (`Survive`, `GatherWood`, `GatherStone`, `GatherFood`, `Haul`, `Defend`, `Farm`, `Build`, `Socialize`, `Sleep`, `MigrateToCamp`, …).
- **HTN registry (`htn.rs`):** per-goal decomposition. Methods carry `precondition` / `utility` / `expand(ctx) -> Vec<Task>`; dispatcher argmaxes and routes the head, prefetching the tail on `ActionQueue`.
- **Tasks (`tasks.rs`):** the agent's *current* action (`TaskKind` enum). `Idle` between tasks.
- **`Disposition { entrepreneurial, gregariousness, curiosity, martial }`** (`u8`, scattered at spawn) drives per-agent preferences.
- **`GoalClass`** (`Survival > Subsistence > Safety > Belonging > Esteem > Enterprise > Discretionary`, `Ord`-derived) tiers goals so registry argmax picks `Survival → Sleep` over `Enterprise → Craft` regardless of raw score.
- **`GoalScorer` trait + `GoalScorerRegistry` resource**: pluggable scorers (`SurvivalHunger`, `Sleep`, `ReturnSurplus`, `Social`, `Play`, `TameAnimal`, `PersonalBuild`, `CraftDemand`, `Stockpile`, `HealNeed`, `ProvideCare`, `EarnIncome`, …). `register_default_scorers` runs at `SimulationPlugin::build`. Continuous utility curves (`utility_curves.rs`) anchored on legacy `HUNGER_*` / `SLEEP_*` cliffs. Per-scorer disposition lifts live in one tunables block at the top of `goal_scorers.rs`. **Invariant — a survival-maintenance scorer must not emit a goal below its dispatcher's executable gate**, or it pins the agent on an indecomposable, work-preempting goal: `SurvivalHungerScorer` / `fallback_pick` gate personal `Survive` on `EAT_TRIGGER_HUNGER` (180) — the same threshold every food HTN precondition uses; `ThirstScorer` likewise gates on `THIRST_TRIGGER`. Sub-180 hunger prep is faction/economy scope (`GatherFood` / `Stockpile` postings), not personal `Survive`. `goals.rs` `HUNGER_*` constants all derive from `HUNGER_ACT = EAT_TRIGGER_HUNGER as f32` so goal scoring and HTN gates cannot drift.
- **`goal_update_system` is the sole goal-selection path.** Builds `GoalScoringContext` from precomputed gates + `Calendar::time_phase()` and calls `registry.best_with_incumbent(...)`. Hysteresis: `GOAL_CHALLENGER_MARGIN = 0.10` dampens single-tick flips. `interrupt_policy_allows` gates replacement. Legacy imperative cascade survives as `fallback_pick` for SOLO / fixtures.
- **Goal/dispatcher contract (`goal_contract.rs`).** The survival-scorer invariant above generalises to **every** scorer: a scorer may only emit a goal whose HTN dispatcher can decompose it into a task *now*. A broad gate (e.g. "a tameable animal exists anywhere") selects an indecomposable goal; the dispatcher silently `continue`s; the agent sits `Idle`; idle agents bypass the 200-tick cadence so the same goal re-fires every tick — a permanent idle loop, invisible because no method ran so no `MethodHistory` failure was recorded. `goal_contract.rs` centralises the scan-radius constants (`TAME_SEARCH_RADIUS`, `SOCIAL_PARTNER_RADIUS`, `PLAY_PARTNER_RADIUS`, `PLAY_ITEM_RADIUS`, re-exported `HEAL_SCAN_RADIUS` / `DRINK_*`) so a scorer gate and its dispatcher scan cannot drift. `has_tameable_animal` is now gated on a candidate within `TAME_SEARCH_RADIUS` of the agent, not a global existence check. Per-scorer executable gates extend this: `GoalScoringContext.{has_local_care_patient, has_social_partner}` are computed in `goal_update_system`'s scored build site — `has_local_care_patient` from radius-filtered `CareNeed` opportunities + injured-agent positions within `HEAL_SCAN_RADIUS`; `has_social_partner` from a box sweep over a **fresh per-tile `Person`-count grid** (built same-tick from live `Transform`s, not `SpatialIndex` — a cold index would otherwise flip a freshly-spawned `Socialize` agent off its goal on tick 1), bounded behind the social urgency threshold. `ProvideCareScorer` / `SocialScorer` decline when their gate is false. `Play` deliberately has **no** scorer gate — its solo fallbacks (stone-throw / seed-plant / toy) make a cheap gate false-negative-prone and a faithful one a 5-way scan duplicating `htn_play_dispatch_system`; instead every `htn_play_dispatch_system` no-task exit (including the primary "no play option" `continue`) records the `goal_contract::blocked` backstop. `Craft` is gated by `should_craft` (recipe-input availability); `Build` is deliberately ungated (a player-commissioned blueprint with no materials is built via the gather path — gating would strand it).
- **No-task backstop (`goal_contract.rs`).** Defence in depth: when a dispatcher selects a goal but produces no task it records a *throttled* synthetic `MethodHistory` failure via `record_no_task_backstop` (≥`BACKSTOP_THROTTLE_TICKS=20` apart). After 3 such entries within the history TTL, `chronic_failure_release_system` cooldowns the goal + forces re-evaluation — so a missed/stale scorer gate still recovers. `MethodId::UNKNOWN` entries are filtered out of `recently_failed_count` (no method-scoring poison) but DO count toward `chronic_failure_release`'s plain non-success tally — the channel the backstop uses. Single-dispatcher goals (`TameAnimal`/`ProvideCare`/`Socialize`/`Play`) call `goal_contract::blocked(..)` inline at *every* dispatcher no-task `continue` (including `htn_play_dispatch_system`'s primary "no play option" exit) (also emits `#[cfg(debug_assertions)]` `target:"goal_contract"` logging with a `BlockedReason`). Multi-dispatcher goals (`Craft`/`Build`/`Farm`) are covered by `goal_contract_backstop_system` (head of `Sequential`, after every ParallelB dispatcher) since no single dispatcher can tell "I can't serve" from "a sibling will".
- **Cooperative maintenance for durable work:** `JobClaim` and `ScoutAssignment` keep active episodes intact, but when the agent is idle `goal_update_system` runs a maintenance-only scorer pass (`Survive` / `Drink` / `Sleep` / `SeekCare`). Maintenance preserves the claim/assignment; when it stops scoring, the claimed job or scout goal is restored. `Socialize` / `Play` still use the separate opportunistic cancel-and-replan path.
- **`ScorerInputs` SystemParam** bundles registry, `JobBoard`, `OpportunityIndex`, `DecisionMetrics`, per-agent queries to stay under Bevy's 16-param ceiling.
- **Opportunity index (`opportunity.rs`)** rebuilt every 20 ticks in Economy after `compute_faction_storage_system`. Producers emit `PaidJob` / `MaterialDeficit` / `FoodSource` / `CareNeed`. `EarnIncomeScorer` and `ProvideCareScorer` read it first, with direct fallback.
- **Opportunistic en-route interrupts (`opportunistic.rs`):** scorers opt in via `GoalScorer::opportunistic()`. `opportunistic_interrupt_system` (ParallelA, 20-tick cadence) flips the goal when an eligible scorer scores above `OPPORTUNISTIC_INTERRUPT_THRESHOLD (0.50)` and the prior goal isn't on `GoalCooldown`. Skips `MF_UNINTERRUPTIBLE` chains and `JobClaim`-locked agents.
- **HTN disposition lift:** `Method::disposition_lift(...)` multiplies utility (cap 1.3 to stay below tier breakpoints). Overridden by Socialize / Hunt / Play methods. `dispatch_for_goal(...)` is the shared argmax helper.
- **Goal-update cadence:** active agents re-evaluate every 200 ticks; idle agents bypass the gate. Target validation is narrow — only despawned `target_entity` invalidates. Stage/kind drift on a live target is caught at arrival.

### Healing pipeline (`medicine.rs`)

- **`Injury { severity, applied_tick, last_damage_tick }`** derived reactively from `Health.current < Health.max` by `injury_tracking_system` (observer on `Changed<Health>`); clears at full restore.
- **Goals:** `AgentGoal::{SeekCare, ProvideCare}`. `HealNeedScorer` is **Survival** (Safety wouldn't preempt routine craft); `ProvideCareScorer` (Subsistence 0.65) fires for Healer/Apprentice when any same-faction injured exists.
- **Provider chain:** `htn_provide_care_dispatch_system` (ParallelB) routes nearest patient within `HEAL_SCAN_RADIUS = 12` → `Task::Heal { patient }`. `heal_task_system` (Sequential, after combat) requires chebyshev ≤ 1, ticks `Injury.severity -= 1`, grants Medicine XP, removes `Injury` at zero.
- **Patient chain:** `htn_seek_care_dispatch_system` routes to nearest faction-owned `Shrine` via `WorkshopOwnership`; short-circuits within `SEEK_CARE_AT_SITE_RADIUS = 6`. **Intentional no-task state:** a patient already inside `SEEK_CARE_AT_SITE_RADIUS` of a care site holds `AgentGoal::SeekCare` with an empty `ActionQueue` — it is *parked*, waiting for a Healer to arrive. This is a legitimate no-task state, not a contract violation: `SeekCare` is a maintenance goal (exempt from `chronic_failure_release`) and is not covered by the `goal_contract` backstop.
- **`htn_provide_care_dispatch_system`** no-patient `continue` carries the `goal_contract::blocked` backstop (`BlockedReason::NoCarePatient`) — a Healer who selected `ProvideCare` but has no patient within `HEAL_SCAN_RADIUS` recovers via chronic-failure release rather than looping idle.
- **Chief auto-promotion:** `chief_healer_assignment_system` (Economy, daily quarter) mirrors `chief_craft_assignment_system`; target is injury-driven (`HEALER_PER_INJURY_DIVISOR=4`, capped by `HEALER_MAX_DIVISOR=3`). Sub-threshold Medicine routes through `ApprenticeProgress { target_profession: Healer }`.

### Thirst pipeline (`drink.rs` + `sanitation.rs` + `medicine.rs`)

`Needs.thirst` decays at ~2× hunger rate, gated by `THIRST_TRIGGER` / `THIRST_SEVERE`. Never damages `Health`.

- **`ThirstScorer`** (Survival) fires `AgentGoal::Drink`; `htn_drink_dispatch_system` tries inventory `clean_water` first, then ranked routed candidates: nearest local **water-bearing** `WellMap` entry, nearest local fresh tile, nearest home/camp wet well, nearest home/camp fresh tile. The home/camp fallback scans `DRINK_HOME_SCAN_RADIUS=32` around `FactionData.home_tile` only for settled factions or nomads with `CampState::Pitched`, so travelling bands don't path back to old camps. `nearest_well_tile` skips dry wells via `drink::well_has_water`; `nearest_fresh_drinkable_tile` skips Salt **and Brackish** via `world::biome::water_kind_at(...).is_drinkable()`; severe thirst widens scans. **Phase 6 salinity:** `WaterKind::{Fresh,Brackish,Salt}` from `Globe::salinity_at` (Endorheic basins brackish 0.6 → non-drinkable; ocean salt; rivers/open lakes fresh); `is_drinkable()` = Fresh-only is the single rejection rule (drink + animal water-seek). **Physical wells (`well.rs`):** `WELL_REACH_Z`/`well_reaches` are **gone** — a well's water is the live `RuntimeWater` column at its central shaft tile. `well_water_column(runtime_water, shaft)` reads the depth; `well_has_water` = column `≥ well::WELL_MIN_DRINKABLE_DEPTH (0.3)`. `perform_drink` draws the column down `WELL_SIP_DRAWDOWN_Z (0.02)` per sip (the cell is never removed — its `AQUIFER_SEEP_RATE` source recharges it); a column below the drinkable floor reads graceful `DrinkOutcome::WellDry` (re-meaning'd: "shaft empty", not "table unreachable") and refills. An all-dry-well village still reads "fresh water far" so `WaterAccess` pressure responds.
- **`drink_task_system`** consumes from hands or sips an adjacent fresh tile / well; raw drinks bump `Sickness.severity` 60, contaminated tiles 140 (well tiles read `SanitationMap::is_contaminated` exactly like river tiles, so wells without a nearby `Latrine` get tainted by `WastePile`). Multi-sip loop mirroring `eat_task_system`: drinks until `thirst ≤ DRINK_SATIETY_FLOOR (40)`, source exhausted, or `MAX_SIPS_PER_ACTION (4)` reached — one dispatch fully quenches the agent. Sickness severity scales with sips taken. `ThirstScorer` uses `utility_curves::thirst_utility` (two-stage smoothstep, weights 0.60/0.40) so urgency at `THIRST_TRIGGER` ≈ 0.60 — at-or-above hunger's curve.
- **Physically-excavated wells (`well.rs`)** — `BuildSiteKind::Well` (right-click → Build Well, `WELL_DIGGING` Neolithic) is a *staged* structure, not a one-shot finalize. `convert_well_blueprint_system` (Sequential, before `construction_system`) turns a placed Well blueprint into a `WellSite`: resolve depth from the aquifer via `well_spec_at` (`aquifer_z_at` shared frame; shaft dug to one Z below the table, capped `MAX_HAND_DUG_WELL_DEPTH_Z=8` — deeper aquifers reject as `TooDeep`). Geometry is a 5×5 stepwell — centre water-shaft column, an 8-tile descending helix of carved `Dirt` floors (workers descend it; A* routes `|Δz|≤1` steps with no `Ramp` tiles needed), and a 16-tile outer rim of lining `Wall` blueprints (one gateway gap). `well_site_progression_system` (Sequential, after `construction_system`) drives `WellPhase::{Excavating → Lining → Capping}`: excavation reuses `TerraformSite`/`terraform_system` (one per descending footprint tile); on completion it charges the `RuntimeWater` column (`WELL_INITIAL_CHARGE_Z`) and spawns the lining walls; when those finalize it spawns the finished `Well { faction_id, shaft_tile, bottom_z, surf_z }` in `WellMap`. `DrinkSource::Well { tile }` (`tile` = shaft/`WellMap` key). `organic_settlement::SettlementPressureKind::WaterAccess` queues wells when local fresh water is far or population > 40 / 90. **Footprint validation:** `convert_well_blueprint_system` rejects (despawns) a Well blueprint whose 5×5 `well_footprint` overlaps any `StructureIndex` / `WellMap` / `WellSiteMap` / other `BlueprintMap` entry — manual/organic placement only vets the centre tile, so the conversion chokepoint is where overlap is caught. **Carved-geometry persistence:** carved Air/floor tiles are chunk deltas not re-applied on regen (only `Wall` entities + `RuntimeWater` survive), so a well footprint straddling a chunk boundary would lose its helix on unload. `restamp_wells_on_chunk_load` (FixedUpdate, after `chunk_streaming_system`, chained before `restamp_runtime_water_on_chunk_load`) re-carves any `Well` / completed-`WellSite` geometry intersecting a freshly-loaded chunk — `Well` (`surf_z` + `bottom_z`) is the durable truth, via the shared `carve_well_geometry` helper. Seed wells (`spawn_seeded_structure_at_tile`) skip the worker pipeline — stamp the `Well` + charge `RuntimeWater` directly; the OnEnter `carve_seeded_wells_system` then carves their visible shaft + helix and projects the water column (depth stays a fixed shallow `surf - 3` seed simplification).
- **Boiling**: `2 raw_water + 1 wood → 1 clean_water` (CraftRecipe 13, `FIRE_MAKING`, Workbench).
- **`SanitationMap`**: sparse `(tile → f32)` populated daily from `WastePile` via `1/(d²+1)` falloff in `CONTAMINATION_RADIUS = 6`; decays `2^(-1/4)`/day; `is_water_contaminated` above `0.5`. `WastePile` emerges via `agent_defecation_system` (one per `DEFECATION_INTERVAL_TICKS = 2 days`, staggered by entity index). `Latrine` within `LATRINE_ROUTING_RADIUS = 8` marks the pile `LatrineContained`. `BuildSiteKind::Latrine` (right-click → Build Latrine, 2 wood + 1 stone).
- **`Sickness`** (orthogonal to `Injury`): daily decay `-16`; `sickness_work_factor(severity) → [0.5, 1.0]` multiplies movement-driven work progress.
- **Animals:** `AnimalNeeds.thirst`, `AnimalState::Drinking`, `animal_water_seek_system` + `animal_drink_system`.

### Needs

7 needs (hunger, thirst, sleep, shelter, safety, social, reproduction), `[0,255]`, decay over time. Maslow tiers (`Physiological → Safety → Belonging → Esteem → SelfActualization`) layer on top via `MaslowTier::next_unmet(needs)`. Decay rates anchored on the 180-sec game-day: `HUNGER_RATE = 2.0/s`, `SLEEP_RATE = 1.2/s` paired with `SLEEP_RECOVER_RATE = 6.0/s` (12.0 on a bed). `WILLPOWER_WORK_DRAIN = 1.8/s`, `WILLPOWER_IDLE_DRAIN = 0.15/s`. `con_scale` (Constitution) attenuates hunger+sleep decay (floor 0.25×). `tick_needs_system` only *accrues* the sleep need (when not `Sleeping`); the **`sleep::sleep_task_system`** executor (Sequential) owns sleep *recovery* + *retirement* keyed on the typed `Task::Sleep` (not on `ai.state`) — see ActionQueue rules below.

### Energy (`energy.rs`)

`Energy { current, max, exhausted }` — a physiological exertion/fatigue pool, **separate from `Needs`** (it must NOT feed `worst()`/`avg_distress()` — physiology, not morale). 0–255 float; `default()` is full. Spawned on every Person (`person.rs` founder bundle, `reproduction.rs` newborn, `test_fixture.rs` `PersonBuilder` with `.energy(..)` override). Thresholds with hysteresis: `ENERGY_EXHAUSTED=40` (flips the stateful `exhausted` flag), `ENERGY_TIRED=90`, `ENERGY_RECOVERED=140` (only crossing this clears `exhausted`). `energy_factor()` ramps 1.0→`ENERGY_FACTOR_FLOOR=0.45` below `ENERGY_TIRED` (mirrors `medicine::sickness_work_factor`).

- **Drain:** movement (`movement_system`, per tile, scaled by terrain effort + mounted ×0.15), labor (`energy_tick_system`, ParallelA after `tick_needs_system`, while `AiState::Working`), combat (`combat_system`, per swing; a *tired* attacker's cooldown ×1.3).
- **Recovery:** sleep (`sleep_task_system`, same bed ×2 multiplier as sleep/willpower), idle (`energy_tick_system`, while `AiState::Idle`).
- **Effects:** movement speed and work-progress accrual both ×`energy_factor()` (alongside the sickness factor, in `movement_system`). **Exhaustion goal gate:** `GoalScoringContext.is_exhausted` (set from the `energy_q` in `ScorerInputs`); `GoalScorerRegistry::best_with_incumbent` drops every non-`Survival`/`Safety` candidate while it holds, so an exhausted agent sheds noncritical labor and idles (recovering) until `RECOVERED`. Eat/drink/sleep/heal/flee/defend stay ungated.

### Swimming (`swimming.rs`)

Amphibious *pathfinding* (see `pathfinding/CLAUDE.md` → Amphibious traversal) lets a human plan a route across water; `swimming.rs` owns what happens **while** wet. `SwimmingState { wet_ticks, exhausted_ticks, last_xp_tick, last_safe_tile }` is attached on water entry and removed on bank arrival by `swimming_system` (Sequential, after `movement_system` — reads the post-move tile). While wet: heavy `Energy` drain (`SWIM_ENERGY_DRAIN`), periodic `SkillKind::Swimming` XP. **Fatigue-first risk:** drowning (`Body` torso damage) begins only once an *exhausted* swimmer has been in **deep water** (`depth ≥ DEEP_WATER_DEPTH`) past `DROWN_GRACE_TICKS`. `SkillKind::Swimming = 9` is innate (not tech-gated). **Currents:** `swimming_system` reads `world::water_current::WaterCurrentField` (`Option<Res>` — absent in the headless fixture); swimming into a river current scales energy drain up to +50%, resisting a meaningful current grants bonus XP. Emergency bank-retreat retarget for exhausted swimmers is deferred (v1 — `last_safe_tile` is tracked but not yet consumed).

### Ambient work-social multitasking (`social_contact.rs`)

Workers doing compatible primary work near a same-root-faction coworker gain *secondary* social interaction without leaving their job. `SecondarySocial { partner: Option<Entity>, mode: Ambient, expires_tick }` is **spawned on every Person** (`inactive()` default) and only ever *mutated* — never insert/removed — so the Person archetype never churns (construction/seeding layout is sensitive to Person `Query` iteration order). `ambient_social_pairing_system` (ParallelA, after `tick_needs_system`, before `goal_update_system`/`opportunistic_interrupt_system`): per compatible agent, on a per-entity-staggered `PAIRING_RESCAN_CADENCE` it picks the deterministic nearest (`(chebyshev, entity-bits)` min) compatible coworker within `SOCIAL_RADIUS` and stamps `expires_tick = now + PAIRING_WINDOW (400)`; off-cadence it clears a now-invalid pairing. `is_ambient_social_compatible` rejects SOLO/Dormant/Drafted/CombatTarget/Raid/Rescue/Migrate/Pack/maintenance/dedicated-Socialize; accepts the durable-work goal set (Gather*/Build/Craft/Farm/Haul/Stockpile), `AiState` deliberately not gated.

Shared gate **`is_social_contact`** (dedicated `Socialize` **OR** live `SecondarySocial`, non-Dormant) is the "any social channel" predicate used by every refactored consumer.

| Consumer | Ambient behaviour |
|---|---|
| `faction::social_fill_system` (social-need relief) | yes — **tightened**: relief requires being socially active AND counts only same-root-faction `Person` neighbors (was: any indexed entity incl. animals/items, ungated) |
| `knowledge::awareness_gossip_system` (tech/settlement awareness) | yes — full strength, all neighbors |
| `knowledge::cluster_tier_promotion_system` (SharedKnowledge tier bubble) | yes — full strength, all neighbors |
| `jobs::wage_gossip_system` (wage EMA) | yes — full strength, all neighbors |
| `memory::conversation_memory_system` (relationship affinity) | yes — **two-tier rate**: dedicated Socialize pair → `DEDICATED_AFFINITY_STEP=5`/tick uncapped; purely-ambient pair → `AMBIENT_AFFINITY_STEP=1`/tick capped at `AMBIENT_AFFINITY_CAP=40` (a pair is "dedicated" if *either* end is on explicit `Socialize`) |
| `knowledge::tech_teaching_system` (mastery transfer) | **no — deliberate-only** (own `Socialize` check; casual chatter ≠ instruction) |

**Why the affinity cap (`memory.rs`):** `RelationshipMemory` affinity feeds the cohabitation / bed-assignment logic in `construction.rs` (`best_partner_bed_for`, thresholds `PARTNER_AFFINITY_THRESHOLD=60` / `REASSIGN=80`). At the dedicated +5/tick rate, ambient work crews would saturate affinity within ~a game-minute and "move in together" en masse (a 5-bed reassignment spike by tick ~60 → standalone Bed blueprints, caught by `onenter_era_seeding::neolithic_runtime_no_paleo_beds_or_excess_hearths`). `relationship_decay` is daily (every 3600 ticks) and far too weak to bound monotonic per-tick accrual, so a *reduced rate alone only delays* the spike — the **cap below 60** is what keeps ambient bonds at "acquaintance"; crossing into courtship/cohabitation requires deliberate `Socialize` (which also preserves a real incentive to Socialize). Detour suppression: `opportunistic_interrupt_system` skips the `Socialize` challenger while a live `SecondarySocial` exists; the `social_fill` ambient drain keeps `needs.social` low so `SocialScorer` / the legacy cascade don't fire either.

## Method-design rules

- **Scoring = viability + distance, with time-of-day/fatigue scaling for AcquireFood / AcquireGood{wood,stone}.** Base tiers: `UTIL_BASELINE=1.0` / `UTIL_VISIBLE_GROUND=1.5` / `UTIL_CLAIMED_HAUL=2.0` / `UTIL_EXPLORE_FALLBACK=0.3`. Chebyshev distance penalty capped at `MAX_DIST_PENALTY=0.30`. Recent failures bias via `MethodHistory` + `score_method_with_history` (`METHOD_FAILURE_PENALTY=0.4`, ring len 6, TTL 600 ticks). Two stacked failures push a baseline method below the Explore fallback. **Context-aware penalty:** `ScoringScope::ContextAware { time_phase, dusk_remaining, fatigue }` multiplies the geometric value by time-of-day (Day 1.0 / Dawn 1.10 / Dusk ramp / Night 4.0) and fatigue (`1.0 + fatigue`). Cap rises to `MAX_DIST_PENALTY_NIGHT=1.50` only at Night.
- **AcquireFood concrete-first partition (`plans/food-pickup.md`).** `htn_acquire_food_dispatch_system` argmaxes over **concrete** food methods first (Withdraw / Scavenge / Forage / Fish), and only falls through to `ExploreForFoodMethod` when no concrete's precondition passes. Failure-bias still reorders among concretes (e.g. depleted fish spot loses to forage via `MethodHistory`), but a heavily-biased concrete can no longer fall below `UTIL_EXPLORE_FALLBACK` and let Explore win against a live storage/scavenge target. All four food concretes sit at `UTIL_BASELINE = 1.0 − dist_penalty` — `ScavengeFoodFromGroundMethod`'s legacy `UTIL_VISIBLE_GROUND` premium was dropped once preconditions read live world state (live `SpatialIndex + GroundItem` storage scan; live `SpatialIndex` scavenge fallback when `CurrentVision` is bucket-stale). Tier baselines now encode *intent* (Withdraw/Scavenge/Forage/Fish are interchangeable means; Explore is last resort), not certainty of availability.
- **In-hand fast-path for haul dispatchers.** `htn_acquire_good_dispatch_system::Haul` and `htn_build_claimed_blueprint_dispatch_system` Path B check `Carrier + EconomicAgent` inventory before `WithdrawMaterial`; if already carrying, dispatch `HaulToBlueprint` directly. **Never count agent inventories in posting creation or chief blueprint-candidate scoring** — those layers read deposited storage only.
- **Farming is `Farm`-goal only.** Both halves HTN-driven (`PlantFromStorage`, `HarvestPlant`). Seed↔plant mapping centralised in `PlantKind::seed_resource()` + `PlantKind::ALL`.
- **`MF_UNINTERRUPTIBLE`** marks methods whose multi-leg chain must survive across goal-dispatch ticks.

## Player command authority (`player_command.rs`)

UI emits `PlayerCommandEvent` (event bus). Sim drains in `SimulationSet::Input` (exclusive), attaches `Commanded { command, status, issued_tick, command_id }`, then `dispatch_player_command_system` (ParallelB) routes the command into the typed-task pipeline. **All routing is sim-side** — UI never mutates `PersonAI` / `ActionQueue` / authority markers directly.

- **Variants:** `Move`, `Gather`, `Mine`, `Build`, `Deconstruct`, `DigDown`, `PickUpItem`, `PickUpCorpse`, `AttackEntity`, `Teach`, `HoldLecture`, `ReadItem`, `EncodeTablet`, `Muster`, `Disband`, `MilitaryMove`, `MilitaryAttack`, `PackCamp`, `PitchCamp { tile, z }`. - **Lifecycle (`player_command_lifecycle_system`):** Move/MilitaryMove → chebyshev arrival; Gather → target despawned OR agent idle; Mine → tile no longer Wall/Stone; Build → blueprint despawned; PickUp* → target gone; Attack/MilitaryAttack → foe dead. `reap_terminal_commands_system` strips `Commanded` once status is terminal; UI reads `RemovedComponents<Commanded>` for feedback. **MilitaryMove arrival reads `PersonAI.dest_tile`** (per-actor slot tile), not the click anchor.
- **Multi-unit `MilitaryMove` formations (`military/formation.rs`).** Events with `actors.len() > 1` expand into a compact ring of per-actor slot tiles. `expand_military_move_system` runs `plan_compact_ring(anchor, n, is_passable)` + `greedy_assign`; output in `PendingFormationSlots`. `dispatch_player_command_system` routes each actor to its slot + inserts `MilitaryFormationSlot { anchor, slot_index, group }`. - **Faction-level commands** (e.g. `EncodeTablet`) carry empty `actors` and apply directly in `drain_player_command_events_system`.
- **Adding a new order type:** new `PlayerCommand` variant + `dispatch_one` arm + `player_command_lifecycle_system` arm + UI button emits the event.

## ActionQueue and typed Task variants (`typed_task.rs`)

`aq.current: Task` is canonical "task running now"; `Task::Idle` default. Behind it sits a fixed `queued: [Task; 4]` prefetch ring (private; access via `enqueue` / `pop_next` / `peek_next` / `queued_len` / `clear_queued`).

- **Producers** (HTN dispatchers, `ui/orders.rs`, `teaching.rs::ReadItem`, legacy-only producers `building_upgrade_system` / `terraform_dispatch_system` / `military_task_system` reroute) route through `aq.dispatch(task)` — enqueues then promotes head into `current` if `current == Idle`. Returns `DispatchOutcome::{Promoted, Queued, Rejected}`. `Rejected` (queue full) fires `debug_assert!` inside `dispatch`; silent burial is no longer possible. Most callers ignore the outcome since `Queued` is legitimate for chain prefetch.
- **Autonomous direct dispatch** that bypasses a registry `Method` (job-driven single-leg tasks such as field preparation) must use `tasks::dispatch_autonomous_task_with_routing`. It wraps `assign_task_with_routing` + `aq.dispatch(...)` and stamps `AutonomousTaskLifecycle { owner_goal, task_kind, job_id, preserve_across_goal_dispatch }` onto the running task. `goal_dispatch_system` consumes that metadata before the legacy preserve-arm list, so these tasks survive stale-reset ticks only while their owning goal/task still matches. Explicit player/military/scripted authority paths may still dispatch manually because their own command lifecycle owns teardown.
- **Consumers** (executor exit paths in `gather.rs`, `dig.rs`, `corpse.rs`, `construction.rs`, `items.rs`, `production.rs`, `teaching.rs`, `sleep.rs`, plus `MilitaryMove` arrival) call `aq.advance()` instead of writing `current = Idle`. `sleep::sleep_task_system` owns `Task::Sleep` end-to-end: arrival `Working`→`Sleeping` flip, recovery, `finish_task` retirement when rested, and **orphan self-heal** (`cancel_chain` when `current == Sleep` but `state` is `Idle`/`Attacking` — an external preempt reset state without cancelling). `htn_dispatch_system` only does the *initial* plan+route+dispatch and returns immediately whenever `current == Sleep` (the executor owns the rest); this is why the old Sleeping/in-flight/arrival guards collapsed to one. Canonical exit helpers: `aq.finish_task(&mut ai)` (success — `state = Idle` + `work_progress = 0` + advance) and `aq.cancel_chain(&mut ai)` (chain abort — same fields + cancel). Long non-cargo work uses `goals::yield_for_maintenance_boundary` at safe slice boundaries; it clears `active_method` before dropping the current chain so HTN history records neither fake success nor failure.
- **External preempts** (`dispatch_player_command_system`, hunter demote, stale reset, `movement::release_to_idle` on path failure / stranded-Z recovery, `combat::combat_retaliation_cleanup_system` draining `CombatRetaliationStartedEvent`) call `aq.cancel()` / `aq.cancel_chain(&mut ai)`, dropping both `current` and queue. The retaliation cleanup runs in `Sequential` right after `combat_system` because `combat_system`'s `attacker_query` holds `Option<&mut ActionQueue>` mutably across iteration, so the victim's `ActionQueue` is only reachable post-iteration via the event-deferred system. **A *Sleeping* victim emits `CombatRetaliationStartedEvent` on *every* damaging hit (not only the first retaliation), regardless of whether `target_combat.0` is already set** — so a multi-attacker swarm (or a body+health damage pair in one resolution) can't skip the deferred cancel and orphan the victim's `Task::Sleep`; `sleep_task_system`'s orphan self-heal is the structural backstop if an event is ever still missed. `goal_update_system` overrides (Lead/Defend/Raid/Rescue/Scout/Migrate chief / commanded forced-flip / earnincome override) must also call `aq.cancel()` — otherwise the typed channel carries a stale task that the next dispatcher's `aq.dispatch` would silently park behind. **Debug-time enforcement:** `aq.dispatch` fires a `debug_assert!` when the incoming task shares a `TaskKind` with `current` *and* the queue is empty — the canonical stale-typed-channel pattern. Legit chain prefetch interleaves variants and always has `queued_len > 0` at that point, so the assert doesn't fire on it.
- Per-tick "pin" sites (lecture/teach pin writes) stay as direct `aq.current = X` writes — idempotent re-assertions.

**Variants:** `Idle`, `WalkTo { tile, z, why }`, `WithdrawGood { filter }`, `WithdrawMaterial { resource_id, qty }`, `WithdrawFood { tile }`, `Equip { slot, resource_id }`, `Construct { blueprint }`, `ConstructBed { blueprint }`, `Deconstruct { tile }`, `Terraform { tile }`, `MilitaryAttack { foe }`, `Gather`, `Dig`, `Scavenge`, `Read/Teach/HoldLecture/AttendLecture`, `PickUpCorpse`, `HuntPartyMuster`, `Hunt`, `HaulCorpse`, `Butcher`, `TameAnimal`, `Sleep`, `Eat`, `HaulToBlueprint`, `HaulToCraftOrder`, `WorkOnCraftOrder`, `DepositToFactionStorage`, `WalkAndTakeFromMember`, `PlayThrow`, `PlayPlant`, `Explore`, `Migrate`, `UnpitchStructure`, `UnloadCampCargo`, `PitchStructureAt`, plus `Lead`/`Defend`/`Raid`/`RescueAlly`/`Socialize`/`Play`.

**Rules:**

- Systems mutating typed task **must** include `&mut ActionQueue` alongside `&mut PersonAI`.
- Every executor exit must `aq.advance()` (success) or `aq.cancel()` (chain-drop). Prefer the bundled helpers `aq.finish_task(&mut ai)` / `aq.cancel_chain(&mut ai)`.
- `military_task_system`, `withdraw_good_task_system`, `withdraw_material_task_system`, `drink_task_system`, `sleep_task_system` fall back to `Idle` (`cancel_chain`) if `aq.current_task_kind() == X` but `aq.current` is the wrong variant — defence in depth. `sleep_task_system` extends this to the `state`-orphan case (`current == Sleep` but the agent is neither asleep nor walking toward sleep).
- **Reading the task discriminant.** All consumers read `aq.current_task_kind()` (derives `u16` from `aq.current` via `task_kind_for`). The legacy `PersonAI.task_id` mirror is gone; `UNEMPLOYED_TASK_KIND` (`u16::MAX`) is the "no task" sentinel returned for `Task::Idle`.

## HTN domain (`htn.rs`)

- **`AbstractTask` / `AbstractTaskKind`** — high-level goal a method decomposes (`Sleep`, `Eat`, `AcquireFood`, `AcquireGood { resource_id }`, `StockpileFood`, `Scout`, `EquipHuntingSpear`, `ReturnSurplus`, `TameWildAnimal`, `PlantFromStorage`, `HarvestPlant`, `ConstructBlueprint`, `Socialize`, `DeliverMaterialToCraftOrder`, `WorkOnCraftOrder`, `HarvestGrainForCraftOrder`, `Play`, `JoinHuntParty`, `EngagePrey`, `DeliverHuntKill`, plus single-step combat/faction goals).
- **`Method` trait** — `precondition` / `utility` / `expand(ctx) -> Vec<Task>` / `flags` / `tech_gate` / `profession_gate` / `policy_gate` / `name`. `policy_gate` returns `&'static [(ResourceId, RequiredFlag)]` for resource-policy gating.
- **`MethodRegistry`** built once in `SimulationPlugin::build` via `register_builtin_methods`. Read-only after init.
- **`MethodFlags`** = `u8` bitmask. **`MethodId`** = `u16` newtype with one `pub const` per registered method (stable identity for `MethodHistory` keying).
- **`MethodHistory`** — per-agent ring (len 6, TTL 600). Spawned at every Person spawn site.
- **Failure-biased scoring:** multi-method dispatchers score via `score_method_with_history(method, abstract_task, ctx, history, now) = utility - failures * 0.4`. Routing/missing-target failures push `FailedRouting` / `FailedTarget`.
- **Cancel paths record failure.** Executor `aq.cancel()` paths that drop a chain push the agent's `active_method` onto `MethodHistory` via `record_target_failure(...)` / `record_routing_failure(...)`. `MethodId::UNKNOWN` is the sentinel for cancels with cleared `active_method`. Sites: `gather::finish_gather`, `items::item_pickup_system`, `items::finish_scavenge`, `production::finish_withdraw_material`.
- **Goal-flip Abandoned (`record_abandoned_method_system`)** ParallelA, after `goal_update_system`, filtered on `Changed<AgentGoal>`. Pushes `Abandoned` on the agent's `active_method` and clears it — unless `MF_UNINTERRUPTIBLE`, in which case `goal_dispatch_system`'s preserve-arms keep the chain alive.
- **Terminal Explore fallback** (`htn_acquire_food_dispatch_system` + `htn_stockpile_food_dispatch_system`): when every precondition fails, fall through to synthetic `Task::Explore { kind: AnyEdible }` rather than returning silently.
- **Reachability-aware deposit pick** (`StorageTileMap::nearest_for_faction_reachable`) and **vision pickers** (`CurrentVision::nearest_gather_target` / `nearest_scavenge_target`) both take an `is_reachable` closure with two-pass fallback (filter, then connectivity-blind). The `source_tile` for deposits is the agent's post-pickup tile.
- **Detour-aware (river-aware) target/deposit selection.** All gather/scavenge/cluster/storage/bid selection ranks candidates by `pathfinding::detour::DetourEstimator` (chebyshev-equivalent tiles, `max(chebyshev, chunk-hop cost)`) instead of straight-line distance, so a target across a river costs the walk-around, not the straight line — workers no longer trek around a wide river to a far-bank target a closer same-bank one beats. Sites: `memory.rs` vision/scavenge pickers (now take a `dist` closure), `shared_knowledge.rs` cluster picker (`GatherKnowledge` gained `ChunkRouter`/`ChunkGraph`/`ChunkMap`; `gk.nearest_target_tile` gained an `agent_z` param), `nearest_for_faction_reachable` (gained `chunk_router`), `jobs.rs` U_bid `C_action`. Plain `nearest_for_faction` stays Manhattan (cheap accessor for SOLO/landlord/bookkeeping; not a river-detour deposit). `dist_penalty`/`full_trip_penalty` stay chebyshev — they arbitrate *between methods for an already-detour-selected target*, not target selection, and live in `Method::utility` (no router access). Drink/heal/hunt short-range scans (≤ small radius) keep chebyshev — a river detour can't occur there. The `nearest_with_cluster_filter` ring-walk lost its speculative `bs < (r+1)*CHUNK_SIZE` early-out (detour distance isn't monotone in chunk-ring radius); it now scans every ring up to `max_chunk_radius` (sparse `by_chunk`, bounded 16 — cheap).
- **Chronic-failure release** (`goals::chronic_failure_release_system`, Economy daily quarter): agents with ≥3 non-success entries within TTL get the goal stamped on `GoalCooldown` (ring of 4 with `(goal_disc, expires_tick)`). Then bifurcates: claim-holders release the claim; autonomous agents land in `ForceGoalReevaluate` (drained by `goal_update_system` next tick to bypass the 200-tick cadence). Maintenance goals are exempt — terminal Explore is the relief valve for food/sleep-style failure.
- **Success recording:** dispatchers stamp `ai.active_method = Some(method.id())` on success; `htn_method_completion_system` observes `current == Idle && queued_is_empty && active_method.is_some()` and records `Success`.

### Dispatch + chain handoffs

All dispatchers run `ParallelB` after `goal_dispatch_system`. Each gates on goal + Idle + non-Dormant + `aq.is_idle()`, builds `PlannerCtx`, runs argmax, routes the head via `assign_task_with_routing` + `aq.dispatch(...)`, prefetches the tail. SOLO/unsettled skip storage-dependent methods. Spear-arming runs *before* food dispatchers so unarmed hunters fetch a spear first.

Trailing legs of multi-task chains live in exit helpers on the head executor:

- **`production::finish_withdraw_food`** primes `Eat`.
- **`production::finish_withdraw_material`** releases reservation; routes `HaulToBlueprint` / `HaulToCraftOrder`; primes `Equip` / `PlayThrow` / `Play`.
- **`gather::finish_gather`** (5 exit sites) routes `DepositToFactionStorage` / `HaulToBlueprint` / `HaulToCraftOrder`; primes `Eat`. Takes `FinishGatherOutcome::{Completed, TargetInvalid}` — `TargetInvalid` cancels the chain so empty-handed walks don't happen.
- **`items::finish_scavenge`** is the mirror; primes `Eat`. Same `FinishGatherOutcome` parameter.
- **`drop_items_at_destination_system`** executes `TaskKind::DepositResource`, credits `JobClaim::Stockpile`, `aq.advance()`.

### Stale-reset (`goal_dispatch_system`)

Runs before HTN dispatchers. Stale `aq.current_task_kind() != UNEMPLOYED_TASK_KIND` clears with `aq.cancel()` unless `ActionQueue::autonomous_lifecycle()` says the running autonomous task still belongs to the current `(goal, task-kind)`, or a legacy preserve-arm for the pair applies. Catch-all resets `Explore` arrivals.

### Adding a method

New `AbstractTask` variant (if needed) + `Method` impl + register call in `register_builtin_methods`. Method tests in `htn::tests` exercise the trait directly without an `App`. New abstract-task *kinds* also need a per-goal dispatch system.

## Memory & gathering (`shared_knowledge.rs`)

3-tier `SharedKnowledge` map is the system of record for static-resource queries.

- **`SharedKnowledge`** keyed by `KnowledgeTier::{Household(u32), Settlement(SettlementId), Faction(u32)}`. Resources stored as `ResourceCluster` influence nodes (center + radius + LRU `representative_tiles[4]` + `estimated_count`). `report_sighting` merges within `CLUSTER_MERGE_RADIUS=8`; `report_depleted` decrements + despawns at zero. `nearest_in_tier_set(...)` walks finest-tier-first. `cluster_decay_system` drops clusters older than `CLUSTER_DECAY_TTL_TICKS=25200` daily.
- **Cluster-saturation skip:** dispatchers inject `gather_claims.cluster_is_saturated(...)` to skip clusters at `MAX_PARALLEL_GATHERERS_PER_CLUSTER = 3`. **Pressure-aware rep selection** picks the least-pressured rep slot.
- **`ResourceOwner`** — `Public | Person | Household | Settlement | Faction`. `is_accessible_to(viewer, ...)` gates harvest.
- **`LandClaim` on Plants** stamped at planting: `HouseholdMember` → Household; chief Farm posting → Faction; else → self. Wild reseed → no claim.
- **`GatherClaims` (`gather_claims.rs`)** — per-`(tile, kind)` reservation. `pressure(tile, now, viewer)` feeds `claim_penalty` ×4 weight. Staked at dispatch; released at every `gather::finish_gather` exit and stale-cancel. `CurrentVision::nearest_gather_target` consumes the same weight.
- **Vision write-through (`memory.rs::vision_system`)** — active agents report plant/item/prey/stone sightings to the finest tier (Household if member, else Faction). Settlement/Faction tiers materialise via gossip. Vision is additive only — `report_depleted` fires from gather-arrival, never vision.
- **`CurrentVision` vision-first dispatch** — `vision_system` populates `CurrentVision { entries: Vec<VisionEntry> }` (cleared per bucket-active pass). Food ground items dual-write under `AnyEdible` + `Resource(rid)`. Resource-target dispatchers consult `nearest_gather_target` / `nearest_scavenge_target` before `gk.nearest_target_tile`. `htn_acquire_food_dispatch_system` adds a **live `SpatialIndex` scavenge fallback** when `CurrentVision::nearest_scavenge_target` returns `None` (bucket-cadence vision misalignment): scans within `STANDARD_VIEW_RADIUS` for `IndexedKind::GroundItem` with edible+qty>0, applies storage-tile exclusion + reachability + detour-distance ranking (mirrors the live storage scan's filter set; LOS deliberately omitted, same as the storage scan — next vision bucket overwrites with a proper LOS-checked target).
- **Stale-target neighbor-scan retarget (`gather.rs`):** when `gather_system` arrives to find the plant gone, `retarget_neighbor` scans chebyshev `P6B_RETARGET_RADIUS = 2` for a same-kind mature plant; one swap per `P6B_RETARGET_COOLDOWN = 40` ticks.
- **HTN under-foot fast path** (`plants.rs::nearest_mature_plant_under_agent`): probes `PlantMap` directly within radius 2, gated on `pressure == 0`. Wired into `htn_acquire_food_dispatch_system` and `htn_stockpile_food_dispatch_system`.
- **LOS (`line_of_sight.rs`)** — 3D Bresenham at eye height `current_z + 1`. Two callers: `has_los` for combat / sound / projectile (own walls still defend); `has_vision_los(chunk_map, wall_map, door_map, wall_q, from, to, observer_faction)` for fog + memory (own constructed `Wall` and own `Door` are transparent regardless of door state, so a player-walled courtyard isn't blacked out from outside; natural rock + enemy walls still block).
- **Vision sources (`vision.rs`, `plans/lookout-base.md`)** — three radii ride one helper: per-agent `STANDARD_VIEW_RADIUS=15`, active lookouts `LOOKOUT_VIEW_RADIUS=50`, era-scaled base reveal `base_vision_radius_for_era(Era)` returning Paleo 16 / Meso 20 / Neo 25 / Chalco 30 / Bronze 40. `effective_vision_radius(active_lookout)` is the shared per-agent selector consumed by both `rendering::fog::fog_update_system` and `memory::vision_system`. **`ActiveLookout { anchor_tile, anchor_z, radius, expires_tick: Option<u64> }`** marks a Person currently lookout-pinned: `expires_tick: None` is manual / indefinite (via `PlayerCommand::Lookout { tile, z }` from the right-click "Lookout here" menu); `Some(t)` is the autonomous Hunter scout pause (`autonomous_scout_lookout_pause_system` converts a Hunter's `Task::Explore` arrival into `Task::Lookout { expires_tick: now + AUTONOMOUS_LOOKOUT_TICKS=120 }`, preserved by `goal_dispatch_system` arms for `Survive|GatherFood + Lookout`). Executor `lookout_task_system` (Sequential, after `movement_system`) pins the agent at the anchor + attaches `ActiveLookout` + `CachedVisionSet` on arrival, finishes the task on expiry; `prune_active_lookouts_system` removes both components when the agent drifts off the anchor. **Static-source caching** — settlements, camps, and active lookouts carry a `CachedVisionSet { tiles, dirty, faction, radius, origin }` so the LOS raycast runs once per source and the result is reused every fog tick. `attach_base_vision_caches_system` attaches the cache to fresh player-faction Settlement/Camp anchors (idempotent). `invalidate_vision_caches_system` reads `TileChangedEvent` / `WallConstructed` / `WallDestroyed` and marks any cache whose bounding box contains the changed tile dirty; era advance is folded in by comparing `base_vision_radius_for_era(current_era(faction.buildable_techs))` against `cache.radius`. `recompute_dirty_vision_sets_system` raycasts each dirty source once via `compute_vision_set`. `fog_update_system` (Update) unions every player-faction `CachedVisionSet.tiles` into `FogMap.visible` — pure set-union, no LOS work. `Wall.owner_faction: Option<u32>` (None = natural bedrock from chunk streaming; Some = constructed) drives the faction-aware LOS check; constructed wall finalize fires `WallConstructed { tile, faction }` for cache invalidation.
- **Tier gossip promotion (`knowledge.rs`)**: `HouseholdMember` who *is a social contact* (`is_social_contact` — dedicated Socialize OR live ambient work-pairing) within 3 tiles of a same-root-faction `Bureaucrat`/`Chief` promotes household clusters to Settlement tier; two officials → Settlement bubbles to Faction.
- **Cluster-density-gated chief postings** (`chief_job_posting_system::faction_knows_cluster`) walks faction-tier + owned settlement-tier knowledge. No cluster → skip food/Wood/Stone posting. CraftOrder-driven Stockpile and Build are ungated.

`AgentMemory` is ~30 lines: `visited_settlements: [Option<(SettlementId, u8)>; 8]`. `relationship_decay_system` walks `RelationshipMemory` only — `cluster_decay_system` owns cluster freshness.

- **Faction archetypes (`archetype.rs`):** `FactionCapabilities { home, storage, shelter, settlement, posting, land, economic_policy, income, inheritance }` on `FactionData.caps`. `derive_from_legacy(lifestyle, preset, catalog)` mirrors the legacy cross product. - **Organic settlement AI (`organic_settlement.rs`, `survey_task.rs`):** persistent `SettlementBrain` per full settlement tracks phase, anchors, traffic heat, soft districts, planned `road_segments` / `road_tiles`, organic parcels, frontier, seed, layout hash. Runtime surveys are async declarative diffs: `schedule_survey_tasks_system` snapshots bounded terrain, structure keys, member offsets, and a compact faction view; `AsyncComputeTaskPool` computes a new brain plus road-queue pushes; `apply_survey_results_system` validates owner/home/peak-pop freshness and applies at most `PerfWorkBudget.settlement_plan_applies_per_tick` results per tick. Pressure/morphology/project selection remain main-thread validation systems. `chief_directive_system` consumes `SelectedSettlementIntents` first; legacy `generate_candidates` is fallback.
- **Parcel allocation (`build_parcels`):** non-ag parcels via `build_parcels_road_driven` (road-fronted sweep, `Agricultural` deliberately absent from `KIND_ORDER`); then `build_ag_belt` packs the agricultural fields as contiguous 16×16 blocks **OUTSIDE** the built-up footprint (union of non-ag parcels + Civic/Residential/Crafting district discs, inflated 3 tiles), grown as a connected blob from the most-fertile clear seed. **Self-anchored on a home-aligned 16-lattice — NO dependency on the old home-biased `best_fertile_tile`/`Field` anchor** (that silently no-op'd on low-fertility starts and biased fields toward the base). Scan reaches beyond the footprint (`fp_extent + 4·block`, floored at `survey_radius + 2·block`, capped). Gates: clear (`rect_clear_for_parcel`), not in footprint, `mean_fert > 0` (keeps fields off Sand/Snow/Stone, no arbitrary floor). Road access is a **soft score bonus**, not a hard gate (a track gets carved as traffic builds) — the belt never silently produces zero farms for lack of a road. Deterministic (home-anchored lattice, explicit sort, `tile_hash` tiebreak), shares `MAX_PARCELS`, emits `frontage_edge: None`. Camps/nomadic keep `build_parcels_frontier_driven` (ag still allocated there). **`compat_plan_from_brain` emits Agricultural ONLY from belt/frontier parcels** — the district-broad and legacy `build_settlement_plan` fallbacks are suppressed for Agricultural (they were home-centred and carved farms across the whole base). No belt parcel ⇒ no Agricultural zone that survey (retries as the layout evolves). Once carved, fields flip to `TileKind::Cropland` (legibility + road protection — see Construction).
- **Initial survey (`organic_settlement::survey_one_settlement`)** remains synchronous for `kickoff_initial_survey_system` (OnEnter, before seeding), sharing the same compute core so `SettlementBrain` exists before seed picks anchors.
- **Brain-aware seed anchors (`construction::pick_seed_house_anchor`):** with `Option<&SettlementBrain>` parameter, walks Residential parcels, biases `−100` for frontage-edge parcels, falls back to legacy plan-zone scan, then radial spiral. Layout-decision foundation: `walled_house_tile_plan(...)` enumerates wall+door+bed cells consumed by both `plan_building` and `seed_walled_house_at`.
- **`SimulationState::{Warmup, Active}` SubState** flips to `Active` at the tail of OnEnter chain. No system currently gates on `Active` (avoids tick-0 stall); scaffolding for future opt-in.

### Construction (`construction.rs`)

`BlueprintMap`, `WallMap`, `BedMap`. `faction_blueprint_system` decides; `construction_system` consumes resources and finalizes. Every finished structure carries `StructureLabel(&'static str)` from `.label()`; `StructureIndex` (tile→entity) maintained by `on_structure_label_add` / `on_structure_label_remove` hooks.

- **Wall durability** (`plans/vehicle-system-tanks.md` Phase 1): every `Wall` entity carries `combat::Health` (`WallMaterial::max_hp` — Palisade 40 → CutStone 240). `apply_wall_damage(&mut Health, raw, material)` is the single damage entry point (mitigates by `WallMaterial::damage_resist`, saturating-sub). `wall_destruction_system` (Sequential) despawns a dead-`Health` wall, clears the `WallMap` entry, reverts the tile to passable `Grass`, emits `TileChangedEvent` + `WallDestroyed { tile }`.
- **Wall ladder** (`best_wall_material`): Palisade < WattleDaub (`PERM_SETTLEMENT`) < Mudbrick (`FIRED_POTTERY`) < Stone (`COPPER_WORKING`). Walls survive chunk streaming; natural-bedrock fallback walls get `WallMaterial::Stone` + `StructureLabel("Stone Wall")`. The wall **entity** is durable (`WallMap` is the truth, kept across unload, never put in `TileSpriteIndex.by_chunk`); the wall **tile** is a chunk delta lost on regen, re-projected on reload by `restamp_walls_on_chunk_load` (`construction.rs`, FixedUpdate restamp chain) — see `world/CLAUDE.md` → Streaming schedule.
- **Scarcity-aware selection (`select_wall_material`)** wraps `best_wall_material`. `None` availability (seed mode + wall-upgrade pass — materials stamped free) returns the pure tech-best rung verbatim. `Some(&MaterialAvailabilityView)` applies **procure-primary-first**: keep the highest tech rung that's buildable, market-haul its `Scarce` inputs; step down the tech ladder (`WALL_LADDER_BY_TECH`, tech-progression order — *not* enum discriminant order) only when a rung has an `Unavailable` input; return `WallSelection::EmergencyShelter` when every rung is blocked. `classify_resource` tiers each input vs the per-structure recipe qty: `Available` (deposited stock ≥ need) / `Tight` (short but `faction_knows_cluster` → existing gather resolves it) / `Scarce` (not stored/gatherable but `affordable_qty` ≥ need at the market node) / `Unavailable`. `classify_construction_procurement_system` (Economy, `.before(chief_job_posting_system)`, `CHIEF_POSTING_INTERVAL` cadence) rebuilds `FactionData.{material_view, procurement_plan, procurement_market}` from deposited storage + `faction_knows_cluster` + the faction market node price/stock + treasury. An **empty** `material_view` means "not yet classified" → callers pass `None` (legacy `best_wall_material`, no emergency), so shelter defers one classifier window rather than emit a doomed hut that would stall and hog the concurrency cap.
- **Era emergency shelter** (`find_emergency_bed_tile`, Step 7): when `select_wall_material → EmergencyShelter` and a runtime bed deficit exists at Neolithic+, emit a low-score bare `Bed` on a deterministic **era-keyed annulus** from home (Neolithic 12..32 outskirts / Chalcolithic 8..22 work-yard / Bronze 6..18 civic-overflow; one era-parameterised finder, the annulus *is* the era distinction). Wired into **both** `organic_settlement::pressure_to_intent` (primary `SettlementPressureKind::Shelter` path — emits `Single(Bed)` + suppresses the walled-house) and `generate_candidates` (legacy fallback — defers when view empty, suppresses Hut/Longhouse when emergency). Paleo/Meso keep their hearth-crescent branch; non-shelter builds get no previous-era substitute (defer with reason, no candidate).
- **Market procurement (`HaulSource::{Storage, Market{max_unit_price}}`** on `JobProgress::Haul`, snapshotted onto `ClaimTarget.haul_source`): Phase 3c stamps `Market` on an unmet slot only when `policy_for(rid).chief_allocates_labor` **and** the blueprint's `poster_class ∈ {Chief, Architect}` **and** `FactionData.state_funds_public_works` — preserves Market-mode free-agent labor. The Market-haul dispatcher (in-hand-fast-path-style direct dispatch in `htn_acquire_good_dispatch_system`, no registry method) routes the worker to `procurement_market` and chains `[BuyMaterialAtMarket, HaulToBlueprint]`. `production::buy_material_task_system` (exclusive, Sequential) does one **atomic** advance→`trader_buy_at_node`→return-residual so the worker never carries job capital across a tick (all existing `aq.cancel()` sites stay correct), then drops to Idle so the in-hand fast-path delivers the shared `HaulToBlueprint` leg (`route_haul_to_blueprint_tail`).
- **Door direction + doormat reservations (`doormat.rs`):** `Door { dir: TileEdge, doormat_tile }`. `Blueprint.door_dir` set by `plan_building` / `plan_composite_building` from `frontage_edge` (or `TileEdge::toward(centre, home)` fallback). `entrance_cell_for_edge` returns the centre cell, never a corner. Finalization writes doormat to `Road` via `write_road_tile` and the connector target is chosen by `find_door_connector_target(brain, doormat, 12)` — `Road(target)` (nearest carved or planned spine tile, validated by `placement_reachability::path_exists`), `None` (a Road already sits chebyshev-1), or `HomeFallback` (no spine in radius, no brain). Both the seed walled-house finalize and the runtime door finalize push the connector edge onto `RoadCarveQueue`; the legacy `road_within(.., 4) → home` radial is gone (it produced wagon-wheel layouts). **`DoormatReservations`** consulted by every placement helper (`is_clear_footprint`, `plot_rect_vacant`, `find_palisade_site`, `find_clear_tile_in_zone`, `find_unfilled_civic_zone_tile`, `find_bed_tile_around_hearth`, `seed_walled_house_at`). `Door`'s `on_remove` hook frees the reservation.
- **Door direction is verified, not assumed.** `pick_clear_door_cardinal` tries the preferred cardinal then falls back ranked by doormat→home chebyshev; returns `None` if every cardinal is blocked (planners abort, seed loop stamps anchor as used). **`doormat_reaches_home`** is now a thin wrapper over `placement_reachability::path_exists` (no parallel BFS): it keeps the legacy contract (Wall/unloaded block, Stone-aversion overlay preserved, 1500 cap) but the step model is the agent-faithful 3D check, so a doormat is accepted iff a worker can genuinely walk it home — courtyard-pocketed cardinals still fail.
- **Placement reachability (`placement_reachability.rs`) — one shared layer, no parallel BFS.** `path_exists` is a bounded, agent-faithful A* over the live `ChunkMap` using the canonical `passable_step_3d` / `passable_diagonal_step` rules; a found path proves walkability and within-chunk sealed pockets are rejected. Correct at seed time (reads tiles incl. walls stamped during the seed pass; the connectivity graph is *not* reliably built at `OnEnter(Playing)`). `connectivity_prefilter` is an optional O(1) runtime-only fast-reject (inconclusive ⇒ confirm with `path_exists`). Wrappers: `tile_reachable_from_home`, `rect_reachable_from_home`, `simulate_house_reachable` (validates the house *as built* — doormat→home through finished walls + every interior bed reachable via the door + door↔doormat z-step), `spawn_tiles_from` (BFS frontier so members/households are reachable-from-home by construction). Gated surfaces: `plan_building` / `plan_composite_building` / `seed_walled_house_at` (simulated house), `seed_starting_farms_system` + `append_dwelling_gardens` (rect, prefer reachable offset), `spawn_population` member tiles + `seed_market_households` (reachable pool), `choose_site_for_intent`, and the single `chief_directive_system` pre-spawn gate (covers organic + `generate_candidates` fallback). Farm postings validate the plot rect **once** at posting time (`chief_job_posting_system`) — no per-tile per-tick recompute.
- **Roads are protected from new construction.** All footprint/shape/anchor/perimeter helpers reject `TileKind::Road`. Palisade carvers leave 3-tile gateways for spine flow capacity. **Defensible-core envelope (Settlement realism):** `find_palisade_site(PalisadeMapsRO { bed_map, granary_map, shrine_map, well_map, campfire_map }, brain, ..)` wraps the axis-aligned bbox over the *union* of defended structures within chebyshev 25 of home — not `BedMap` alone — so granaries/shrines/wells stamped outside the original bed cluster get wrapped too. Gateway centres align with the row/column where `brain.road_tiles` actually crosses each edge (nearest planned-road tile to the legacy `hx`/`hy` cardinal); fallback to `hx`/`hy` when no road meets the edge.
- **Farm tiles are protected from road carving.** `write_road_tile` + `road_carve_system` (the single chokepoint every `RoadCarveQueue` producer — doormat extension, spine drain, desire path, survey — drains through) skip any `TileKind::Cropland` tile, and `road_carve_system` additionally skips `land::tile_is_farm_protected` (tile in `PlotIndex.ag_tiles` or carrying a `PlantMap` crop) so pre-stamp Agricultural-plot tiles and standing crops survive too. The survey desire-path diff also drops a target inside any Agricultural parcel rect. Roads tolerate the resulting 1-tile gaps (Bresenham `continue`s, no reroute).
- **Composite footprints (`BuildIntent::CompositeHouse`)**: `plan_composite_building` walks `building_template::shape_tiles(shape, anchor, rotation)`, classifies each cell as Wall / Door / Bed. LShape/UShape support exists, but automatic LShape shelter emission is disabled until the small farmstead mask grows true interior bed cells; growth currently stays on Huts/Longhouses.
- **Civic milestones (`civic_milestones.rs`):** `(Era, peak_pop) → bool` table gates Granary / Shrine / Market / Barracks / Monument / Bridge / Dam growth. Seed-time bypass is dialled by `GameStartOptions.maturity: StartSettlementMaturity::{Founder, Established (default), Developed}` via `should_seed_civic(...)` — `Founder` re-imposes runtime gates, `Established` matches the legacy "society in progress" verbatim, `Developed` additionally force-enables Market/Barracks/Monument for Chalcolithic+ starts. The `GameStartOptions` resource is bundled into `BuildingMapsRO` so `chief_directive_system` stays under the 16-param cap.

- **Organic layout randomness** (`settlement.rs`): `culture_hash` seeds `fastrand::Rng` for ±1 tile zone jitter. `generate_candidates` rolls Hut/Longhouse (Longhouse forced when `bed_deficit ≥ 4`).
- **`SettlementPlan` compatibility projection:** `settlement_planner_system` prefers `organic_settlement::compat_plan_from_brain` when a brain exists. `SettlementParcelIndex` is authoritative; `SettlementPlan` is transitional/debug.

### Game-start seeding (`seed_starting_buildings_system`)

`OnEnter(Playing)` chain: `spawn_population → auto_found_default_settlements_system → {sync_faction_techs_from_chief_system → derive_tech_adoption_system → refresh_construction_poster_pool_system, settlement_peak_population_system} → kickoff_initial_survey_system → seed_starting_buildings_system → resurvey_after_seeding_system → project_initial_settlement_plans_system → carve_plots_system → seed_starting_farms_system → populate_seed_reservation_system → clear_obstacles_under_seeded_structures → mark_warmup_complete_system`. The post-seed re-survey + carve segment (`plans/spawn-farm-seeding.md`) ensures `PlotIndex.by_faction_hash` is established at tick 0 against the *actually built* footprint — without it the first runtime carve mismatches and tears down the seed plot, orphaning the starter Cropland.

**Bootstrap P1 — `SeedReservation` (`seed_reservation.rs`):** persistent additive `AHashSet<(i32,i32)>` of every tile the seed pipeline stamped or queued. `populate_seed_reservation_system` (OnEnter, after seed-buildings + seed-farms, before clear-obstacles) consolidates from `StructureIndex` ∪ `DoormatReservations` ∪ `RoadCarveQueue` (Bresenham-rasterised via `rasterize_line_into`) ∪ `SettlementBrains.road_tiles` ∪ `PlotIndex.ag_tiles`. Three consumers gate on it: `plants::seed_target_tile_ok` rejects reserved tiles for wild scatter; `world::chunk_streaming::spawn_chunk_plants` (via the bundled `StreamPlantParams` SystemParam) skips reserved tiles when chunks stream in after warmup; `clear_obstacle::{clear_obstacles_under_seeded_structures, react_obstacle_under_structure_system}` treat reserved tiles equivalently to `StructureIndex` / `BlueprintMap` hits. Catches the "plant streams onto planned-but-uncarved doormat / road" class without requiring a streaming hook for every queued tile.

**Bootstrap P5 — deferred (no profiling data).** The original `plans/settlement-bootstrap.md` flagged P5 ("replace reactive `react_obstacle_under_structure_system` with a plan-driven `cleanup_tiles` walk") as opportunistic. Without measurements showing the reactive path is hot, keeping it is the right call — `react_obstacle_under_structure_system` already consults `SeedReservation` (P1) so late-streamed obstacles on bootstrap tiles get the same treatment as stamped-structure tiles.

**Bootstrap P4 — stranded-member relocation (minimal).** `relocate_stranded_members_system` (`settlement_bootstrap.rs`, OnEnter, after `populate_seed_reservation_system` + `clear_obstacles_under_seeded_structures`) walks every Person and reassigns any founder whose spawn tile overlaps `StructureIndex` / `WallMap` / `BedMap` / `DoormatReservations` / `BlueprintMap` / `SeedReservation` / `PlotIndex.ag_tiles` (or sits on `TileKind::Wall` / `Road` / non-passable) to the nearest safe `spawn_tiles_from(home)` tile. Per-faction reachable pool is cached so multi-stray relocations within one settlement share one BFS. Skips `SOLO` agents. Updates `Transform` + `PersonAI.{target_tile, dest_tile, current_z, target_z}` in lock-step so the next sim tick reads the new tile. This is the **observable** P4 invariant from `plans/settlement-bootstrap.md` ("members never land on a tile a stamp later writes over"); the full `SettlementBootstrapPlan` struct + validator + spawn reorder are deferred — the chief-tech-priming flow makes the reorder costly without a synthetic `FounderSpec`, and the relocation pass achieves the same end state with one isolated system.

**Bootstrap follow-ups (post-P4).**
- **Seed-time civic hearth cap = 1.** `generate_candidates` forces `desired_hearths = 1` at Neolithic when `seed_techs.is_some()`. A 20-founder Established start no longer stamps 3 civic hearths lined up next to home — extra hearths come from dwelling templates instead.
- **Longhouse carries an interior hearth.** `walled_house_tile_plan` takes an `Option<(i32, i32)> interior_hearth` arg; Hut keeps `None` (3×3 has no spare interior cell), Longhouse passes `Some((0, 0))` so the 5×3 centre tile is a Campfire flanked by beds at ±1. Reflected in `BuildIntent::Longhouse::{required_goods, required_kinds}` (now includes `BuildSiteKind::Campfire`). Seed pipeline and runtime `plan_building` both spawn the interior Campfire.
- **All carved roads are 2 tiles wide.** `road_carve_system` writes an extra perpendicular-adjacent tile alongside every Bresenham step (`widen_dx`/`widen_dy` follow the dominant axis). `seed_reservation::rasterize_line_into` mirrors the same 2-tile rasterisation so the planner reserves the full footprint. Closes the 1-tile-gap-between-road-and-house slot where wild plants kept sprouting.

**Bootstrap P3 — seed-time Longhouse at Neolithic.** `generate_candidates` residential gate (`construction.rs`) lifts from `techs.has(CITY_STATE_ORG)` to `techs.has(CITY_STATE_ORG) || (seed_mode && bed_deficit >= 2.0)`. A multi-adult Neolithic founder pool (P2 kin partition into 2–4 adult households) now seeds visible dwelling variety — 2-bed Longhouses (5×3) alongside 1-bed Huts (3×3) — rather than identical hut rows. Runtime ladder unchanged (CITY_STATE_ORG remains the post-seed gate); the change is seed-only.

**Bootstrap P2 — kin households + reciprocal affinities (`settlement_bootstrap.rs`).** `seed_starting_relationships_system` (OnEnter, after seed-buildings) partitions each Subsistence/Mixed faction's founders into deterministic kin groups of ≤4 (sorted by entity bits) and forms one `FactionRegistry::spawn_household` per group, then stamps `HouseholdMember{ household_id }` on every member. Per group of N: 2 = spouse pair, 3 = spouse + 1 sibling, 4 = spouse + 2 siblings, 5–7 = one 4-group + a remainder. Reciprocal `RelationshipMemory` affinities seeded **directly into the entry slots** (not via additive `update`): spouse = **79** (above `PARTNER_AFFINITY_THRESHOLD = 60` so the homeless-pass treats them as paired, but strictly below `REASSIGN_AFFINITY_THRESHOLD = 80` so the runtime's already-housed-woman migration doesn't fire on tick 1 and queue a Bed BP adjacent to the husband); sibling = **60** (kin reads as partner, no spouse-style cohabitation). Skips: SOLO factions, nomadic factions (`caps.home.is_mobile()`), Market factions (`caps.inheritance.seed_storage_tile == true` — handled by the pre-existing one-person `seed_market_households` path), and any member that already carries a `HouseholdMember` (Market re-entry guard). **`chief_directive_system` gates on `faction.parent_faction.is_none()`** — household sub-factions don't run their own construction agenda; the village owns building.

### Bridges & Dams

`BuildSiteKind::Bridge` adds a tile-replacing finalize path (alongside `Wall`): writes `TileKind::Bridge` over the prior `River` cell and spawns `Bridge { restore_tile, faction_id, tile }`. Deconstruct refunds `2 wood + 1 stone` on the nearest passable bank.

**Dams mirror this pipeline exactly** (Phase 4 of `plans/water-simulation.md`). `BuildSiteKind::Dam` → `TileKind::Dam=25` over a `River`/`Water` cell + `Dam { faction_id, tile, restore_tile, crest_z }` in `DamMap`. The **entity is the durable truth**; the tile kind is its cache projection, restamped from `DamMap` on chunk reload by `world::water_runtime::restamp_runtime_water_on_chunk_load` (shared `stamp` closure with Bridge). Recipe `6 stone + 4 wood`, 180 work, dedicated **`DAM_BUILDING`**-gated (Bronze Age, `AdoptionScale::Institutional`, prereqs `BRIDGE_BUILDING + MONUMENTAL_BUILDING`; `BuildRecipeIdx::Dam` appended last so existing vec positions stay stable). Unlike Bridge, the cell is **not** water-like/fresh/drinkable — water is blocked. Finalize calls `RuntimeWater::register_dam(tile, crest_z=bp.target_z)` (recorded in `RuntimeWater.dam_crests` — a cache of `DamMap`, consumed by the Phase 5 fluid sim to pool water upstream). Deconstruct shares the generalised `water_anchored_refund_tile` path with Bridge: restores the prior tile, `RuntimeWater::clear_dam`, refunds `3 stone + 2 wood` on the nearest bank. `FurnitureMaps` bundles `dam_map` + `runtime_water` (under the 16-param cap). Built **right-click Build Dam on River/Water** *and* **AI-planned** by `organic_settlement::dam_intent_emitter_system`.

- **Recipe**: "Timber Bridge" — 4 wood + 2 stone, 120 work_ticks, `tech_gate = Some(BRIDGE_BUILDING)`.
- **Tech**: `BRIDGE_BUILDING` (Chalcolithic, `AdoptionScale::Institutional`, prereqs `PERM_SETTLEMENT + DUGOUT_CANOE + COPPER_TOOLS`). **Civic gate**: `CivicKind::Bridge` at `(Chalcolithic, 20)` peak pop.
- **Worker adjacency**: `Blueprint.work_stand: Option<(i32, i32)>` caches a chebyshev-1 bank tile. `BuildSiteKind::is_water_anchored()` returns true for **Bridge and Dam**. `work_stand_for_bridge(...)` populates it at spawn (prefers cardinals, rejects other-Bridge neighbours); the `player_command` water-anchored work-stand routing generalises to Dam automatically via `is_water_anchored()`.
- **AI generation**: `organic_settlement::bridge_intent_emitter_system` scans each faction's `SettlementBrain.road_segments` for river runs of `1..=MAX_BRIDGE_SPAN = 4` bounded by passable banks; one bridge per faction per cadence.
- **AI dams** (`dam_intent_emitter_system`, scheduled beside the bridge emitter): gated `DAM_BUILDING` + `CivicKind::Dam` (Bronze, 30 pop) + settled non-SOLO. Pure `score_dam_site` composes three motives and argmaxes one site/settlement/cadence: **Irrigation** (nearest watercourse upstream of the Ag-belt parcels, must gravity-feed: water_z ≥ belt_z), **Reservoir** (nearest watercourse to home × dryness `river_distance_at`/16 × population × seasonal-low lift), **RoadCrossing** (a `detect_bridge_runs_in_trace` short crossing — dam crest carries the road). `DAM_SEARCH_RADIUS=40`, `DAM_MIN_SPACING=24`, `DAM_SCORE_THRESHOLD=1.0`; author-less like the bridge emitter (Dam recipe has no tier picks). `season_low` from `Calendar::discharge_multiplier`.

- **River-aware organic planner** (`build_road_network`): pre-`BRIDGE_BUILDING`, drops river-crossing segments via `trace_crosses_river` and filters anchors through `same_bank_bfs`. Primary spoke order is `RiverAxis`-aware. Post-tech, filters lift and `bridge_intent_emitter_system` back-fills crossings. **Anchor-driven phase additions (Settlement realism):** Village emits a perpendicular cross only when (a) `member_count ≥ 16` OR (b) a WaterAccess/Field/Market anchor projects > 6 tiles off the spine axis (endpoint = anchor projection, asymmetric). Chiefdom drops the symmetric ±18 parallels and instead emits secondary spurs toward the top-3 unmet (perp-projection > 2) anchors, weight-sorted, capped at 3. Secondary/Alley endpoints get ±1 tile jitter keyed on `layout_hash`; Primary spine endpoints stay verbatim so the door connector can always dogleg onto them.

Ingredient gate uses `faction.storage.stock_of()` (deposited stock only). Missing **gatherable** inputs pull-post `Stockpile { resource_id }` per non-Wood/Stone shortfall; missing **craftable** inputs (e.g. Tools) are skipped (covered by `craft_demand`).
- **Functional craft demand (`jobs::compute_craft_demand`, no population quota):** `resource_demand_system` (Economy, 60-tick) computes `FactionData.craft_demand` — a per-output netted deficit. Modelled goods: **Weapon** (unarmed Hunters + raid-party members), **Armor** / **Shield** (those combatants lacking combat mitigation — `combat.rs` reads `armor_stats`), **Cloth** (members with a bare TorsoArmor slot, gated on the faction knowing `LOOM_WEAVING`), **Ard Plow** (faction has `ARD_PLOW` + a state-owned Agricultural plot + no plow stored/in-flight), and **Tools** (derived: only as the Ard Plow recipe ingredient). Each is netted vs storage + in-flight crafts/orders; equipped/carried/inventory items satisfy their holder (lower the deficit), they are not counted as spare. Luxury / cart-parts get **no** autonomous chief posting — absent key ⇒ zero demand. The household Belonging→Cloth contract self-dedups against chief cloth demand via the `in_flight` scan (a live `JobProgress::Crafting` posting counts as spare cloth). `resource_demand` keeps only gatherable blueprint + food demand (so `opportunity.rs` MaterialDeficit stays correct); `craft_demand` is the separate crafted-goods channel.
- **Clothing mood (`mood.rs`):** `derive_mood_system` applies a flat `CLOTHING_MOOD_PENALTY` (12) when a member's TorsoArmor slot is empty and the faction knows `LOOM_WEAVING` — the functional consumer that justifies autonomous Cloth craft demand. No clothing/temperature `Need`.
- **Maintenance slices:** Build/Craft workers advance real blueprint/order progress for up to `MAINTENANCE_WORK_SLICE_TICKS = 120`; unfinished non-cargo labor then yields to idle while preserving `JobClaim`. Cargo chains (withdraw→haul, scavenge→deposit) stay atomic.

- **Posting target curves:** `GATHER_TARGET_CAP=1500`, `MATERIAL_GATHER_CAP=96`. Food multiplied by `food_seasonal_multiplier` (Spring/Summer 1.0, Autumn 1.5, Winter 1.3).
- **WithdrawMaterial intent:** dispatcher picks the resource. Each dispatcher reads `FactionStorage.totals` + `StorageReservations` to find the nearest tile with effective stock.
- **`StorageReservations`** (`(tile, ResourceId) → reserved_qty`, mutex-wrapped). Successful `WithdrawMaterial` increments + stashes `(reserved_tile, reserved_resource, reserved_qty)` on `PersonAI`; `release_reservation` decrements on every teardown.

## Knowledge & technology

- **`PersonKnowledge`** carries `aware` and `learned` (`u64` bitsets, learned ⊆ aware) + `learned_at: [u32; 64]`. `complexity(tech)` is era-based (Paleo 1 → Bronze 5; Cuneiform/Lunar/City-State 6). `learning_slowdown(stats, k) = 1 + complexity_used / (intelligence × 2)` is applied to `add_study_progress` and passive teaching's chance. Newborns inherit `Aware ∪ Learned` from both parents. `study_progress: AHashMap<TechId, u32>`; at `study_threshold(tech) = complexity * STUDY_TICKS_PER_COMPLEXITY (3600)` → `try_learn`.
- **Discovery (per-action):** `discovery_system` consumes `DiscoveryActionEvent`s. Eligible = prereqs Learned + not already Aware. Roll `base × (1 + INT_mod × 0.1) × (1 + skill_xp / 1000)`, cap 0.5; success grants Aware and jump-starts `study_progress` by `complexity × INSIGHT_PROGRESS_PER_COMPLEXITY (1200)` capped below threshold.
- **Awareness gossip:** `awareness_gossip_system` (Economy) ORs `aware` bitsets + `AgentMemory.visited_settlements` between adjacent **social contacts** (`is_social_contact` — dedicated Socialize OR live ambient work-pairing).
- **Passive teaching:** `tech_teaching_system` scans pairs within 3 tiles, rolls `0.004 × INT_scale / learning_slowdown(student)` per shared Aware-but-not-Learned tech. Highest-complexity teachable lands directly in Learned. **Deliberate-only** — gated to explicit `AgentGoal::Socialize`, never ambient work-pairing (casual chatter ≠ instruction; see Ambient work-social multitasking).
- **Directed teaching (`teaching.rs`):** **Read** `+1`/tick; **Lecture** drafts ≤8 same-faction adults within 6 tiles, `+2`/tick for 600 ticks; **1-on-1 Teach** `+3`/adjacent-tick. All carry `Drafted`.
- **Community adoption (`technology_adoption.rs`) — analytics only, gates nothing.** `FactionData.tech_adoption: [AdoptionStage; TECH_COUNT]` is still derived by `derive_tech_adoption_system` (Economy, every `TICKS_PER_DAY/4`, six stages) but is consumed **only** by `ui/tech_panel` + the `TechAdopted`/`TechInstitutionalized` activity-log entries. **No construction/civic/tier/shelter gate reads it.** The single build-tech surface is `FactionData.buildable_techs` (poster-pool union of resident chief + architect Learned); `community_has` / `community_adoption_bitset` both return it. `can_direct_tech(faction, tech)` = chief-Aware (planning authority); per-person execution = `PersonKnowledge::has_learned`. `seed_prime_tech_adoption_system` deleted.
- **Decay:** `derive_tech_adoption_system` is symmetric — conditions falling away walk the stage down. Downgrades throttled to one stage per game-day; upgrades immediate.
- **Founder seeding (`PersonKnowledge::seeded_realistic_through_era`):** chief learns Personal+Household+Subsistence+Specialist+Institutional through era ≤ E; ~1/8 members Specialist (+MilitaryTransport); rest Personal+Household+Subsistence Learned. Everyone Aware of the full era.
- **Tablets and books:** `Good::ClayTablet` and `Good::Book` carry `Item.tech_payload: Option<TechId>`. Recipes gated on `CUNEIFORM_WRITING`, Workbench. Reading doesn't consume.
- **Tablet posting (`chief_tablet_posting_system`):** every 3600 ticks. Picks highest-complexity chief-Learned bit that <50% of adults are Aware of, posts `JobKind::Craft` with `tech_payload`. Player override via `PlayerCraftRequest` (`JobSource::Player`, priority 180).
- **Tech tree (`technology.rs`):** 50 techs, 5 eras, prereq DAG, per-tech `triggers: &[TechTrigger]`. `TechBonus` adds yield/storage/combat bonuses; bonuses aggregate via `faction.techs` (chief-aware). Bronze-Age war-vehicle line: `SIEGE_ENGINEERING (47)`, `ARMOR_PLATING (48)`, `POWERED_TRACTION (49)` — gate the siege/tank vehicle parts + templates.

It maps the worker's `AgentGoal` to the `ToolForm` their next episode will hard-gate on (`GatherWood→Axe`, `GatherStone→Pick`, `Farm→Sickle`, `Craft→`first `tool_requirement` on the worker's claimed recipe via `JobBoard::get`); when the `ToolKit` lacks that form and faction storage holds a satisfying `Item` (form + tier floor checked per ground-item stack — a Bone awl can't satisfy a Copper-awl gate), it dispatches `Task::WithdrawTool { req }` head + queues `Task::StowToolKit { req }` tail. `WithdrawTool` shares `TaskKind::WithdrawMaterial` with the bare-count withdraw; the production executor's `as_withdraw_tool()` branch handles the item-picking. `goal_dispatch_system` preserve-arms key on `aq.current.as_withdraw_tool().is_some()` (WithdrawTool leg) and `TaskKind::StowToolKit` (stow leg). ## Raids (`raid.rs`)

Faction-level raid state machine — `FactionData.raid_phase: RaidPhase` is the source of truth; `raid_target` / `under_raid` are derived projections kept in sync each tick so existing HTN / goal / UI call sites compile unchanged.

- **`RaidPhase`** `{ Idle, Preparing { since_tick, target }, Marching { .. }, Engaged { .. }, Cooldown { until_tick } }`. `is_active()` = `Preparing|Marching|Engaged`.
- **Sustained-pressure trigger (`faction_decision_system`, Economy):** raid only when BOTH `food_deficit` (`food_total < members`) AND `hunger_crisis` (`avg_hunger >= RAID_MIN_AVG_HUNGER=140`) have held for `RAID_TRIGGER_DAYS=2` days. Per-faction streak counters `food_deficit_streak_tick` / `hunger_crisis_streak_tick` track the last tick each condition was *false* (lazy-init to `now`). No emergency override; no single-tick flip. `RAID_COOLDOWN_DAYS=10` after every raid (half-length on an aborted prep).
- **Target filter** (`pick_raid_target`): excludes SOLO / self / same-root kin / tributary pairs (`dominance_over` / `subordinate_to`) / food-poor (`food <= members * RAID_RIVAL_RESERVE_PER_MEMBER=5`) / beyond `RAID_MAX_TRAVEL_TILES=500` chebyshev. Picks nearest.
- **Capability drafting** (`raid_capability` + `select_raid_party`): ranks members by `0.5*hp_frac + 0.3*str_norm + 0.2*armed`; rejects dead / destroyed-torso / <50% HP. Party cap = `min(RAID_MAX_PARTY_ABS=8, ceil(members * 0.35))`. Chief and `Drafted` excluded; chief topped up only if party would be `< RAID_MIN_PARTY=2`. Stored in `FactionData.raid_party` — the **only** set `goal_update_system` may put on `AgentGoal::Raid` (via `FactionRegistry::is_raid_party_member`).
- **Preparing phase:** `raid_prep_dispatch_system` (ParallelB, before the combat dispatcher) routes unarmed party members to withdraw+equip a `weapon` from faction storage (mirrors the hunting-spear dispatcher; `WithdrawMaterial → Equip` chain, `production::finish_withdraw_material` primes the Equip leg). `Preparing → Marching` when all party armed, or half-armed past `RAID_PREP_TIMEOUT_TICKS/2`. Aborts to `Cooldown` on food/hunger recovery, target loss, or genuine inability to arm by `RAID_PREP_TIMEOUT_TICKS=3 days`.
- **March / engage:** `htn_combat_faction_dispatch_system`'s `Raid` arm only marches a `Marching|Engaged` faction (skips `Preparing`). `raid_detection_system` flips `Marching → Engaged` when a raider reaches the enemy home (chebyshev ≤ 1) and sets the target's `under_raid` alarm.
- **`raid_execution_system`** (Economy): only `Engaged` raiders within chebyshev ≤ 1 of the enemy home steal food. Per-raider steal cooldown via `PersonAI.last_raid_steal_tick` (`RAID_STEAL_COOLDOWN_TICKS`); reserve gate keeps the target above `members * 5`; successful steal bumps `FactionData.raid_stolen_food`.
- **End conditions (`Engaged → Cooldown`):** food recovered, target drained below reserve, `RAID_ENGAGED_TIMEOUT_TICKS=2 days`, or living party `< RAID_MIN_PARTY`. Clears `raid_party`, resets `raid_stolen_food`.

## Plant lifecycle (`plants.rs`)

Calendar-driven, season-edge-triggered. `Plant.growth: u16` is a stage-time accumulator; on each season change `plant_lifecycle_system` adds `season_growth(kind, prev_season)` and runs at most one stage transition.

- **Per-season growth (`season_growth`):** Winter dormant (lethal for Grain). Grain Spring/Summer/Autumn = 4/5/2; BerryBush 5/4/2; Tree 4/5/3.
- **Stage thresholds:** Grain Seed/Seedling/Mature = 3/5/2 (annual). BerryBush 5/30/4 with Harvested mirroring Seedling (≈3 calendar years to first fruit). Tree 12/48/18 (≈4 years to maturity).
- **Sprout chance:** Seed → Seedling rolls `f32() < 0.20`; failure despawns. A plant carrying the `Cultivated` marker (deliberately sown via `Task::Planter`, inserted by `production.rs`) **skips the roll** and always sprouts; wild scatter / `PlayPlant` keep the 20% odds.
- **Harvest seed cofactor:** a Grain harvest returns `+2` `grain_seed` (`harvest_extra_yields`) so a seed-short founding village can grow its stock year-over-year (`+1` only broke even).
- **Mature → fruiting:** single dice roll at transition (Grain 20% / BerryBush 10% / Tree 5%) for one Seed in chebyshev radius (Grain/Berry r=2, Tree r=3) onto `Grass | soil_like` (incl. `Cropland`, via `seed_target_tile_ok`). Post-roll Grain enters Overripe, BerryBush reverts to Harvested, Tree reverts to Mature.
- **Winter mortality:** every Grain plant despawns at Winter onset (mature ones scatter dice first).
- `plant_lifecycle_system` runs in `FixedUpdate` `SimulationSet::Sequential` after `advance_calendar_system`. Edge-triggered via `Local<Option<Season>>`.

## Fishing (`fishing.rs`)

Terrain-based renewable food, unlocked by the Mesolithic `FISHING` tech. Fish are local renewable **stocks**, never per-tile entities.

- **`FishStock`** Resource — sparse `AHashMap<(i32,i32), FishStockCell { habitat, biomass, capacity }>`. **Lazily** initialized per water tile on first fishing touch, deterministic from `WorldSeed` (`tile_hash` → capacity `0.75..1.25 ×` habitat base, +15% near rivers). Untouched tiles read implicit-full. Lives across chunk streaming — simulation state, **not** restamped on chunk load (mirrors `farm::FieldTileIndex`).
- **`FishHabitat { River, Lake, Marsh, Coast }`** — **derived** by `habitat_at(chunk_map, globe, tile)` from `TileKind` + salinity (`River`/`Marsh` fresh; `Water` → `Coast` if Salt/Brackish else `Lake`). `Bridge`/`Dam` are not spots. Never stored as terrain.
- **`FishingMethod`** — v1 uses `Handline` (90 work ticks); `Trap`/`Weir`/`Net`/`BoatLine` reserved (data-model only).
- **`fish_regen_system`** (Economy, daily) — logistic regrowth with a flat `REGEN_SEED` floor so depleted stocks recover; seasonal multiplier (spring/autumn river runs ↑, winter ↓). Drops recovered cells to keep the map sparse.
- **HTN methods** — `FishForImmediateFoodMethod` (`AcquireFood` → `[Fish, Eat]`) and `FishForStorageMethod` (`StockpileFood` → `[Fish, DepositToFactionStorage]`), registered in `register_builtin_methods`, `UTIL_BASELINE` tier so fishing competes in the argmax with forage/scavenge/withdraw (distance-discounted). The `htn_acquire_food_dispatch_system` / `htn_stockpile_food_dispatch_system` populate `PlannerCtx.fish_spot_tile` via `nearest_fishable_water` (a `ChunkMap`-only scan within `FISHING_SEARCH_RADIUS`) — **only when the faction knows `FISHING`**, so the `fish_spot_tile.is_some()` precondition doubles as the tech gate. Each dispatcher's head-match has a `Task::Fish` arm (routes via `assign_task_with_routing`, stakes a `GatherClaims` entry keyed `(spot, Resource(fish))`). Depleted-spot avoidance is **`MethodHistory` failure-biasing**: the executor records a `record_target_failure` on exhausted/invalidated stock, `score_method_with_history` biases the method down, and forage wins until `fish_regen_system` refills the fishery — no `FishStock` lookup in the dispatcher. Pure terrain scan — no memory clusters (water is static terrain).
- **`fish_task_system`** (Sequential, before `gather_system`) — on `Working` adjacency to the water spot, accumulates `FishingMethod::work_ticks`, harvests `FishStock` (yield scaled by habitat × season × `SkillKind::Fishing` × faction `food_yield_multiplier`), deposits the catch into a free hand (overflow spills as a `GroundItem`), logs `ActivityKind::Fishing`, grants Fishing XP. `finish_fish` releases the claim, routes the trailing `Eat` (in place) / `DepositToFactionStorage` (to faction storage), and on a failed catch records a `MethodHistory` failure. `Task::Fish { spot_tile, method, output_resource }` / `TaskKind::Fishing = 55`; `goal_dispatch_system` preserves `(GatherFood|Survive, Fishing)`.
- **Resources** — `fish` (edible, perishable) + `preserved_fish` (`preserved` tag → two-pass eating ration); both in `core.ron`, `core_ids` accessors, auto-counted by `food_total()`. `Preserved Fish` craft recipe (index 19): `2 fish + 1 wood → 3 preserved_fish`, `FOOD_SMOKING`-gated, Workbench. `SkillKind::Fishing = 8` / `ActivityKind::Fishing = 13` (count bumps to 9 / 14). `LOG_RAFT`'s discovery trigger re-pointed onto `Fishing` activity (the `FISHING` tech keeps its `Foraging` trigger — fishing can't be discovered by an activity it gates).

## Realistic tools (`tools.rs`)

Separate functional tools — `ToolForm` decides *what* (Knife/Axe/Pick/Hammer/Sickle/Awl/FishingKit), an `Item`'s material+quality → `ToolTier` (`Bone<Stone<FineStone<Copper<Bronze`) decides *how well*. `ToolKit` component (`Vec<Item>`, era-scaled `capacity_for_era` 1/3) holds tools separate from `Carrier` hands; spawned on every Person (`person.rs` founder, `reproduction.rs` newborn `default()`, `test_fixture.rs` `fixture_full_toolkit()` — a full stone set so baseline tests aren't gated).

- **Hard gates** (graceful — a worker with *no* `ToolKit` reads as satisfied; an *empty* kit blocks/degrades): `gather_system` — Axe to fell trees (no-Axe → low-yield deadwood, tree NOT despawned + cluster not depleted), Pick to mine Stone/Wall/Ore (no-Pick → 1-stone trickle, tile NOT carved), Sickle to reap mature Grain (no-Sickle → FailedTarget; berries hand-gatherable). `dig_system` — Pick only when the carved floor tile `is_stone_like()` (soil hand-diggable). `fish_task_system` — Fishing Kit (no kit → failed catch). `craft_order_system` — recipe `tool_requirements`. Better `ToolTier` shortens the gather/mine work threshold via `work_speed_mult`.
- **Seeding** — `seed_starting_tools_system` (OnEnter after `seed_starting_farms_system`): per faction derives era from `current_era(techs)`, builds `starting_tool_loadout`, pre-stows a core tool into each member kit (so gates work tick 0, no deadlock) + spills the rest to faction storage as `GroundItem`s. Nomadic distributes the whole loadout round-robin across member kits. Sandbox skipped.
- **Tool acquisition** — `Task::WithdrawTool { req }` (shares `WithdrawMaterial` executor/`TaskKind`) selects the best-tier matching tool `Item` from storage into `Carrier` hands; trailing `Task::StowToolKit { req }` (`TaskKind::StowToolKit = 57`, `tools::stow_toolkit_task_system`) moves it into the `ToolKit`, evicting the lowest-tier resident to ground if full. *HTN dispatcher prepend is deferred* — seeded kits cover the gates; the chain is built/tested for a follow-up.
- **Faction demand** — `compute_faction_tool_deficits` (jobs.rs) nets the era loadout against member-kit + storage tool counts; `resource_demand_system` feeds the per-form deficits into `compute_craft_demand` so the chief posts missing tool recipes (`craft_priority` 3, above plow/armor). Generic `tools` commodity survives only as the Ard-Plow recipe ingredient.

## Animal domestication & husbandry (`animals.rs` + `husbandry.rs`)

Taming is species-agnostic. `Tamed { owner_faction }` is the single source of truth for ownership; `attach_pack_inventory_system` (Sequential, `Added<Tamed>` hook) auto-inserts both `DomesticAnimal { species, training, preferred_home, last_cared_tick }` and (for pack-eligible species) `PackAnimalInventory`. Species map: `Horse → DomesticSpecies::Horse`, `Cow → Cattle`, `Pig → Pig`, `Cat → Cat`, `Wolf → Dog` (tamed-wolf disposition).

**Goal/HTN/executor.** `AgentGoal::TameAnimal` replaces the old `TameHorse`. `TameAnimalScorer` (Subsistence, curiosity-lifted) fires when the faction is Aware of any of `HORSE_TAMING` / `ANIMAL_HUSBANDRY` / `DOG_DOMESTICATION` AND `GoalValidationQueries.wild_tameable_q` (Horse / Cow / Pig / Cat without Tamed) is non-empty. `htn_tame_animal_dispatch_system` (ParallelB) scans `SpatialIndex` within radius 15 for the nearest untamed candidate the faction has tech for: Horse → HORSE_TAMING, Cow/Pig → ANIMAL_HUSBANDRY, Cat → DOG_DOMESTICATION (wolves intentionally excluded — dogs come from seeding, not auto-taming). Single method `TameWildAnimalMethod` always wins; expands to `[Task::TameAnimal { target }]`. `tame_task_system` (Sequential) ticks `work_progress` to 100, then inserts `Tamed` and — if the target carried a `WildHerdMember` — calls `WildHerdRegistry::remove_member(herd_id, entity)` so the bloom/collapse cycle won't despawn or restore the now-domestic animal.

**Indexing.** `IndexedKind` includes `Cow`, `Pig`, `Cat` (`is_mobile_agent()` true for all). Spawn sites in `animals.rs::spawn_animals`, wild-herd bloom (`wild_herd::spawn_herd_members`), reproduction-bloom, and the new `seed_starting_tamed_animals_system` all carry `Indexed::new(...)`.

**Seeded starting animals (`seed_starting_tamed_animals_system`).** `OnEnter(Playing)` after `spawn_population` + `spawn_animals` + `sync_faction_techs_from_chief_system`, before `mark_warmup_complete_system`. For every eligible non-SOLO faction, spawns starter herds keyed off chief-seeded tech awareness: `DOG_DOMESTICATION` → 2 dogs + 1 cat; `ANIMAL_HUSBANDRY` → 2 cattle + 2 pigs; `HORSE_TAMING` → 2 horses. Scale bonus +1 per 20 founders, capped 6 per species. First two of each species are M+F for breeding viability. Global cap 200 animals across all factions to bound entity count. Settled factions place within radius 6 of `home_tile` on grass/scrub/cropland; nomadic factions insert `FollowingBand { faction, last_redirect_tick: 0 }` so `following_band_animal_redirect_system` keeps the herd with the band through LOD churn.

**Husbandry infrastructure (`husbandry.rs`).** Four new `BuildSiteKind` variants — `Pen` (6 wood + 2 stone, 80 ticks, `ANIMAL_HUSBANDRY`, capacity 4, species mask cattle+pig+dog), `Stable` (10 wood + 4 stone, 140 ticks, `HORSE_TAMING`, capacity 2, horse+cattle), `FeedTrough` (3 wood, 30 ticks, `ANIMAL_HUSBANDRY`, 20kg grain stock), `HitchingPost` (2 wood, 20 ticks, `ANIMAL_HUSBANDRY`, v2 cart/plow placeholder). Tile-indexed maps (`PenMap`, `StableMap`, `FeedTroughMap`, `HitchingPostMap`) follow the `BedMap` / `GranaryMap` on_add/on_remove pattern. `assign_preferred_home_system` (Sequential, `TICKS_PER_DAY/4` cadence) walks unhoused `DomesticAnimal` entries and assigns the nearest same-faction Pen/Stable with free capacity + matching species mask. Over-capacity animals stay unhoused. `feed_trough_consume_system` (Economy, daily) drains 50g grain per adjacent housed animal to relieve hunger. `husbandry_intent_emitter_system` (Economy, daily) emits a housing `Blueprint` near home for any settled faction whose animals outgrow housing: the census is split by class — horses count against Stable capacity (→ emit `Stable`), cattle/pig/dog against Pen capacity (→ emit `Pen`); cats are ignored (no housing structure). Stable need wins ties (a horse with no stable is unhouseable). Skips a faction with a pending blueprint of the needed kind.

**Animal work claims** (`AnimalWorkClaim { worker, use_kind, expires_tick }`) reserve an animal for a specific role (`AnimalUse::{Pack, Mount, Plow, Cart, Companion, Guard}`). v1 ships the data shape + `animal_work_claim_expiry_system` (Sequential, 60-tick cadence) that drops claims whose worker died or whose TTL elapsed.

## Animal husbandry v2 — draftwork (`draftwork.rs`)

Plow + cart halves both shipped end-to-end via the standard `JobBoard` + `JobClaim` pipeline.

1. **Animal training (`animal_training_progress_system`, Sequential, `TICKS_PER_DAY/4` cadence).** Per housed `DomesticAnimal` (`Tamed` + `preferred_home.is_some()`), bumps `training += TRAINING_INCREMENT_PER_PASS` (saturating). Cattle / Horse cross `TRAINING_THRESHOLD_DRAFT = 80` in ~80 days (~3 seasons). Pig / Dog / Cat tick the same field but are filtered out by the dispatcher's species gate.
2. **Chief plow posting (`jobs::chief_job_posting_system` Plow branch, Economy, daily Spring).** Per settled non-SOLO faction with `ARD_PLOW` tech AND ≥1 `ard_plow` in faction storage, walks state-owned Agricultural plots and emits one `JobKind::Plow` per plot with `plowed_year != Some(current_year)`. Posting shape: `JobProgress::Plow { plot_id, area, plowed_tiles: 0, target_tiles: plot.rect.area(), assigned_worker: FarmPlotAssignments.assigned_farmer(plot), animal: None }`. `posting_target_workers(Plow) = 1` (single farmer + single ox). Same-plot dedupe via a per-pass `plow_already_posted` set. Reachability gated by `rect_reachable_from_home`.
3. **Claim (`jobs::job_claim_system`).** Workers with `AgentGoal::Farm` claim the posting via standard U_bid scoring. `assigned_worker` gate (mirrors Farm's `assigned_farmer`) restricts claiming to one farmer — only that farmer can pick up the posting. `JobKind::Plow → AgentGoal::Farm` mapping in `JobKind::to_goal()`.
4. **Plow dispatcher (`htn_plow_dispatch_system`, ParallelB).** For each `JobClaim::Plow` holder on goal `Farm` with an idle `ActionQueue`, reads the posting, picks the next un-plowed tile via row-major scan keyed on `posting.plowed_tiles` (`tile_at_index(area.min, area.max, n)`), then picks an animal via `pick_idle_draft_animal`: prefers an un-claimed trained Cattle / Horse owned by the faction; **falls back to `None` (human-drawn) when no eligible draft animal exists**. Routes the worker via `assign_task_with_routing(TaskKind::Plow)` and dispatches `Task::Plow { plot_entity, animal: Option<Entity> }`. When the dispatcher newly commits to an ox (and the posting wasn't already tracking it), it stamps `posting.animal = Some(ox)` AND inserts `AnimalWorkClaim { use_kind: Plow, expires_tick: now + PLOW_CLAIM_TTL_TICKS }`. Human-drawn dispatches leave `posting.animal` as `None` so the *next* dispatch can upgrade if an ox becomes free mid-job. If a stamped animal despawns, the dispatcher un-stamps and falls through to a fresh search (which may downgrade to human-drawn).
5. **Plow executor (`plow_task_system`, Sequential, before `gather::gather_system`).** Per-tile work: each `Task::Plow` covers exactly one tile. Tile cost depends on the mode: `plow_work_ticks(animal) = 6` for ox-drawn (`Some`), `12` for human-drawn (`None`) — historically peasants without draft animals yoked themselves to the ard and made slower progress. Accumulates `work_progress` until the per-mode threshold, then credits `posting.plowed_tiles += 1` and `aq.finish_task`. The dispatcher's next pass picks the next tile. On the final tile (`plowed_tiles >= target_tiles`): stamps `Plot.plowed_year = Some(calendar.year as u16)`, calls `record_progress_filtered(JobKind::Plow, None, 0)` to fire `JobCompletedEvent { completed: true }` + despawn the posting, releases the `AnimalWorkClaim` (only when `animal.is_some()`), removes the worker's `JobClaim`, grants 6 completion XP. Per-tile XP +2; defence-in-depth cancel on vanished plot or typed-channel mismatch. **Both modes stamp the same `Tilled` marker** — the 1.4× harvest bonus applies whether the field was plowed by ox or by hand; the difference is throughput, not crop quality.
6. **Tilled marker stamp (`production_system` Planter branch).** When a Grain plant is spawned on a tile inside a plot with `plowed_year == Some(calendar.year)`, the plant entity gets a `Tilled` marker (`draftwork::Tilled`). Other plant kinds skip the marker.
7. **Yield bonus (`gather_system` Grain branch).** When the harvested Plant entity carries `Tilled` (read via `GatherRoutingResources.tilled_q`), the nutrient-tier base yield is scaled by `apply_plow_yield_bonus(base) = base * 7 / 5` (1.4× in integer math). Bonus stacks with `food_mul` from faction activity multipliers.

**Plot tillage state.** `Plot.plowed_year: Option<u16>` (`land.rs`) is the durable per-plot stamp. Re-plowing next spring overwrites it; no explicit cleanup needed — the harvest path reads the `Tilled` marker on the Plant (set at planting), not the plot field, so last year's stragglers don't get retroactively tilled.

**Ard Plow implement (`ard_plow` resource + recipe 14).** Wood×3 + Tools×1 at a Workbench, `ARD_PLOW`-gated. Tool-class, `two_hand` bulk, sprite reuses `item_hammer`. Posting gates on `faction.storage.stock_of(ard_plow) > 0` — a Chalcolithic faction that researches `ARD_PLOW` but hasn't crafted a plow still plants the old way (no Plow posting, no Tilled marker, no 1.4× bonus). Plow not consumed at use.

**Preserve-arm.** `goal_dispatch_system` preserves `(AgentGoal::Farm, TaskKind::Plow)` so mid-job ticks don't stale-reset the Task::Plow chain between Sequential and the next ParallelB dispatcher pass.

**Chief funding.** Plow postings go through `chief_post_funding_system` like other chief postings — Mixed/Market factions debit treasury at posting time and pay claimants on completion via the standard `JobEscrow` + `job_payout_system` path. Subsistence factions stay unpaid (communal labor).

**Plan complete (`plans/animal-husbandry-v2-draftwork.md`).** Plow + cart halves shipped; the v2.2 close-out fixed the Stables-intent emitter and added behavioural fixture tests. Plow + plant stay two separate postings — inline plow+plant was the design's rejected alternative, not deferred work. Bulk cargo hauling and player-direct hitch/unhitch are owned by the Vehicle system (below).

## Vehicle system (`vehicle.rs`) — complete (Phases 1-7)

Freeform customizable-vehicle subsystem (`plans/vehicle-system.md`; v2 large-grid + variants + weapon modules per `plans/evaluate-the-users-xiao1-civgame-plans-v-silly-lobster.md`). **Replaces the retired `cart.rs`** (deleted in Phase 4). Designs live on a bounded **3D cell grid** (`GRID_MAX` **10 wide × 8 deep × 6 tall** since v2) — height is load-bearing for mass / stability / clearance / combat.

- **Data model.** `VehiclePartKind` (Frame/Deck/Wall/Axle/Wheel/Hitch/Yoke/CargoBay/CrewSeat/WeaponMount/Engine/Track/ArmorPlate/Turret), `VehicleCell { kind, material: ResourceId, durability, variant: Option<VehiclePartVariantId>, module_id: Option<VehicleModuleId> }` (still `Copy`), `VehicleGrid { cells: Vec<(IVec3, VehicleCell)>, modules: Vec<VehicleModuleInstance> }`, `VehicleDesign`, `VehiclePurpose`, `VehicleStats`, `VehicleFootprint` (XY offsets pre-rotated for 4 headings), `VehicleInventory` (the add/take/qty API), plus `Vehicle`/`VehicleCrew`/`VehicleDraft`/`VehicleState` components.
- **Variants (v2).** `PartVariantDef` (`core.ron` `variants:`) gives a part kind a behavioural identity — `mass/support/traction/turn_radius/cargo_volume/durability/engine_power` multipliers (default `1.0` = identity). `VehicleCell.variant = None` = the plain part. `variant_of` resolves it; `cell_mass_g` / `support_limit_g` / `derive_stats` / `cell_durability(kind, material, variant, data)` fold the multipliers in (turn radius is a *product* over all cells). Some variants carry `tech_gates` (`iron_rim_wheel`/`metal_track` → `bronze_casting`).
- **Weapon modules (v2).** `VehicleModuleDef` (`core.ron` `modules:`) is a multi-cell weapon that fires/strikes **once per module** — `footprint`, `firing_arc` (`None`/`Front90`/`Full360`), `range`/`damage`/`siege_damage`, `crew_required`/`gunner_required`, `cooldown_ticks`, `required_support`, `extra_bill`, `tech_gates`. A placed `VehicleModuleInstance { id, def, cells, facing }` lives on `VehicleGrid.modules`; owned cells back-reference it via `module_id`. `cells` stays the single occupancy/footprint/mass/health source — modules are grouping metadata. `validate_grid` adds module checks (`ModuleOutOfBounds`/`ModuleOverlap`/`UnsupportedModule`/`BadModuleFacing` — a `Front90` module must touch the body's front edge). `collect_design_tech_gates` unions every part/variant/module gate; custom designs register with the derived gate set (no longer `Vec::new()`), and `vehicle_assembly_system` enforces the faction knows them all before consuming the bill.
- **Catalog.** `assets/data/vehicles/core.ron` (materials keyed by catalog `ResourceId`; part defs; stock templates Handcart / Ox Cart / Four-Wheel Wagon / Light Chariot / War Chariot). `load_vehicle_assets()` (tolerant loader, mirrors the archetype loader) builds `VehicleData` + `VehicleDesignRegistry`, inserted in `SimulationPlugin::build`.
- **`validate_design`** — deterministic 3D checks (connectivity, floating cells, wheel↔axle, control cell, draft capacity, axle overload, cargo reachability, chariot rule). **`derive_stats`** — mass / 3D center-of-mass / `stability = track_width/com.z` / payload / draft power / speed caps, all from the cell bill-of-materials.
- **VehicleYard** (`BuildSiteKind::VehicleYard`, 12 wood + 6 stone, `ANIMAL_HUSBANDRY`) — assembly + parking anchor; `VehicleYardMap` on the `PenMap` hook pattern. `HitchingPost.parked_cart` renamed `parked_vehicle`.
- **Assembly.** `PlayerCommand::QueueVehicle { design_id }` (faction-level, empty `actors`) → `VehicleAssemblyQueue`; right-click a player VehicleYard → "Assemble Vehicle ▸" submenu. `vehicle_assembly_system` (Economy, 60-tick cadence) drains the queue: `design_bill` (one unit per cell material + a `tools` per mechanical cell) checked against faction storage, consumed, then a parked `Vehicle` is spawned at the yard.

- **Pathfinding (Phase 3, wired Phase 4).** `pathfinding::vehicle_path::footprint_astar` — heading-aware A* over `VehicleNode { x, y, z, heading }`. Successors: forward / forward-diagonal translates (z ±1) + turn-in-place (cost from `turn_radius`); each candidate rotates the footprint to the heading and tests every cell via a generic `cell_ok(x,y,z)` closure. The live caller is `vehicle::plan_vehicle_route`, folding `ChunkMap::passable_at` + `vertical_clearance_at` + `VehicleOccupancyIndex` (excluding the vehicle itself). `ChunkMap::vertical_clearance_at(x,y)` counts open `Air`/`Ramp` Z-levels above the surface — clearance gating for tall vehicles.
- **Occupancy (Phase 3).** `VehicleOccupancyIndex` (tile → `Vehicle` entity, 2D) — full rebuild each tick by `vehicle_occupancy_sync_system` (Sequential, after `sync_indexed_after_move_system`). `footprint_tiles(design, anchor, heading)` is the shared footprint enumerator.
- **Rollover (Phase 3).** Pure functions: `step_tip_torque(stats, RolloverContext)` scales turn/slope/rough/overload disturbance by COM height; `vehicle_rolls_over(stats, torque)` compares accumulated torque to `stability`. Tuned so a road-bound loaded ox cart never tips while a tall narrow overloaded design on a slope does.
- **Cargo hauling (Phase 4) — vehicle-leads movement.** Replaces the retired cart pipeline. The **vehicle is the authoritative mover**: it owns a `VehiclePathFollow { path: Vec<VehicleNode>, cursor, tip_torque }` planned by `footprint_astar` (clearance- + occupancy-checked); the driver and draft animals ride it.
  - **`htn_vehicle_haul_dispatch_system`** (ParallelB, before the hand-haul dispatcher) — for an idle `JobClaim::Haul` holder whose posting still needs `>= VEHICLE_HAUL_MIN_REMAINING = 12` units, claims an idle owner-faction cargo vehicle (`Vehicle.hauler == None`, `design_is_cargo_capable`, not `Overturned`) + `design.required_animals` trained Cattle/Horse onto `VehicleDraft.hitched`, then routes the worker **on foot** (plain person A*, `assign_task_with_routing`) to the vehicle tile so they can board it.
  - **`vehicle_cargo_haul_task_system`** (Sequential) — boards the worker (`BoardedVehicle`) once they reach the vehicle (`ai.state == Working`), then drives the two-phase state machine: plans the vehicle's `footprint_astar` route via `plan_vehicle_route` (tries the target tile + surrounding rings as goal anchors); on arrival (`footprint_reaches`) transfers `min(blueprint need, capacity_units(payload))` into `VehicleInventory` (load) / deposits + credits the posting via `record_progress_filtered` (deliver). On completion releases draft claims, re-parks (`hauler = None`), unboards the worker, drops the `JobClaim`.
  - **`vehicle_movement_system`** (Sequential, after `movement_system`) steps each vehicle along `VehiclePathFollow` at its terrain speed cap, updating `anchor_tile`/`heading`/`z` and accumulating rollover `tip_torque` per node step; removes the component on arrival. Holds when the driver is Dormant.
  - **`vehicle_rollover_system`** — overturns a vehicle whose `tip_torque` beats `stability`: `VehicleState::Overturned`, ejects the driver (`BoardedVehicle` removed, placed on an adjacent tile, fall damage `height_z × 8`), spills `VehicleInventory` as `GroundItem`s, releases draft animals.
  - **`vehicle_crew_sync_system`** snaps the boarded driver + hitched animals to the vehicle each tick. `movement_system` / `recover_stranded_agents_system` skip `Without<BoardedVehicle>`.
  - **`vehicle_haul_recovery_system`** (Economy, 60-tick cadence) re-pools a vehicle whose driver was lost mid-haul.
  - `Task::VehicleCargoHaul { vehicle, blueprint, resource_id }` / `TaskKind::VehicleCargoHaul = 54`; `goal_dispatch_system` preserves `(AgentGoal::Haul, TaskKind::VehicleCargoHaul)`. `compute_faction_storage_system` folds `VehicleInventory` into faction `storage.totals`. `entity_sprites::spawn_vehicle_sprites` attaches the sprite. Sub-12-unit hauls still hand-carry.

- **AI provisioning (Phase 5).** `vehicle_yard_intent_emitter_system` (Economy, daily) drops a `VehicleYard` blueprint for a settled `ANIMAL_HUSBANDRY` faction with `≥ YARD_MIN_MEMBERS = 10` members and no yard (built or pending). `vehicle_ai_queue_system` (Economy, daily) enqueues a conservative stock template (Handcart) into `VehicleAssemblyQueue` for a faction that owns a built yard but no vehicle — idempotent while one is pending. Inspector: a selected `Vehicle` shows a "Vehicle" `CollapsingHeader` (design, state, footprint, mass/payload, speed, stability, cargo, driver, draft, route) via the nested `VehicleInspectorParams`.
- **Freeform designer (Phase 5; v2 workbench).** `ui/vehicle_designer.rs` — an `egui::Window` (`VehicleDesignerState` resource, HUD "Vehicles" toggle), a three-column workbench: left = part palette + variant picker + material `ComboBox` + a `Module` mode with module palette / rotation / footprint preview; centre = Z-layer tabs + the `10×8` slice grid; right = live `derive_stats` / `validate_grid` / `design_bill`, a "What this vehicle can do" summary, and a tech-gate line. Hover text on every part / variant / module / material / stat / validation error / occupied cell. `EditMode::{Cell, Module}` — cell mode places one part cell (kind + variant), module mode stamps a whole module footprint; right-click on a module cell removes the whole module. Queue is disabled when the design is invalid **or** the player faction lacks a required tech. Queue emits the faction-level `PlayerCommand::QueueCustomVehicle { name, grid, purpose, required_animals }`; `drain_player_command_events_system` derives `tech_gates` via `collect_design_tech_gates`, registers the design (`author_faction = player`), and enqueues it. "Saved & proposed designs" lists player-faction-authored designs with a Load button.
- **AI design proposals (Phase 5).** `vehicle_ai_design_proposal_system` (Economy, weekly) generates one metal-reinforced freeform design per faction with `ANIMAL_HUSBANDRY` + `BRONZE_CASTING` (below metalworking a "freeform" all-wood design equals a stock template). Upgrades the tech-best stock template's wheels + axles to copper; stored in `VehicleDesignRegistry` with `author_faction`, **not** auto-queued — proposals are browse/edit/queue fodder for the designer. Idempotent: one authored design per faction.
- **Rendering (Phase 5).** `entity_sprites::spawn_vehicle_sprites` keys off the design: a stock 1-tall design (`author_faction == None`, no Z-extent) reuses the hand-drawn `entity_cart` sprite; a freeform / multi-Z design renders as a **composed** body — one `VisualChild` colored quad per cell (`vehicle_cell_color` part-kind palette), XY-placed and lifted by grid-Z so a tall body extends upward.
- **Player vehicle orders (Phase 5).** Right-clicking a player-faction `Vehicle` (or selecting one and clicking a destination) opens a "Vehicle ▸" menu: Move / Load / Unload / Assign Crew / Hitch / Unhitch / Salvage, plus Right for an overturned hulk. Each emits a faction-level `PlayerCommand::VehicleOrder { vehicle, kind: VehicleOrderKind }`; `drain_player_command_events_system` queues it onto `PendingVehicleOps`; `vehicle_player_command_system` (Sequential, before `vehicle_movement_system`) applies it directly — `MoveTo` plans a driverless `footprint_astar` route, `Right` flips `Overturned→Parked`, `Load`/`Unload` move cargo to/from the nearest storage tile, `Hitch`/`Unhitch` claim/release draft animals, `Deconstruct` refunds half the design bill and despawns.

- **Chariot crew, draft & combat (Phase 6).** `VehicleHealth` (component, on every spawned `Vehicle`) is a per-cell live-health mirror of the design's cell durabilities plus a `disabled: VehicleDisableFlags` bitset. Attacks aimed at a `Vehicle` resolve **against individual cells**:
  - `combat_system` gains a `vehicle_query` so a bodiless `Vehicle` target is recognised as live (not cleared); `vehicle_combat_system` (Sequential, after `combat_system`) does the per-cell resolution. `pick_hit_cell` is **height-weighted toward the low cells** — melee biases wheels/axles. `apply_vehicle_cell_damage` applies damage and, on a destroyed cell, returns a `VehicleHitOutcome`.
  - Destroyed **Wheel/Axle** → `VehicleDisableFlags::MOVEMENT` (route dropped, vehicle parks — the intended ancient counterplay); **CargoBay** → cargo spills as `GroundItem`s; **Hitch/Yoke** → draft team released; **CrewSeat** → an occupant ejected; a destroyed **low support cell** (Frame/Deck/Axle on the bottom Z) overturns the vehicle. When the **last cell** falls the vehicle is *wrecked* — cargo/animals/crew released, hulk despawned.
  - **Crew** are a mobile platform: `AssignCrew` seats a driver + passengers up to `crew_seat_count` (the `CrewSeat` cell count) and boards each via `BoardedVehicle`; `vehicle_crew_sync_system` snaps every rider (hauler + driver + passengers + gunners) to the vehicle; crew fight any adjacent foe through the **normal `combat_system` rules** — no new combat code. `vehicle_rollover_system` / `Deconstruct` eject the seated crew.
  - `WeaponMount` cells are modelled + validated (mechanical cell in the bill, `war_chariot`-gated); advanced ranged/turret behaviour stays dormant.

The vehicle system is **complete** — `cart.rs` deleted in Phase 4. Simplifications vs. the plan: Phase 2 ships an autonomous `vehicle_assembly_system` (the old `cart_assembly_system` precedent) rather than a full `JobKind::Assemble` worker pipeline; a vehicle re-parks in place after a haul (no drive-back-to-yard); rollover *recovery* is a player `Right` order (autonomous AI righting deferred); Phase 6 crew fight through stock combat rules without bespoke vehicle damage modifiers.

### Tank / siege content (`plans/vehicle-system-tanks.md`)

The reserved `Engine/Track/ArmorPlate/Turret` parts are now active.

- **New parts (`core.ron`):** `engine` (abstract powered traction — non-zero `engine_power_g`, gate `powered_traction`), `track` (`traction_pct` cuts rolling resistance + load-bearing like a broad axle, gate `armor_plating`), `armor_plate` (`armor_durability_mult` 2.5× per-cell HP, gate `armor_plating`), `turret` (1 gunner + `mounted_weapon_range`/`mounted_weapon_damage`, gate `siege_engineering`). `PartDef` gained these `#[serde(default)]` fields; `cell_durability(kind, material, data)` now folds `armor_durability_mult` in.
- **New templates:** `Battering Ram`, `Siege Tower` (multi-Z), `Armored Wagon`, `Ballista Vehicle`, `Tank` (Engine + Track, 0 draft animals).
- **`derive_stats`:** sums `engine_power` over Engine cells (`is_engine_driven`); engine-driven speed caps derive from power-to-mass; Track `traction_pct` reduces resistance + raises offroad speed; `Track` adds to `support_limit_g`. `is_structural()` now includes `Engine`/`Track`.
- **`validate_grid`:** an Engine design skips the Hitch/Yoke draft check; `DesignError::UnderpoweredEngine` when `engine_power < draft_power_needed`.
- **Turret fire (`vehicle_turret_fire_system`, Sequential):** iterates weapon **modules** first — a multi-cell ballista/turret fires **one** projectile per `cooldown_ticks` (keyed `(vehicle, VehicleModuleId)`), gated on `gunner_required` (pool = `gunners + passengers`, since `AssignCrew` never seats `gunners`), every required cell's health, and the firing arc. `FiringArc::Front90` filters targets to a ±45° front cone via `target_in_arc(heading_vec(...), origin, target)`; `Full360` keeps the box scan. A legacy fallback then fires any single-cell `Turret`/`WeaponMount` with `module_id == None` on the old per-cell cooldown.
- **Siege (`vehicle_siege_system`, Sequential):** a crewed, siege-capable vehicle carrying a `SiegeOrder { target_tile }` parked chebyshev-1 adjacent to a `WallMap` wall ticks `construction::apply_wall_damage` on a cooldown. `design_is_siege_capable(design, data)` / `design_siege_damage(design, data)` now key on **ram modules** (a module with `siege_damage > 0`) — a bare `WeaponMount` cell no longer makes a design siege-capable, so a War Chariot is not a siege engine. `VehicleOrderKind::SiegeWall((x,y))` stamps the `SiegeOrder`. Raider AI does not yet use siege autonomously.

## Animal spawn distribution (`animals.rs::spawn_animals`)

Initial-condition only. Per-species `SocialPattern`: `HERD` (Deer/Horse/Cow, 8–15, r=5), `PACK` (Wolf/Pig/Rabbit/Fox, 3–6, r=3), `SOLITARY` (Cat). `cluster_spawn_tiles` pops shuffled centers from species biome pool and lays members within `cluster_radius` and stamps `HerdMember { cluster_id }` on every HERD member (Deer/Horse/Cow). `HerdClusterGen` issues monotonic ids; `wild_herd::seed_wild_herds_system` also draws cluster ids from it so wild-herd bloomed individuals share the same flow-field plumbing. `animal_reproduction_system` propagates the mother's cluster id to newborn HERD calves; `wild_herd::spawn_herd_members` propagates the wild herd's `cluster_id` at bloom.

## Animal pathfinding (`animal_paths.rs` + `animal_movement_system`)

Per project convention — per-agent local nav is A\*, flow fields reserved for many-agent shared goals.

- **Per-tile crossing gate (always):** `animal_movement_system` consults `chunk_map.passable_at(nx, ny, nz as i32)` before each tile step; on reject (world changed under a cached plan) it clears the path, demotes to `Wander`, snaps to the current tile, and zeroes `wander_timer` so the next bucket-active pass repicks. Replaces the prior obstacle-blind world-space lerp that walked across Water/River/Wall.
- **Solo + PACK species** (Wolf/Fox/Cat/Pig/Rabbit) — inline short-horizon A\* via `pathfinding::astar::find_path_in` with `ANIMAL_PATH_BUDGET = 256` nodes. Scratch lives in `AStarPool` index `2` (movement worker uses 0, person path worker uses 1). Cached on `AnimalAI.path` + `path_cursor` + `path_goal`; replan triggered by empty path / drifted `target_tile` / consumed cursor. `BudgetExhausted` keeps the partial; `Unreachable` falls into Wander stall.
- **HERD species** (Deer/Horse/Cow) — per-herd flow fields managed by `animal_paths.rs`:
  - `HerdClusterRegistry` (Resource) keyed by `cluster_id`. Each entry tracks `center_tile`, `members` (bloomed count), `cohesion_field`, `repulsion_field`, and the active `repulsion_threat_tile`.
  - `herd_cluster_update_system` (Economy, daily): recomputes `center_tile` as the mean of bloomed member positions; invalidates `cohesion_field` when the centre drifts > `HERD_REBUILD_DRIFT = 4`; evicts clusters with zero bloomed members.
  - `herd_cohesion_field_system` (ParallelA, per tick, lazy): builds at most one missing cohesion field per tick via `pathfinding::flow_field::build_flow_field` with goal = `center_tile`, no extra cost. Single-chunk field anchored on the centre's chunk; HERD members outside that chunk fall through to A\*.
  - `herd_threat_detect_system` (ParallelA, every 20 ticks): walks bloomed members; for each cluster picks the nearest Wolf/Fox within `HERD_THREAT_RADIUS = 12` via `SpatialIndex`. On hit, rebuilds `repulsion_field` with goal = a safe anchor projected away from the threat through the herd centre + `extra_cost` penalty of `HERD_REPULSION_PENALTY = 800` within `HERD_REPULSION_RADIUS = 4` of the threat. Repulsion field expires `HERD_THREAT_COOLDOWN = 100` ticks after the last sighting.
  - **Consumption in `animal_movement_system`:** for HERD species with a `HerdMember`, the replan branch tries `try_replan_via_flow_field` first — `Flee` consumes the repulsion field, `Wander` consumes cohesion, other states fall through to A\*. Cross-chunk targets always fall through to A\*. Path-following loop is identical for both planners.
- Path storage on `AnimalAI` is `Vec<(i32,i32,i8)>`; the type lost `Copy` (Vec) but keeps `Clone`/`Default`. No callers were relying on Copy.

## Nomadic mode

`Lifestyle::{Settled (default), Nomadic}` on `FactionData`. Nomads skip Settlement spawning, run a camp pipeline, migrate seasonally. AI commits atomically (Surveying → PendingCommit → PackingCamp → Traveling → PitchingCamp); player nomads transition via `PlayerCommand::PackCamp` / `PitchCamp`.

- **`Deployable { packed_form, refund_pct, refund_resource, refund_qty }`** (`pack_deploy.rs`) on every nomadic structure. Bedroll (1 skin + 2 wood) and Yurt (8 wood + 6 skin, gated on `PORTABLE_DWELLINGS`) are `fully_packable`; Tent (6 wood + 3 skin) is `refund_only(0.5, wood, 6)`.
- **Build sites:** `BuildSiteKind::{Bedroll, Tent, Yurt}`. Bedroll finalizes as `(Bed { tier: Crude }, Deployable, StructureLabel("Bedroll"))`. Yurt packs into `packed_yurt` good.
- **Camp seeding (`seed_nomadic_camp`):** hearth → bedrolls (radial 2..=5, one per founder), tents (outer 5..=7, ~1 per 4), yurts (Neolithic+, inner 3..=5, cap 2).

### Lifecycle

`FactionData.camp_state: CampState::{Pitched, Packed { since_tick }}` + `migration_phase: MigrationPhase::{Idle, Surveying, PendingCommit, PackingCamp, Traveling, PitchingCamp}` + `nomad_autopilot: bool`. AI runs the FSM; player nomads (`autopilot == false`) skip it for free-form Pack → Move orders → Pitch Camp Here.

**AI migration pipeline (`nomad.rs`):**

- `nomad_migration_system` (Economy, daily, exclusive): **strict-AND trigger** — a band enters `Surveying` only when stored food `< members × NOMAD_TRIGGER_FOOD_DAYS (3.0)` **AND** known local food clusters score `< members × NOMAD_KNOWN_FOOD_TARGET_PER_MEMBER (1.0)`. Neither signal alone uproots the band; there is no hunger override. Cooldown is capability-derived via `migration_cooldown_ticks(caps.home)` (a `Mobile` archetype's `migration_period_min_days`), not a flat `TICKS_PER_SEASON`. Picks 2–3 lowest-need members, stamps `AgentGoal::Scout` + `ScoutAssignment`, transitions to `Surveying`.
- `nomad_survey_dispatch_system` (ParallelB) routes Scouts via `Task::Explore { kind: AnyEdible }` through `SCOUT_CHECKPOINT_STEP = 24` chebyshev checkpoints so maintenance can run between long scouting legs.
- `nomad_survey_completion_system` (Economy daily, exclusive): at `SURVEY_WINDOW_TICKS = TICKS_PER_DAY*4`, runs `pick_migration_candidates`. If the best candidate scores `>= NOMAD_MIN_ACCEPTABLE_SITE_SCORE (15.0)` → write `pending_migration`, `PendingCommit`. Otherwise → return to `Idle`, no `pending_migration`, schedule a retry ~`NOMAD_NO_CANDIDATE_RETRY_DAYS (2)` days out (by back-dating `last_migration_tick`). The blind `fallback_direction` autopilot commit is **gone** — `fallback_direction` is retained `#[allow(dead_code)]` for manual/debug tooling only.
- `pick_migration_candidates` score = food + herd + water + biome-season + danger + recency − `d * DIST_WEIGHT (0.4)`. `NOMAD_MIN_TARGET_DIST = 8`, max `200`. **Herd visibility is knowledge-gated** — herd score comes from `MemoryKind::HerdSighting` clusters in faction-tier `SharedKnowledge` (seeded by `memory::vision_system` when a scout sees a Deer/Horse/Cattle), **not** an omniscient `WildHerdRegistry` scan. Candidate centre tiles that are `Water`/`River`/`Wall`/`Ore` are early-rejected before scoring.
- `score_danger` penalises `MemoryKind::HostileFactionSighting` clusters only. It must **not** read `MemoryKind::Prey` — `Prey` mixes predators and game, so a deer meadow is good grazing, never a threat.
- `nomad_migration_commit_system` (Sequential, exclusive) is a lifecycle driver: `PendingCommit` validates target → `PackingCamp` flips `camp_state = Packed`, dispatches `UnpitchStructure` labor for Deployables + no-cargo Campfires → `Traveling` stamps every member with `MigrationTarget`, inserts `FollowingBand` on owned `Tamed`, keeps `home_tile` at old camp → `PitchingCamp` begins on arrival; only then `home_tile` moves, recent_camps updates, `CampMoved` log fires.
- `MigrationTarget { tile, route_tile, started_tick, last_dispatched_tick, bounce_count }` + `WalkReason::Migration` + `TaskKind::Migrate`. `bounce_count` cap 2 reroutes via reachable peer/centroid.
- `nomad_migration_dispatch_system` (ParallelB, `ParamSet`): no `Without<Drafted>` filter — drafted hunters migrate with the band.
- `nomad_migration_arrival_system` (Sequential, after movement): strips marker on `MIGRATE_ARRIVAL_RADIUS = 4`, `MIGRATE_TIMEOUT_TICKS = TICKS_PER_DAY*3`, or `MIGRATE_STALL_TICKS = TICKS_PER_DAY/2`.
- `goal_dispatch_system` preserve-arms: `(MigrateToCamp, Migrate)`, `(Scout, Explore)`.

**Player Pack / Pitch:** chief-actor faction-scoped commands write to `PendingCampOps`. `apply_pack_camp_command_system` runs observable HTN labor — stamps non-drafted members with `PackingDuty` (preempts task chains, releases reservations, drops `JobClaim`), enumerates `Deployable`s within `seed_nomadic_camp_extent` of `home_tile`, dispatches `Task::UnpitchStructure` to nearest worker. `unpitch_structure_task_system` (`nomad_pack_labor.rs`) accumulates to `UNPITCH_WORK_TICKS = 40`, despawns, drops packed_form / packed_bundles / refund as `GroundItem`s. `camp_state = Packed` flips immediately. `apply_pitch_camp_command_system` keeps the synchronous `seed_nomadic_camp` call.

**Manual scout dispatch (`PlayerCommand::SendScout { direction, range }`):** chief-actor command queues `PendingManualScout`; stamps `ScoutAssignment { kind: ScoutKind::PlayerManual { faction_id }, … }`. `manual_scout_completion_system` runs `pick_migration_candidates(intent, k=4)` on arrival; folds into `FactionData.candidate_sites` (ring cap 16). `PlayerCommand::SetMigrationIntent` writes `faction.migration_intent`; AI consumes `intent.weights()`.

**Intent-weighted scoring** — six intents: `FreeRoute` (uniform), `FollowWater` (water×2), `FollowHerds` (herd×2), `SeekWinterShelter` (biome×2.5, dist×0.8), `SeekSummerPasture` (food×1.5, biome×2), `AvoidDanger` (danger×3). Candidates carry per-tile `reasons: Vec<CandidateReason>`.

**Cargo manifest / pitch repair:** `FactionData.cargo_manifest: CampCargoManifest { required, loaded, abandoned, deployed, pitching_started_tick, repair_unlocked }`. AI final-destination pitch unloads carried `bedroll`/`packed_yurt`/`wood`/`skin` near the target, consumes via `PitchStructureAt`. Once a minimal camp exists, migration completes after 80% arrival or `TICKS_PER_DAY/2` since pitch start; `camp_state = Pitched`; `nomad_chief_directive_system` repairs missing shelter. Temporary route waypoints never unlock repair.

**Mobile-state goal gate (`mobile_state_goal_gate_system`, ParallelA after `goal_update_system`):** `CampState::Packed` demotes settled-life goals + strips `JobClaim`. Player `Forage` uses `allowed_while_packed` (Survive/Sleep/GatherFood/Socialize/Defend/Rescue/Play/FollowingPlayerCommand/MigrateToCamp/Scout/care/Drink). AI caravan phases use stricter `allowed_while_ai_caravan` (Survive/Defend/Rescue/SeekCare/ProvideCare/Drink/FollowingPlayerCommand/MigrateToCamp).

**Player-locked packed migration:** `PackedMigrationAutonomy::{Hold, Forage}`. Player-driven nomads default to `Hold`, reset on every `PackCamp`. `Hold` strict-demotes everything except `FollowingPlayerCommand`, `PlayerManual` Scout, sim-owned `MigrateToCamp`. `Forage` falls through to legacy `allowed_while_packed`.

**Pitch conserves shelter count:** `apply_pitch_camp_command_system` sweeps Deployables within `seed_nomadic_camp_extent` of old home (Tents drop materials, fully-packable silently discard), then `seed_nomadic_camp` re-seeds and debits one `bedroll`/`packed_yurt` per spawned shelter from member then pack-animal inventories. First pitch is forgiven; subsequent cycles conserve exactly.

**Slim chief directives (`nomad_chief_directive_system`):** gates on `caps.home.is_mobile() && pending_migration.is_none()`. Targets via `nomad_shelter_targets(members)`. Spawns one Single-tile blueprint per kind per tick, cap `NOMAD_DIRECTIVE_BP_PER_TICK = 2`. No job postings.

**Band redistribution (`nomad_pool.rs`):** `nomad_band_pool_balance_system` shrinks max-min spread for `bedroll`/`packed_yurt`/`preserved_meat` to ≤1 unit across the band within `POOL_BAND_RADIUS = 12` chebyshev.

**Pack-animal logistics:** `PackAnimalInventory { items: [(ResourceId, u32); 6], capacity_g }` on `Tamed`. Capacities: Horse 60kg / Cow 80kg / Pig 30kg / Dog 15kg. `attach_pack_inventory_system` auto-inserts on `Added<Tamed>`. `compute_faction_storage_system` folds pack inventories into nomad faction `storage.totals`. `combat::death_system` drops contents as `GroundItem`s.

**Preserved meat ration:** `2 meat + 1 wood → 3 preserved_meat` (CraftRecipe 12, gated on `FOOD_SMOKING`). `eat_task_system` two-pass: fresh first, preserved only if nothing else on hand.

**Wild herds (`wild_herd.rs`):** `WildHerdRegistry` per-herd `(id, species, aggregate_count, leader_tile, range_center, bloomed, members, flee_until_tick, last_birthed_tick)`. `seed_wild_herds_system` (Startup) places `WILD_HERD_COUNT = 3` herds at grasslands, `WILD_HERD_AGGREGATE = 120` each. `wild_herd_migration_system` (daily): predator flee × 3 / water seek / camp avoidance × 4 / non-Winter birth +12 capped at 200. Bloom/collapse at camera distance 32/48 spawns up to 60 individuals; predation shrinks the herd across cycles.

**Sedentarization (`nomad_sedentarize_system`):** emits `SwitchArchetype` when `member_count ≥ 12`, no `pending_migration`, stable for ≥ `TICKS_PER_SEASON * 4`. **Reverse collapse (`sedentary_collapse.rs`):** **combined-failure** trigger — a settled faction is "failing" only when a sustained food deficit (`food_total < members × COLLAPSE_FOOD_DAYS=3.0`) coincides with a structural failure: population crash (`< SEDENTARY_COLLAPSE_MIN_MEMBERS=6`) OR shelter loss (faction-owned beds `< ceil(members × COLLAPSE_MIN_BED_COVERAGE=0.5)`). A single weak signal never collapses a village. `collapse_failing(food_deficit, pop_crash, shelter_loss)` is the pure predicate. `collapse_streak` must reach `COLLAPSE_TRIGGER_TICKS = TICKS_PER_SEASON × COLLAPSE_TRIGGER_SEASONS=2` (~10 in-game days) → `SwitchArchetype`. Bed counting (`usable_beds`) scans `COLLAPSE_BED_FALLBACK_RADIUS=32` around home and excludes beds owned by *other* factions (bed → assigned occupant → faction; unassigned beds count toward the home faction). On settled→nomadic, `handle_switch_archetype` resets the band to `CampState::Pitched` / `MigrationPhase::Idle`, clears `pending_migration`, and stamps `last_migration_tick = now` so it can't immediately re-survey.

## Pluralist Economy

Per-resource policy flags + per-settlement markets + sub-faction households + Bureaucrat/Trader/Healer professions + P2P currency + escrow + U_bid scoring + tribute + craft contracts. Currency invariant: `EconomicAgent.currency + FactionData.treasury + Settlement.treasury + JobEscrow.held()` is conserved across every operation (`held() = amount` wage `+ purchase_pool` Market-haul procurement capital).

### Settlements (`settlement.rs`)

The **economic** unit (market + treasury + market_tile), distinct from `SettlementPlan` (layout). One faction can own many settlements.

- `Settlement` (Component): `id`, `owner_faction`, `market_tile`, `founding_tick`, `name`, `treasury: f32`, `market: SettlementMarket`, `peak_population`.
- `SettlementId(u32)`; `SettlementMap` (Resource) with `by_id` / `by_megachunk` / `by_faction` indices.
- `auto_found_default_settlements_system` (FixedUpdate Economy + OnEnter, idempotent) spawns one Settlement per non-SOLO faction at its `home_tile`.

### Currency + escrow

- **`pay(world, from, to, amount) -> bool`** (`transactions.rs`) — atomic agent-to-agent transfer. Only sanctioned way to move currency.
- **`FactionData.treasury: f32`** — faction-level wealth pool.
- **`JobEscrow { amount, beneficiary, purchase_pool }`** sidecar per funded posting. Producer debits wallet + spawns sidecar. `job_payout_system` (Economy, exclusive, after `job_claim_release_system`) splits `escrow.amount` (wage) across claimants on completion, then sets `amount = residual` and despawns — the `on_remove` hook refunds `held()` (wage residual **+ unspent `purchase_pool`**) to `beneficiary`, so Market-haul procurement capital reconciles through the same path. On failure the hook refunds the full `held()`. `chief_post_funding_system` sizes `purchase_pool = max_unit_price·target` for `HaulSource::Market` postings (0 otherwise) and debits `wage + purchase_pool` from treasury (per-candidate live-treasury debit serializes concurrent Market hauls). Workers get `Earnings(VecDeque<EarningEntry>)` ring (cap 16) + `ActivityEntryKind::WagePaid`.
- **Chief-funded postings:** `chief_post_funding_system` (Economy, exclusive, after `chief_job_posting_system`) computes `chief_wage_for(progress)` (Stockpile/Haul/Craft → `trade_base_value × qty × CHIEF_MARGIN (0.5)`, Haul ×0.5; Calories/Build/Planting → flat per-day, cap `CHIEF_BUILD_WAGE_CAP=30`), debits `faction.treasury`, spawns `JobEscrow`. Subsistence factions skip funding.
- **Wage signal:** `FactionData.wage_signal: AHashMap<(JobKind, Option<ResourceId>), WageEMA>`. `faction_wage_signal_system` (Economy, daily, after `job_payout_system`) folds last-day Earnings into EMA at `α ≈ 0.129` (5-day half-life). Cross-faction perception via `PerceivedFactionWages` (cap 32). `wage_gossip_system` piggybacks `Socialize` within `WAGE_GOSSIP_RADIUS = 3` with `exp(-age / TICKS_PER_DAY)` staleness.
- **Hook registration:** `JobsPlugin::build` registers `on_remove(on_job_escrow_remove)`.

### Per-resource policy (`economy/policy.rs`)

`ResourceControlPolicy` flags: `chief_allocates_labor`, `private_actors_allowed`, `state_sells_at_market`, `prices_fixed_by_state`, `fixed_price`. `Default` = all-communist; `capitalist()` preset. `FactionData.economic_policy` is empty by default. `Method::policy_gate` returns required flags; `method_passes_policy_gate(...)` is the check. SOLO agents reject any non-empty gate.

### Sub-factions / households

A household is a `FactionData` with `parent_faction = Some(village_id)`. Reuses the entire faction primitive.

- `FactionData.{parent_faction, household_head, children_factions}`.
- `FactionRegistry::spawn_household(parent, home_tile, head, &catalog)` — wires links, sets `household_head`. Non-empty parent `economic_policy` → household stamped `capitalist()`; empty → empty.
- `FactionRegistry::root_faction(id)` walks `parent_faction` to the village.
- **Formation:** `CoSleepTracker.bond_strength` accumulates ticks of cosleep with the same partner; at `HOUSEHOLD_BOND_THRESHOLD` (one game-week) `household_formation_system` spawns the household.
- **Market preset spawn seeding:** `spawn_population` calls `seed_market_households` after the per-faction loop. Every adult founds a one-person household with its own plot tile, dedicated `FactionStorageTile`, and `HOUSEHOLD_SEED_TREASURY (15.0)`.
- **Inheritance:** `pregnancy_system` threads the mother's `Option<&HouseholdMember>` to the newborn.
- **`HouseholdMember` is a marker only** — parents keep `FactionMember.faction_id` pointing at the village.

### Wage-aware labor market

End-to-end: chief-funded postings → escrow → worker payout → faction wage signal → EV-driven profession choice.

- **Capital recognition (`capital.rs`):** `capital_factor(...)` averages three axes in `[1.0, 2.0]`: `tool_capital_factor` (+0.5 when a held item maps via `tool_profession`: weapon→Hunter, tools→Crafter); `workshop_capital_factor` (+1.0 household-owned affine workshop within `WORKSHOP_AFFINITY_RADIUS = 12`, else +0.5 village-owned); `land_capital_factor` (Farmer: +0.5 for household-held Agricultural plot under non-StateOwned tenure). `OwnedBy { faction_id, kind, tile }` stamped at workshop finalize; add/remove hooks maintain `WorkshopOwnership`. `WorkshopKind::affine_to`: Market→Bureaucrat, Workbench/Loom→Crafter, Shrine→Healer.
- **Skill decay (`skills.rs`):** `Skills` clamped at `SKILL_MAX=255`; `SkillPeaks` + `SkillUseTicks` ratchet via `skill_peaks_tracker_system` (observer on `Changed<Skills>`). `skill_decay_system` (daily, half-life 90 days): mastered skills (peak ≥ `SKILL_MASTERY_LINE=80`) decay toward `SKILL_MASTERED_FLOOR=30`; below mastery toward `max(SKILL_FLOOR_BASE=5, peak × 0.30)`.
- **Profession choice (`profession_choice.rs`):** `expected_wage = aggregate_wage_per_day × skill_competence(primary_skill) × capital_factor`; `skill_competence(s) = 0.2 + (s/SKILL_MAX) × 0.8`. Target-driven assignment systems sort candidates by `(expected_wage, primary_skill)`. Shared `demote_profession_state` releases reservations, cancels tasks, strips `Carrying`.
- **Survival override:** below `FARMER_SURVIVAL_FLOOR = 16.0` per-head food, every non-Farmer assignment system zeros target, demoting Hunters/Bureaucrats/Crafters/Healers. Bands: <16 stand-down; 16–32 Farmer ramp; >32 × `FARMER_DEMOTE_RATIO (1.6)` shed excess. Apprentices stay bound (override zeros promotion target, not active links).
- **Asymmetric demotion hysteresis:** `HUNTER_DEMOTE_BUFFER = 1`, `BUREAUCRAT_DEMOTE_BUFFER = 1` add one-slot tolerance. Promotion eager; demotion only on `current > target + buffer`. `want == 0` (survival/treasury) bypasses the buffer. Crafter uses EMA-band hysteresis (`crafter_target_with_hysteresis`).
- **Cross-profession switcher (`cross_profession_switch_system`):** daily Economy pass; switches Hunter↔Bureaucrat↔Crafter when `EV(target) > EV(current) × EV_SWITCH_HYSTERESIS (1.20)`. `switching_cost_skill_regret = peak[primary]/SKILL_MAX × 0.20 × aggregate_wage(current)`. Faction caps via `faction_cap_for`. Sub-`APPRENTICE_THRESHOLD` Crafter targets route through `Apprentice`.
- **Apprenticeship (`apprenticeship.rs`):** novice-Crafter onboarding. `None → Crafter` with `Skills[Crafting] < APPRENTICE_THRESHOLD (30)` scans mentor pool (Crafter + `Skills[Crafting] >= MASTER_THRESHOLD (100)` + no `MentorOf`). Pairs `ApprenticeOf { mentor }` / `MentorOf { apprentice }` + `ApprenticeProgress { ticks, target_ticks: TICKS_PER_DAY × APPRENTICESHIP_DURATION_DAYS (30), target_profession }`. No mentor → fall back to direct Crafter. `apprentice_progress_system` (Economy, daily) ticks + graduates (lifts `Skills[Crafting]` to threshold, strips links, sets `*prof = target`). Stale mentor demotes orphan to `None`. Payouts split: apprentice `share × WAGE_FRACTION_APPRENTICE (0.4)`; mentor `share × WAGE_FRACTION_MENTOR_FEE (0.1)`; residual `0.5·share` refunds. Deliberate-practice 2× XP via `apprenticeship::xp_with_apprentice_bonus`.

### U_bid scoring at job-claim layer

`job_claim_system` branches on `posting.reward`:

- **Paid (`reward > 0.0`):** `U_bid = E(R) + priority_bonus + affinity_bonus − C_action − C_opportunity` where `E(R) = posting.reward × wealth_modifier(currency) × disposition.earn_income_multiplier()`; `disposition multiplier = 1.0 + entrepreneurial/255` (fallback 1.5); `wealth_modifier(c) = 1.0 + 0.5 / (c + 50)`; `priority_bonus = posting.priority * 0.01` (chief 200, household 180, individual 100); `C_action = euclidean × BID_DIST_DISCOUNT`; `C_opportunity = 0.0` (stub).
- **Unpaid (`reward == 0.0`, chief / legacy):** legacy `priority + skill + bias − distance`.
- **Affinity:** `CRAFTER_AFFINITY_BONUS = 3.0` lifts Crafter paid-Craft; unpaid `profession_bias` arms `(Crafter, Craft) = 0.5`, `(Crafter, Stockpile) = 0.1`.

## Land ownership (`land.rs`)

Plot-based ownership layer over the compatibility `SettlementPlan` projection generated from `SettlementBrain` parcels. Plots start `Tenure::StateOwned`; in Mixed/Market presets the listing system publishes them, households acquire what they can afford, rent collects monthly with eviction after two misses, sharecrop plots split harvest yield at gather time.

- **`Plot` (Component):** `id, settlement_id, faction_id, rect, z, zone_kind, tenure, holder, base_value, last_valued_tick, missed_payments, frontage_edge, access_tile, parent_plot`. `Tenure ∈ {StateOwned, Leased, Sharecropping, Freehold}`. `TenureHolder ∈ {State, Household}`.
- **`PlotIndex`:** `by_id`, `by_settlement`, `by_tile` (surface-only), `by_faction_hash`, `next_id`. `plot_at(x, y)` is the hot lookup.
- **`carve_plots_system`** (FixedUpdate Economy, after `settlement_planner_system`, before `chief_directive_system`; **also runs once at `OnEnter(Playing)`** in the unified survey → project → carve segment so tick-0 plots aren't torn down by the first runtime stagger — `plans/spawn-farm-seeding.md`): re-carves on `culture_hash` mismatch, values via `compute_plot_value`. Plot sizes: Residential 6×6, Crafting/Storage 4×4, Agricultural **16×16** (matches organic parcel size 1:1 — see Farming below); Civic/Sacred/Market/Defense kept whole. For Agricultural plots every tile lands in `PlotIndex.ag_tiles` (road protection covers the whole 256-tile rect) and gets a deterministic role via `field_tile_role(culture_hash, faction_id, tile, rect)` → `{Cropland (60%), CroplandLow (15%), SoilFallow (20%), GrassEdge (5%)}`; perimeter tiles bias toward GrassEdge by +0.30 so the field silhouette softens from a perfect golden square into a mottled blob with stubble at the edges. Idempotent across re-carves, stale teardown drops `ag_tiles` (the Cropland tile intentionally stays — abandoned fields look tilled). The OnEnter `seed_starting_farms_system` pass then stamps a bounded starter Cropland patch inside the carved plot.
- **`compute_plot_value`** (`PLOT_BASE_VALUE = 50.0`): `BASE × zone_mul × (centre_factor + home_factor) × terrain_factor`. Distance anchors on `market_tile` + `home_tile`; terrain samples fertility at centre + 4 corners.
- **`LandPolicy`** (`economy/policy.rs`): set by `land_policy_for(EconomyPreset)`. Subsistence all-false; Mixed = rent + sharecrop; Market = adds sale + freehold. `default_lease_period_days = 30`, `rent_yield_pct = 0.04`, `default_share_to_landlord = 0.30`.
- **`tile_buildable_by`**: rejects placements on plots held by another faction/household. Wired into `chief_directive_system` after candidate selection.
- **`land_listing_system`** (Economy, every `TICKS_PER_DAY/4`): publishes Sale + Lease listings for `StateOwned` plots when `LandPolicy` permits. Civic-class zones skipped. Per-(faction, zone-kind) caps (`RESIDENTIAL_LISTINGS_CAP=6`, `AGRICULTURAL_LISTINGS_CAP=6`, `CRAFTING_LISTINGS_CAP=4`); a plot triple-listing as Sale+Lease+Sharecrop counts as one slot. Sale asking = `base_value`; lease asking = `base_value × rent_yield_pct` per month, floored at `MIN_MONTHLY_RENT`.
- **`household_land_acquisition_system`** (Economy, daily): per-zone-kind acquisition track. Households browse parent listings for Residential and Agricultural separately; can hold one of each. Affordability: lease ≤ 40%, buy ≤ 70% of treasury. Preference: Sale → Lease → Sharecrop. Agricultural acquisitions prefer `Profession::Farmer`-headed households when any exist. Acquiring an Agricultural plot also spawns a household-private `FactionStorageTile` for private farm seed/harvest routing. Also claims nearest unowned same-village Agricultural plot within 12 chebyshev as child (mirrored tenure).
- **`rent_collection_system`** (Economy, every 30 game-days): Leased plots with expired `paid_through_tick` attempt household→landlord transfer. Failure bumps `missed_payments`; at `EVICTION_MISS_THRESHOLD = 2` → tenure `StateOwned`, emits `PlotEvictedEvent`.
- **`evicted_plot_cleanup_system`** drains queue per `caps.land.eviction_policy`. `Demolish` walks `StructureIndex` over `plot_rect`, refunds via `Deployable::compute_refund_drop`, `despawn_recursive`s.
- **Sharecropping:** `state_sharecrops` flag adds `ListingKind::Sharecrop` for Agricultural plots. Zero upfront; tenure `Sharecropping { share_to_landlord, paid_through_tick }`. `gather.rs` harvest hook routes landlord share via `split_sharecrop_yield(qty, share_to_landlord)` as a `GroundItem` at landlord's nearest `FactionStorageTile`.

## Farming (`farm.rs`, plot-scoped, seasonal)

- **Seasonal priority (jellyfish-followup plan).** In-season field work is a food-security priority, not a low-share communal job:
  - **Season-aware `farm_pressure(faction, &Calendar)`** — `0` in Winter, full annual-deficit ratio (vs `farm::annual_grain_target` = `members×48×5/4`, **not** a token `members×4` stockpile) in Spring/Autumn, scaled by `SUMMER_FARM_PRESSURE_SCALE=0.35` in Summer. One signal feeds both `compute_priority` and `compute_workforce_budget`. Annual-planning constants (`GRAIN_PER_PERSON_PER_YEAR=48`, `GRAIN_YIELD_PER_TILE_PLANNING=4`, `SUPPLY_SAFETY_NUMER/DENOM=5/4`) live in `farm.rs`, re-exported from `organic_settlement.rs`.
  - **Seasonal hard-priority** — `compute_priority` lifts an *open* seasonal `FieldWork` posting (`assigned_farmer==None`) to `SEASONAL_FARM_PRIORITY=170` (outranks chief Haul/Build/Craft, `< PRIORITY_PLAYER`). Suppressed when `food_pressure ≥ CRITICAL_FOOD_TRIGGER (80)` so emergency `Stockpile` wins. Summer caretaker Prepare (`assigned_farmer==Some`) and Plow are not boosted.
  - **Phase-weighted claim floor (balanced-farming).** `cap_bucket` keys `FieldWork` claims by `FarmWorkPhase` so Prepare and Plant get independent caps. `job_claim_system` raises an open seasonal `FieldWork` posting's per-phase cap via `farm::seasonal_field_work_floor_share`: Spring splits the legacy 0.65 envelope into `SPRING_FARM_PLANT_SHARE=0.30` + `SPRING_FARM_PREPARE_SHARE=0.35`; when only one Spring phase has open postings, the other phase's share **carries over** so a Prepare-only early Spring (or a Plant-only post-prep window) still uses the full envelope. Autumn keeps the single-phase `SEASONAL_FARM_CLAIM_SHARE_AUTUMN=0.45`; Summer/Winter unchanged. All gated on `food_pressure < CRITICAL_FOOD_TRIGGER`. Spring `Plant` postings also carry a `SPRING_PLANT_PRIORITY_BIAS=+5` over `SEASONAL_FARM_PRIORITY` (set in `compute_priority`) as the tie-break for equidistant Prepare+Plant postings, still `< PRIORITY_PLAYER`. `posting_target_workers` still spreads workers across multiple plots within a phase.
  - **Budget farm-backlog gate** — `workforce_budget_system` passes `farm_backlog` (any non-complete `FieldWork`/`Plow` posting). `compute_workforce_budget`: `farm_backlog==false` ⇒ `raw_farm=0` AND the Farm slot is treated policy-dormant (collapses past `SHARE_FLOOR`, reroutes to `free`) — deflates the slot in Winter *and* the mid-Spring post-planting window, not when grain is finally harvested.
- **Phase-correct dispatch + progress.** Each farm dispatcher honours the claimed posting's `FarmWorkPhase` via `htn::claimed_fieldwork_phase(claim, board)`: Prepare dispatcher acts only on `Prepare`, plant only on `Plant`, harvest only on `Harvest` (`None` = no posting ⇒ autonomous Private/Bootstrap). `jobs::record_fieldwork_progress(board, job_id, phase, delta)` credits a `FieldWork` posting **only when its phase matches** — `prepare_field_task_system`→`Prepare`, planting completion→`Plant`, `gather_system` Grain branch→`Harvest` (the Autumn harvest hook that was previously missing; `JobBoard`+`JobCompletedEvent` bundled into `GatherRoutingResources`).
- **Seasonal cycle (jellyfish plan).** Pure-function mapping `farm_season_phase(&Calendar) → FarmSeasonPhase::{SpringPrepPlant, SummerMaintenance, AutumnHarvest, WinterDormant}`. Chief Farm posting branch (`chief_job_posting_system`) dispatches per phase: Spring posts `FieldWork{phase:Prepare, target=unprepared}` (open) + `FieldWork{phase:Plant, target=plantable.min(grain_seed_stock)}` (open); Summer posts low-priority caretaker Prepare (assigned_farmer:Some); Autumn posts one Harvest per plot with `target=mature_grain_count`; Winter posts nothing. `posting_target_workers(JobKind::Farm)` scales with target = `(target/8).clamp(3, 12)` so Spring 256-tile prep gets 12 workers, Autumn 60-plant harvest gets 7.
- **Field state (`FieldTileIndex`).** Resource keyed `(tile → FieldTileState { plot_id, nutrients: u8, last_crop: Option<PlantKind>, last_worked_year: u16 })`. World-gen `TileData.fertility` is the **immutable** per-tile recovery ceiling; `nutrients` is the live pool, mutated by harvest / prepare / fallow only. Seeded by `carve_plots_system` for every Agricultural-plot tile (`nutrients = TileData.fertility`) — at both `OnEnter` and runtime; `seed_starting_farms_system` no longer touches `FieldTileIndex` (the carve owns it). `backfill_field_tile_index_system` (OnEnter(Playing)) is a safety net covering save-game tiles.
- **Prep gate.** `Task::PrepareField { tile }` (`TaskKind::PrepareField=52`) executor `farm::prepare_field_task_system` (Sequential, after movement, before gather): accumulates `FIELD_PREP_WORK_TICKS=80`, stamps `TileKind::Cropland` (preserves fertility), emits `TileChangedEvent`, credits the Farm posting through `jobs::record_progress` so target completion removes the posting and releases `JobClaim`, grants `SKILL_XP_PER_PREP_TILE=4` Farming XP, bumps `nutrients` to `max(current, EXHAUSTED_FLOOR=30)`. Dispatcher `htn_prepare_field_dispatch_system` (ParallelB) routes a Farm-claimed Prepare-phase worker to the nearest `find_nearest_unprepared_in_rect(...)` tile via `dispatch_autonomous_task_with_routing`, which stamps the Farm/PrepareField lifecycle.
- **Plant gate.** `htn_plant_from_storage_dispatch_system` calls `find_nearest_plantable_in_rect(...)` which requires `tile.kind == Cropland` AND `FieldTileIndex[tile].nutrients >= MIN_PLANTABLE_NUTRIENTS=80`. Bootstrap (no plot rect) keeps the legacy `find_nearest_unplanted_farmland` radius search.
- **Harvest yield + nutrient debit.** `gather.rs` Grain branch reads `FieldTileIndex[tile].nutrients` and scales qty via `grain_yield_for_nutrients`: `≥180→5, ≥120→4, ≥80→3, else 1`. Then debits `nutrients -= HARVEST_NUTRIENT_DEBIT=30 (saturating)`, sets `last_crop = Some(Grain)`, `last_worked_year = calendar.year`. `FieldTileIndex` + `Calendar` are bundled into `GatherRoutingResources` so `gather_system` stays under the 16-param ceiling.
- **Fallow recovery (`fallow_recovery_system`, Economy, season-edge).** `Local<Option<Season>>` gate fires once per season change; walks `FieldTileIndex.by_tile`, for tiles with no live plant AND `last_worked_year < calendar.year` bumps `nutrients += FALLOW_NUTRIENTS_PER_SEASON=15`, capped at the per-tile fertility ceiling. An exhausted (< 30) tile must rest ~4 seasons (4×15 = 60) before it's plantable again — the intended fallow-rotation pressure.
- **Demand-driven plot sizing (`organic_settlement::parcel_targets`).** `target_plots = (target_active / 96).max(1).min(12)` where `target_active = min(food_tiles, labor_tiles, seed_tiles)`: `food_tiles = ceil(members * GRAIN_PER_PERSON_PER_YEAR * SUPPLY_SAFETY / GRAIN_YIELD_PER_TILE_PLANNING)` ≈ `members*15`, `labor_tiles = (members*60/100)*24`, `seed_tiles = grain_seed_stock.max(current_ag_tiles).max(32)`. The demand constants are `pub` in `organic_settlement.rs`: `GRAIN_PER_PERSON_PER_YEAR=48` (derived from `HUNGER_RATE`, the 180-s game-day, grain's 150 cal, and the 20-day year — re-derive if those knobs move), `GRAIN_YIELD_PER_TILE_PLANNING=4`, `SUPPLY_SAFETY=5/4` (bad-year/reserve margin). A 20-member tribe targets ~300 tiles ⇒ ~2 plots — earlier this used a magic `16` that under-provisioned 3×. `current_ag_tiles` (sum of every Agricultural parcel rect on `brain`, via `current_ag_tile_count`) keeps an established village from shrink-planning below its productive footprint on a transient low-seed reading.
- **FarmWorkScorer season gate (`goal_scorers.rs`).** Adds `ctx.farm_season` and `ctx.private_plot_has_seasonal_work`. Returns `None` in Winter or when the household plot has no Prepare/Plant (Spring), Prepare (Summer), or Harvest (Autumn) work in season. `goal_update_system` builds a `households_with_seasonal_work` snapshot per cycle that mirrors `households_with_ag_plot`.
- **Plot size:** Agricultural plots are 16×16 tiles (~576 m² at 1.5 m/tile) carved 1:1 from organic Agricultural parcels — one plot per parcel, no subdivision mismatch. Yields are tuned so a 4-person household subsists on one plot.
- **`FarmPlotAssignments`** (Resource): one-to-one match `farmer Entity ↔ PlotId`. Maintained daily by `chief_farm_plot_assignment_system` (Economy) for villages whose grain policy still has `chief_allocates_labor=true`. Greedy match by chebyshev distance; releases stale entries when farmers despawn / demote, plots leave state ownership, or plots vanish.
- **Seasonal chief Farm postings:** `chief_job_posting_system` Farm branch dispatches on `farm_season_phase(&calendar)` and emits `JobProgress::FieldWork { phase, completed, target, area, plot_id, assigned_farmer }` per plot per phase (Spring: open Prepare + Plant; Summer: caretaker Prepare; Autumn: open Harvest; Winter: nothing). `resolve_farm_scope` reads the same variant. See top-level Farming section for the full phase table. Shared helper `farm::state_owned_ag_plots_for_faction(...)` enumerates the plot set; reused by `faction_profession_system` (§7) so promotion + posting speak the same definition. **No Agricultural plot ⇒ NO Farm posting** (the old `plot_id=None` / `home_tile±5` bootstrap is deleted).
- **Claim restriction:** `job_claim_system` rejects a posting whose `assigned_farmer` doesn't match the candidate worker — only the assigned farmer can claim their plot job.
- **Plant ownership:** `production.rs` cascade is `JobKind::Farm + FactionMember → Faction-owned`, then `HouseholdMember → Household`, else `Person`. Fixes the bug where household members on chief-Farm postings stamped Household ownership.
- **`FarmScope` resolver** (`htn.rs::resolve_farm_scope`, shared by both farm dispatchers): three branches mirror the `production.rs` plant-ownership cascade — **Communal** (`JobClaim::Farm` posting with `plot_id` + matching `assigned_farmer` → village storage, plot rect from `PlotIndex`), **Private** (§4: ANY `HouseholdMember` whose household holds a `ZoneKind::Agricultural` plot → household storage as `source_faction_id`, `fallback_source_faction_id = village` for §5 seed-fallback, deposit override `Some(household_id)`), **Bootstrap** (everything else → village storage, no plot rect). The Farmer-only gate on Private is gone: a Mason in a kitchen-garden household can tend their plot.
- **Plot-restricted planting search:** `htn_plant_from_storage_dispatch_system` walks `FarmScope::seed_source_candidates()` (Private: household first, parent village as fallback; Communal/Bootstrap: village only) and picks the first source with both seed stock and a storage tile holding the seed. Uses `scope.plot_rect()` to drive `find_nearest_unplanted_in_rect(..)`. Bootstrap falls back to `find_nearest_unplanted_farmland(.., radius=15)`. §5 fallback avoids first-season deadlock for a freshly-housed kitchen-garden household that hasn't harvested yet — the parent village's seeded grain stock backstops planting.
- **Plot-restricted harvest search:** `htn_harvest_plant_dispatch_system` reuses the same resolver. Communal/Private walk the plot's rect via `PlantMap`, picking the nearest live `GrowthStage::Mature` plant (Chebyshev) — no more wandering off to forage random berries. Private threads `deposit_target_faction_override = Some(household_id)` so `HarvestMaturePlantForStorageMethod::expand` emits `Task::DepositToFactionStorage { target_faction_id: Some(household_id) }`; `gather_deposit_tile` routes through household storage. Bootstrap keeps the legacy `GatherKnowledge::nearest_target_tile(MemoryKind::AnyEdible)` search.
- **`FarmWorkScorer`** (§4, Subsistence-tier, score 0.90 + 0.05 Farmer lift): household-availability gate replaces the Farmer-only gate — `ctx.private_farm_available` is `true` when the agent is a `HouseholdMember` of a household that holds a `ZoneKind::Agricultural` plot. Any household adult (Farmer / Mason / Apprentice) nominates `AgentGoal::Farm` so a kitchen-garden household isn't blocked when its head isn't a Farmer. Farmers still win EV/skill via the +0.05 lift + downstream `expected_wage` competence. Grain policy `private_actors_allowed=true` still gates (Mixed / Market). Registered before `StockpileScorer`. `goal_update_system` builds the `households_with_ag_plot` snapshot once per cycle so the per-agent check is O(1).
- **`Task::DepositToFactionStorage` carries `target_faction_id: Option<u32>`:** `Some(household_id)` overrides actor-faction routing so private harvest deposits to household storage. Routing honored in `gather.rs::finish_gather` and `items.rs::finish_scavenge`. Default `None` preserves village-storage behaviour.
- **Mixed preset enables capitalist grain:** `apply_preset` no longer skips grain/wood/stone/edibles for Mixed; every catalog resource gets `mixed()` (chief still allocates AND private actors allowed). Chief Farm jobs and `FarmWorkScorer` co-exist in Mixed villages.
- **Game-start seeding (`seed_starting_farms_system`):** OnEnter(Playing), after the in-chain `survey → project → carve` segment (`plans/spawn-farm-seeding.md`). The seeder **does NOT create Plot entities** — `carve_plots_system` owns that in the same OnEnter chain. Instead it finds the **carved Agricultural plot nearest home** (walks `PlotIndex.by_settlement[sid]` → filters `ZoneKind::Agricultural && faction_id == fid`, deterministic tie-break `(chebyshev, rect.x0, rect.y0)`) and stamps the starter patch into it. If no Agricultural plot exists for a faction (no belt parcel from the post-seed survey), the seeder stamps no Cropland but still seeds the grain + food buffer (planting fires once a runtime carve lands). Per founding faction (member count read from `FactionData.member_count`):
  - **Seed grain** = `demand_tiles.max(32)` (`demand_tiles` = same expression as `parcel_targets` `food_tiles`) — scales with population so `seed_tiles` no longer pins planting to a flat 32.
  - **Year-1 food buffer** — edible `grain` `GroundItem`s on faction storage, `members * GRAIN_PER_PERSON_PER_YEAR * 3/5` (~0.6 yr/founder), bridging spawn → first harvest + the partial-first-crop gap. Founders carry provisions; it runs out and foraging covers the rest.
  - **Starter Cropland patch** — a bounded contiguous fraction of the carved plot (`min(demand_tiles/2, plot_area/2)` tiles, row-major) is pre-tilled to `Cropland` (`set_tile` + `TileChangedEvent`) so year 1 skips the from-zero prepare spike. **Fertility is NOT overridden** — `FieldTileIndex` nutrients (already seeded by `carve_plots_system` from natural world-gen `TileData.fertility`) are not re-touched; the patch's planting viability rides on the belt's (fertility-selected) soil, with the year-1 grain buffer covering a weak year. The rest of the plot stays un-prepared for the tribe to till over following seasons.
  `PlotIndex.ag_tiles` + `FieldTileIndex` are populated by `carve_plots_system` — the seeder neither re-inserts nor reads them. Sandbox + nomadic factions skipped.
- **Belt placement near-edge bias (§1, `build_ag_belt`):** in addition to fertility + soft road bonus, each candidate pays a chebyshev distance penalty `0.5 * ((cheb(c, home) - fp_extent).max(0) / (scan - fp_extent).max(bw))` so a "good and adjacent" tile beats a "very slightly better and far" tile, but a 2× fertility delta still wins. Penalty cap (~0.5) is comparable to the road bonus + typical agricultural-suitability deltas. Connected-blob invariant from the contiguity rank is unchanged.
- **Kitchen-garden parcels (§2, `append_dwelling_gardens`):** after `build_ag_belt`, keyed on **actual built dwellings** — `detect_dwellings` reconstructs Hut (3×3) and Longhouse (5×3 / 3×5) footprints from the survey wall snapshot (an axis-aligned wall ring with one door gap + clear interior; longhouse shapes tried first so a longhouse corner isn't mis-claimed). For every dwelling with no belt parcel within `KITCHEN_PROXIMITY = 12` chebyshev, emits a house-sized Agricultural parcel flush against the dwelling: a **3×3** Hut garden on any clear edge; a **3×4** Longhouse garden on a **3-tile short wall only** (never the 5-tile long wall) — skipped if neither short side is clear/reachable. The away-from-home side is preferred. Overlap keep-out excludes Residential parcels + the Residential district disc (a garden belongs in the residential area and may abut its dwelling); `rect_clear_for_parcel` still rejects walls/roads so it can't overlap a neighbour house. The `land_listing_system` 12-tile child-claim path (`CHILD_FARM_MAX_DIST`) binds it as the household's child plot; carving flows through `carve_plots_system` — no new Plot construction path. Routes through `FarmScope::Private` like belt-acquired child plots. (Kitchen gardens only emit once dwellings exist — the OnEnter kickoff survey runs before seeding, so gardens appear on the first post-seed survey.)
- **Storage spawn on child-Ag claim (§3, `land_listing_system::ensure_household_storage`):** the Residential→child-Ag claim path used to mark the household as Ag-holding *without* spawning a `FactionStorageTile` (the storage block was gated on the outer-loop's `target_zone == Agricultural`, which was now skipped). Hoisted into a single helper called from both the direct-Agricultural acquisition path and the child-claim arm. A household that gets its first Ag plot via the child path now gets a storage tile, unblocking `FarmScope::Private` (no storage = no deposit target = no planting).
- **Plot-demand Farmer promotion (§7, `faction_profession_system`):** target_farmers is `max(food_target, plot_target)` where `plot_target = min(state_owned_ag_plots_count, members/3)`. A village at moderate `per_head` (above the demote threshold) with three unworked plots still recruits ≥ 3 Farmers — previously it promoted zero because the food-only ramp didn't fire. Plot-demand also forms an emergency floor (`food_target_when_low.max(plot_target)` below promote threshold) so even an undernourished village covers its plots. Uses `farm::state_owned_ag_plots_for_faction` (shared with `chief_job_posting_system` §6).

## Game lifecycle and regions

- **`GameState`:** `SpawnSelect` (default) | `Playing`. Spawn UI in `SpawnSelect`; spawn systems `OnEnter(Playing)`. Update-stage systems gated `in_state(Playing)`; FixedUpdate sim not gated.
- **Sandbox bypass:** `SandboxPlugin` flips state to `Playing` at Startup.
- **`SettledRegions` (`region.rs`):** `AHashMap<RegionId, SettledRegion>` + `by_megachunk`. `SettledRegion { megachunk, founding_tick, name, camera_bookmark, player_owned }`. `settle()` idempotent.
- **`MegaChunkCoord`:** `MEGACHUNK_SIZE_CHUNKS=16`. Independent of climate cells. Helpers: `from_tile`/`center_tile`/`chunk_range`/`tile_bounds`/`contains_tile`.
- **Player home is mega-chunk-constrained.** `spawn_population` (`person.rs`) special-cases `group_idx == 0`: it calls `region::pick_player_home_in_megachunk(chunk_map, mx, my, world_seed)` so the home tile is *always* inside the cell the player picked on the spawn-select map. The picker is deterministic (`fastrand::Rng` seeded via `home_pick_seed`, a splitmix64 mix of `(world_seed, mx, my)`) with a centre-out spiral fallback and a forced-centre last resort. **`home_pick_seed` must never use `ahash::AHasher::default()` / any entropy-keyed hasher** — ahash's default key is per-process, so identical inputs would hash differently each run and de-determinise every settlement layout (caught by `home_pick_seed_is_process_stable`). AI factions (`group_idx > 0`) keep the 32×32-chunk best-of-200 search so neighbours fan out around the player.
- **Near factions vs. world-map factions** (`plans/world-map-abstract-factions.md`). `spawn_population` spawns only the player + `NEARBY_RIVAL_COUNT = 3` rivals as full entity groups (`near_factions = NEARBY_RIVAL_COUNT + 1`, capped by `INITIAL_POPULATION / GROUP_SIZE`). AI faction homes are scored by `faction_spacing_score` — a continuous farthest-point reward: min straight-line distance to every already-placed home (incl. the player), saturating at `NEAR_FACTION_TARGET_SPACING = 280` tiles, `+0..+100` — blended with the river-proximity curve (`−80..+60`). Replaces the former binary within-300-tiles `−100/+50` penalty, which silently degraded once placed exclusion circles tiled the 1024×1024 spawn window and let later factions cluster. Soft gradient, not a hard minimum. Pure-function tests in `person::tests`.
- **Abstract world-map factions (`abstract_faction.rs`, Phases 2-3 — SHIPPED).** The remaining faction slots (`INITIAL_POPULATION / GROUP_SIZE − near_factions`, today 6) are seeded by `seed_abstract_factions_system` (`OnEnter(Playing)`, after `spawn_population`) as **abstract factions**: a `FactionData` (`materialized = false`, Settled archetype + game economy preset, no chief/members) plus a `faction_id` stamped on one habitable Globe `WorldCell` with `population`/`food_stock`. Homes are farthest-point-spread (`cell_spacing_score`) across the Globe outside the player's pre-gen window keep-out. `world_sim_system` ticks their population/food/raids abstractly (it already simulates claimed unloaded cells). `AbstractFactions` resource (`by_id` + `by_megachunk`).
  - **`FactionData.materialized`** (`faction.rs`, default `true` in `create_faction`) gates entity-driven systems off abstract factions: `auto_found_default_settlements_system` / `auto_found_default_camps_system` skip them (no Settlement/Camp entities), and `faction_decision_system` + `pick_raid_target` (`raid.rs`) skip them (the entity raid FSM never runs for or targets an abstract faction — `world_sim` owns their raids). Other faction-iterating systems are member-query- or chief-driven and naturally no-op on a memberless, chiefless faction.
  - **Materialization (Phase 3).** `materialize_abstract_faction_system` (FixedUpdate, after `chunk_streaming_system`): when a `ChunkLoadedEvent` arrives for the chunk containing an abstract faction's home tile (the player has travelled near), the faction spawns as full entities. `person::spawn_faction_band` — the band-spawn body extracted from `spawn_population` (home `FactionStorageTile` + members from a `spawn_tiles_from` reachable flood + chief; advances `clock.population`/`bucket_size` per member exactly as `reproduction.rs`) — is shared by both call sites so a materialised faction is identical to one spawned beside the player. The system re-anchors `FactionData.home_tile` onto real ground (`nearest_passable_tile`, the cell-centre home may be water/stone), spawns up to `MAX_MATERIALIZED_MEMBERS = 40`, flips `materialized = true`, clears the Globe cell's faction fields (handing the faction to the entity sim — `auto_found_default_settlements_system` founds its settlement next Economy tick, organic AI builds it), and drops it from `AbstractFactions`. **Phase 4 (dematerialise back to abstract data on player departure) is not built** — a materialised faction currently stays materialised (entities persist, Dormant when far; no double-sim since the cell was cleared).
- **Multi-focus chunk streaming (`SimulationFocus`):** `Vec<FocusPoint>` rebuilt each tick from camera + every settled region centre. Chunk DATA loads for any focus disc (camera `LOAD_RADIUS=12`, region `REGION_LOAD_RADIUS=6`); SPRITES + plants + loose rocks only inside camera focus.
- **Focus-aware LOD:** camera distance produces base LOD; entities within 8 chunks of any non-camera focus promote from Dormant to Aggregate. `cohort.rs` (`CohortRegistry`) summarises Aggregate agents by `{faction, settlement_or_camp, profession, age_band, wealth_band, lifestyle}`; `cohort_pin_full_sim_system` (ParallelA after camera LOD) promotes commanded/chief/drafted/combat agents back to `Full` via `PinnedFullSim`.
- **Edge-walk expansion (`region::detect_edge_crossing_system`):** each tick, unsettled mega-chunks the player faction enters get `settle()`, naming "Outpost N", bookmarking, emitting `ActivityLogEvent::RegionSettled`.
