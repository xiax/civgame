# Simulation (`src/simulation/`)

Agent AI, factions, knowledge, hunting, typed-task pipeline, pluralist economy. See root `CLAUDE.md` for cross-cutting `SimulationSet` ordering.

## Agent AI (Goals → HTN → Tasks)

AI dispatch runs end-to-end through HTN.

- **Goals (`goals.rs`):** high-level objectives (`Survive`, `GatherWood`, `GatherStone`, `GatherFood`, `Haul`, `Defend`, `Farm`, `Build`, `Socialize`, `Sleep`, `MigrateToCamp`, …) driven by Needs + Faction state.
- **HTN registry (`htn.rs`):** per-goal decomposition. Methods carry `precondition` / `utility` / `expand(ctx) -> Vec<Task>`; the dispatcher argmaxes over applicable methods and routes the head into the typed-task channel, with the tail prefetched on `ActionQueue`'s queue ring.
- **Tasks (`tasks.rs`):** the agent's *current* action — `TaskKind` enum (Gather, Construct, WithdrawMaterial, Hunt, Butcher, Equip, Migrate, …). Transient; `Idle` between tasks.
- **GoalScorer + Disposition infrastructure (`goal_scorers.rs`, Phase 6 of wage-aware-labor-market-v2):** the plan originally deferred this plumbing to a peer plan that didn't ship; now it lives here. `Disposition { entrepreneurial, gregariousness, curiosity, martial }` Component (`u8` on `[0, 255]`, scattered at spawn) drives per-agent goal preferences. `GoalClass` enum (`Survival > Subsistence > Safety > Belonging > Esteem > Enterprise > Discretionary`, `Ord`-derived) tiers qualitatively-different goals so registry argmax picks `Survival → Sleep` over `Enterprise → Craft` even when the Craft score is massive. `GoalScorer` trait (`score(&self, ctx: &GoalScoringContext) -> Option<GoalScore>` + `name()`) is implemented by individual scorers; `GoalScorerRegistry` (Resource) holds the `Vec<Box<dyn GoalScorer>>` and the `best(ctx)` argmax helper. `GoalScoringContext` bundles every per-agent read so scorers stay pure-function. `register_default_scorers` pushes the default set at `SimulationPlugin::build`.
- **Behavioural-richness scorer set (`goal_scorers.rs` + `utility_curves.rs`, plan: `~/.claude/plans/evaluate-this-plan-please-tingly-catmull.md`).** Phase A landed pure-function continuous utility curves — `hunger_utility / sleep_utility / social_utility / play_utility / material_deficit_utility / disposition_lift` — anchored on existing `HUNGER_*` / `SLEEP_*` thresholds so inflection points match legacy cliffs but slope is continuous. Phase B added nine scorers covering the legacy `goal_update_system` need-cascade: `SurvivalHungerScorer` (Survival, hunger_utility + four reason strings matching legacy — emits `Survive` *only* when one of those four legacy cliffs matches; below `hunger 150` with no food the scorer returns `None` rather than pinning the agent on a goal that can't decompose into a task, since every HTN food/eat method gates on `EAT_TRIGGER_HUNGER = 180`), `SleepScorer` (Survival, sleep_utility × `time_of_day_bonus`), `ReturnSurplusScorer` (Subsistence, held-food × can_return_camp × gregariousness lift), `SocialScorer` (Belonging, social_utility(needs.social, gregariousness)), `PlayScorer` (Discretionary, play_utility × gregariousness lift), `TameHorseScorer` (Subsistence, curiosity lift), `PersonalBuildScorer` (Esteem, gated on `has_personal_build_site`), `CraftDemandScorer` (Subsistence, entrepreneurial lift + Crafter/Apprentice profession bonus), `StockpileScorer` (bifurcated class — Subsistence at score 0.85 when `prioritize_food`, Discretionary at 0.20 otherwise so idle agents fall through to gather without preempting Socialize/Play). Per-scorer Disposition lifts (`SOCIAL_GREG_LIFT = 1.5`, `PLAY_GREG_LIFT = 1.2`, etc.) live in one block at the top of `goal_scorers.rs` for tunability.
- **`AgentDecisionMode` (`utility_curves.rs`):** runtime `Resource` switch between `Legacy` (imperative cascade in `goal_update_system`) and `Scored` (scorer pipeline). **`Scored` is the default since Phase F-1**; `Legacy` is kept as an A/B-comparison fallback and as an escape hatch if a Scored regression is discovered. `goal_update_system` extracts the cascade body into a `legacy_pick` closure; the new selection point branches on the resource. `Scored` mode builds a `GoalScoringContext` with the same precomputed gates the cascade already derives (`is_starving / faction_has_food / can_return_camp / prioritize_food / fallback_gather / fallback_gather_reason / has_horse_taming / has_personal_build_site / should_craft`) plus a `time_of_day_bonus` derived from `Calendar::time_phase()` (`Day 0.0 / Dawn 0.2 / Dusk 0.6 / Night 1.0`), then calls `GoalScorerRegistry::best(...)`. Falls back to `legacy_pick()` on `FactionRegistry` lookup miss. **Test fixture (`TestSim::new`) overrides back to `Legacy`** so existing tests pinned to legacy semantics keep their contract; dedicated Scored-mode tests can flip per-test via `sim.app.world_mut().insert_resource(AgentDecisionMode::Scored)`.
- **Goal-selection hysteresis (`goal_update_system` Scored branch):** `GOAL_CHALLENGER_MARGIN = 0.10` dampens single-tick flips around utility crossover. When the registry's best differs from the agent's current goal, re-score the *current* goal across all scorers at ≥ best's class; if `best.score - cur_score < 0.10` keep the current goal. Survival/Subsistence still preempt lower classes via class ordering inside `GoalScorerRegistry::best`. Anti-flicker complements the existing `GoalCooldown` ring (post-failure cooldown) and method-level `MF_UNINTERRUPTIBLE` (mid-method preemption guard) — three mechanisms together cover goal-level commitment without a new component.
- **`ScorerInputs` SystemParam (`goals.rs`):** bundles `AgentDecisionMode`, `GoalScorerRegistry`, `JobBoard`, and per-agent `Disposition / Skills / Profession` queries into one parameter so `goal_update_system` stays under Bevy's 16-param ceiling. Companion components live alongside `PersonAI` on every Person; missing-component fallbacks (`Disposition::default()`, `Skills::default()`, `Profession::None`) keep the system robust for synthetic / test fixtures.
- **Opportunistic en-route interrupts (`opportunistic.rs`, Phase D).** New `GoalScorer::opportunistic() -> bool` trait method (default `false`); `SocialScorer` and `PlayScorer` opt in. `opportunistic_interrupt_system` runs in `SimulationSet::ParallelA` ordered `.after(goal_update_system).before(record_abandoned_method_system)` — gated on `AgentDecisionMode::Scored` and a 20-tick (1-Hz) cadence so re-evaluation is cheap. Per agent eligibility: not Dormant, not `JobClaim`-locked, `aq.current == Task::WalkTo` (mid-walk between sub-tasks), `ai.active_method` not flagged `MF_UNINTERRUPTIBLE`, current goal class not `Survival | Safety` (explicit membership check — the enum's numeric ordering puts `Subsistence (5)` between `Survival (6)` and `Safety (4)` because Subsistence is semantically more important than Safety, so a naive `>=` comparison rejects ReturnCamp/Gather agents). Builds `GoalScoringContext` (precomputed-gate fields stubbed since only opportunistic scorers consult it), walks scorers with `opportunistic() == true`, picks the highest `(class, score)` whose `score >= OPPORTUNISTIC_INTERRUPT_THRESHOLD (0.50)` and isn't the agent's current goal or on the agent's `GoalCooldown` ring. On interrupt: stamps the *prior* goal into the ring for `OPPORTUNISTIC_COOLDOWN_TICKS (150)` to prevent ping-pong, flips `*goal`, `aq.cancel()`, resets `ai.task_id = UNEMPLOYED`, updates `GoalReason`, increments `OpportunisticInterruptStats { total_fired, last_tick }`. `class_for_goal(AgentGoal) -> GoalClass` is the canonical mapping (kept in sync with individual scorer class assignments — covered by `class_for_goal_covers_every_variant` test). `Legacy` mode skips the whole pass.
- **`AI: Legacy/Scored` debug-panel toggle (`ui/debug_panel.rs`).** Top of the Debug window. Click flips `AgentDecisionMode`; tag below the button describes the active mode (`imperative cascade` vs. `Disposition-driven scorers`). Lets operators A/B compare in-game without recompiling. Phase D's opportunistic interrupts come on automatically with `Scored`.
- **HTN Disposition lift (`htn.rs`, Phase E-1).** New trait method `Method::disposition_lift(&self, Disposition) -> f32` defaulting to `1.0`. New scoring fn `score_method_with_history_and_disposition(method, abstract_task, ctx, disposition, history, now)` multiplies `utility() × disposition_lift()` before applying the failure penalty. Legacy `score_method_with_history` is unchanged so the ~21 unmigrated dispatchers keep working. Three methods override the lift: `SocializeWithPartnerMethod` (gregariousness lift, capped 1.3 — stays sub-`UTIL_VISIBLE_GROUND`), `HuntPreyMethod` (martial lift, capped 1.3 — preserves ranking vs `PickUpFreshCorpseMethod`), `PlayWithPartnerMethod` (gregariousness lift, capped 1.3 — stays sub-`UTIL_CLAIMED_HAUL`). Three dispatchers migrated to the new scoring fn by adding `Option<&Disposition>` to their query: `htn_socialize_dispatch_system`, `htn_engage_prey_dispatch_system`, `htn_play_dispatch_system`. The other ~21 dispatchers stay on the legacy fn — full migration deferred to `civgame/plans/htn-dispatcher-consolidation.md`. Lift magnitude (`disposition_lift(axis, 0.3)` = range `[1.0, 1.3]`) chosen to alter scoring values without crossing the `UTIL_BASELINE` / `UTIL_VISIBLE_GROUND` / `UTIL_CLAIMED_HAUL` tier breakpoints that `haul_full_trip_cap_preserves_ranking_over_bare_withdraw`-style tests pin.
- **`EarnIncomeScorer` (`goal_scorers::EarnIncomeScorer`):** the plan's headline scorer, lifted from the degraded procedural form into a proper trait impl. Score = `posting.reward × skill_competence(primary_skill_for(prof)) × disposition.earn_income_multiplier()` over the highest-reward unclaimed paid posting on the agent's faction's `JobBoard` whose `JobKind` matches `job_kinds_for(prof)`. Declines (returns `None`) for `None / Apprentice` agents, pure-Subsistence factions (both `economic_policy` and `wage_signal` empty), or no matching unclaimed paid posting. Tier = `Enterprise`. Travel cost stays delegated to `job_claim_system`'s `U_bid` at claim time. `earnincome_goal_override_system` consumes `GoalScorerRegistry::best(&ctx)` — picks any `Enterprise+` tier scorer's choice and applies it to a fallback gather goal — and runs ParallelA after `goal_update_system` and `mobile_state_goal_gate_system`. Filters out below-Enterprise (Discretionary) picks so the legacy cascade still owns Subsistence / Safety / Survival branches.
- **Professions (`person.rs::Profession`):** `None | Farmer | Hunter | Bureaucrat | Trader | Crafter | Apprentice | Healer`. Persistent role. Farmer auto-assigned by `faction_profession_system` when food < 100. Hunter is chief-driven via `faction_hunter_assignment_system` (Economy, every `TICKS_PER_DAY/4`): target headcount = `max(1, adults*0.20)` × martial × prey-density, capped at adults/2; demotes do full teardown. Bureaucrat/Trader cover settlement administration and arbitrage (see Pluralist Economy). **Crafter (Phase 5a of wage-aware-labor-market-v2)** is chief-driven via `chief_craft_assignment_system` (Economy, after `chief_bureaucrat_appointment_system`, every `CRAFTER_ASSIGNMENT_CADENCE = TICKS_PER_DAY/4`): target headcount = `max(1, members × CRAFTER_MIN_RATIO (0.25))` capped at `members / CRAFTER_MAX_DIVISOR (3)`, gated on the faction's max Craft `wage_signal` EMA via a hysteresis deadband — `ema ≥ CRAFTER_WAGE_PROMOTE_FLOOR (1.0)` → ratio-driven target; `ema ≤ CRAFTER_WAGE_DEMOTE_CEILING (0.3)` → target 0; in between → hold at current crafter count. Pure helper `crafter_target_with_hysteresis(craft_ema, current_crafters, member_count)` encapsulates the deadband for unit testing. EV-ranked promote / demote share the shared `demote_profession_state` teardown. Crafter affinity into `U_bid`: `jobs::CRAFTER_AFFINITY_BONUS = 3.0` adds to a Crafter's bid on paid Craft postings; the unpaid `profession_bias` path also gets `(Crafter, Craft) = 0.5` and `(Crafter, Stockpile) = 0.1` arms so chief Craft postings prefer Crafter claimants. `capital::tool_profession(tools) → Crafter` and `WorkshopKind::affine_to(Crafter)` (Workbench / Loom) wire Phase 2 capital recognition through to Crafter EV without further plumbing.
- **Capital recognition (`capital.rs`, Phase 2 of wage-aware-labor-market-v2):** read-only helpers that score how well-equipped an agent is to practise a given profession, used by Phase 4's EV-driven `profession_choice_system`. `OwnedBy { faction_id, kind: WorkshopKind, tile }` component stamped on every Workbench / Loom / Granary / Shrine / Market / Barracks / Monument at finalize (both blueprint and seeded paths in `construction.rs`); add/remove hooks (registered in `SimulationPlugin::build`) maintain a `WorkshopOwnership` resource (`AHashMap<u32, Vec<WorkshopEntry>>`) so despawn paths stay untouched. Three capital factors all share a `[1.0, ~2.0]` shape so they compose by averaging — `tool_capital_factor(agent, carrier, prof)` → 1.5 if any inventory/hand item maps to `prof` via `capital::tool_profession` (today: `weapon → Hunter`; the Hoe / Awl / Loom shuttle / Hammer / Bow catalog split with `prof:*` tags is deferred); `workshop_capital_factor(tile, village_fid, household_id, prof, &ownership)` → +1.0 for a household-owned profession-affine workshop within `WORKSHOP_AFFINITY_RADIUS = 12` chebyshev, else +0.5 for a village-owned one; `land_capital_factor(household, prof, plot_q, &plot_index)` → 1.5 for Farmer when the household holds any Agricultural plot under Leased / Sharecropping / Freehold tenure. `WorkshopKind::affine_to(Profession)` maps Market → Bureaucrat today; Workbench / Loom / Shrine wire up once Phase 5 lands `Profession::{Crafter, Healer}`.
- **Asymmetric demotion buffer (Phase 4b of wage-aware-labor-market-v2):** `HUNTER_DEMOTE_BUFFER = 1` and `BUREAUCRAT_DEMOTE_BUFFER = 1` add a one-slot tolerance band on the demote arm of `faction_hunter_assignment_system` and `chief_bureaucrat_appointment_system`. Promotion still fires on any shortfall (`current < want` → promote `want - current` Nones); demotion only fires when `current > want + buffer`. Single-tick flapping (e.g. `nearby_prey_count` jitter rounding the Hunter target between 1 and 2) is absorbed by the buffer, while persistent over-target sheds excess on the next cadence. The `want == 0` early-exit forces full demotion regardless of buffer so the survival override / `bureaucrat_treasury_empty_streak` quit path / `HUNTING_SPEAR` tech-loss case still cleanly stand the labor down. Crafter has analogous behavior via the EMA-band `crafter_target_with_hysteresis` helper.
- **Survival override (Phase 4b of wage-aware-labor-market-v2):** `FARMER_SURVIVAL_FLOOR = 16.0` per-head food units gates each non-Farmer assignment system. `faction_hunter_assignment_system`, `chief_bureaucrat_appointment_system`, and `chief_craft_assignment_system` each compute `per_head = faction.storage.food_total() / member_count` during target-build and force `target = 0` when `per_head < FARMER_SURVIVAL_FLOOR`, regardless of the normal gates (prey density, `state_funds_public_works`, Craft `wage_signal`). Existing incumbents demote via the shared `demote_profession_state` teardown; freed agents land in `Profession::None` for the Farmer ramp (`FARMER_PROMOTE_PER_HEAD = 32`) to poach. Two-floor band: below 16 → all specialised labor stands down; 16–32 → Farmer ramp drives recovery; above 32 × `FARMER_DEMOTE_RATIO (1.6)` → demote excess Farmers. Apprentices are unaffected (the override zeros the promotion target, not active `ApprenticeOf` links — a starving faction's apprentices remain mid-training because nuking their progress would be needlessly punishing). Tests seed food via `seed_faction_food(fid, qty)` to avoid the override firing accidentally in fixtures that don't model nutrition.
- **Unified cross-profession switcher (`profession_choice::cross_profession_switch_system`, Phase 4b of wage-aware-labor-market-v2):** daily Economy pass that lets an employed agent switch *directly* between `{Hunter, Bureaucrat, Crafter}` when their EV in a candidate role exceeds their current by `EV_SWITCH_HYSTERESIS (1.20)`. Closes the gap where the four per-profession assignment systems only transition None ↔ X — without the switcher, a Hunter who would genuinely earn more as a Crafter had to round-trip through `None` across two cadence cycles. Per agent: `EV(target) = expected_wage(faction, target, skills, capital_factor(...))` minus `switching_cost_skill_regret(faction, current, peaks) = peak[primary] / SKILL_MAX × SKILL_REGRET_FRACTION (0.20) × aggregate_wage_per_day(current)` — high-peak agents pay a real EV cost to abandon their current role's skill investment. Faction caps from `faction_cap_for(faction, target) -> Option<usize>`: Hunter ≤ `adults/2` (gated on chief-aware `HUNTING_SPEAR`), Bureaucrat ≤ `max(1, adults × BUREAUCRAT_MIN_RATIO)` (gated on `state_funds_public_works`), Crafter ≤ `adults / CRAFTER_MAX_DIVISOR (3)`. Survival override (`per_head < FARMER_SURVIVAL_FLOOR`) locks every non-Farmer cap to `None` so labor stands down for the Farmer ramp identically to the per-profession systems. Sub-`APPRENTICE_THRESHOLD` Crafter targets route through `Profession::Apprentice` via the same mentor-pool pattern as `chief_craft_assignment_system`. Ordered `.after()` every other profession-mutation system, with `profession_change_log_system` `.after()` the switcher, so a same-tick demote-then-switch settles to one log entry. `Profession` derives `Hash` to support the per-(faction, profession) headcount tables.
- **Profession-choice scaffolding (`profession_choice.rs`, Phase 4b of wage-aware-labor-market-v2):** pure helpers for the (still-deferred) unified EV-driven `profession_choice_system`. `skill_competence(s)` → `0.2 + (s/SKILL_MAX) * 0.8` (floor prevents fresh agents from EV-zero); `job_kinds_for(Profession) -> &'static [JobKind]` maps Farmer → `[Farm, Stockpile]`, Hunter → `[Stockpile]`, Bureaucrat → `[Build]`, Trader → `[Haul]`; `primary_skill_for(Profession) -> Option<SkillKind>`; `aggregate_wage_per_day(faction, prof)` sums `faction.wage_signal[(kind, _)].ema_per_day` across `job_kinds_for(prof)` (target_rid wildcard — `Stockpile{wheat}` and `Stockpile{wood}` both count toward Farmer); `expected_wage(faction, prof, skills, capital_factor) = aggregate_wage × skill_competence(primary_skill) × capital_factor`. Shared `demote_profession_state(entity, ai, aq, reservations, commands)` consolidates the three duplicated teardowns (drop active task, release storage reservation, strip `Carrying` marker). **`faction_hunter_assignment_system`, `chief_bureaucrat_appointment_system`, and `faction_profession_system` (Farmer) now rank candidates by `(expected_wage, skill)`** — `expected_wage(faction, target_prof, skills, capital_factor)` dominates the sort when the faction's `wage_signal` has accumulated samples for the profession's job kinds; the legacy raw skill sort takes over as a tiebreaker when wages are zero. The Farmer system previously promoted in arbitrary scan order; demotion now picks the lowest-EV (lowest-Farming-skill on tie) member instead of the first hit. **Capital is threaded in.** Each system computes per-agent `capital_factor(agent, carrier, agent_tile, faction_id, household, target_prof, &ownership, &plots, &plot_index)` from `capital::capital_factor` — average of tool / workshop / land factors — and passes it into `expected_wage`. Concretely: a weapon-bearing Hunter candidate lifts EV by ~×1.17 vs. an unarmed equal-skill peer; a Bureaucrat candidate near a household-owned Market workshop lifts EV by ~×1.33 (workshop 2.0 averaged with tool/land 1.0); a Farmer in a household holding an Agricultural plot under non-StateOwned tenure lifts EV by ~×1.17. The full unified `profession_choice_system` (single argmax across every profession with hysteresis + explicit survival override) is the remaining Phase 4b deliverable.
- **Healer scaffolding (Phase 5b-stretch of wage-aware-labor-market-v2):** `Profession::Healer` lands additively today. Recognized by `profession_choice`: `job_kinds_for(Healer) = [JobKind::Craft]` (the closest analogue until a dedicated `JobKind::Heal` lands), `primary_skill_for(Healer) = SkillKind::Medicine`, `faction_cap_for(Healer) = adults / CRAFTER_MAX_DIVISOR`. `WorkshopKind::Shrine.affine_to(Healer) = true` so the inspector EV table and the cross-switcher's `EV(Healer)` see a real capital factor when a household holds a Shrine. Apprenticeship plumbing extended via `ApprenticeProgress { target_profession }` — graduation reads `primary_skill_for(progress.target_profession)` to pick which `SkillKind` floor to apply, and writes `*prof = progress.target_profession`, so a Healer-target apprenticeship graduates to `Healer` (writing `SkillKind::Medicine ≥ APPRENTICE_THRESHOLD`). No `chief_heal_assignment_system` exists yet — Healers can't be auto-promoted from `None` until a paid Heal-job pipeline (`JobKind::Heal` + chief posting source + executor against `Health` deltas) lands. Until then the variant is variant-only, but every downstream consumer (capital affinity, inspector, cross-switcher caps, apprenticeship graduation) is wired so the pipeline becomes additive.
- **Apprenticeship (`apprenticeship.rs`, Phase 5b of wage-aware-labor-market-v2):** novice-Crafter onboarding. When `chief_craft_assignment_system` would promote a `Profession::None` candidate whose `Skills[Crafting] < APPRENTICE_THRESHOLD (30)`, the apply loop first scans the faction's `available_mentors` pool (Profession::Crafter + `Skills[Crafting] >= MASTER_THRESHOLD (100)` + no existing `MentorOf`); if a master is available, the candidate becomes `Profession::Apprentice` instead, gets `ApprenticeOf { mentor }` + `ApprenticeProgress { ticks: 0, target_ticks: TICKS_PER_DAY × APPRENTICESHIP_DURATION_DAYS (30) }`, and the master gets `MentorOf { apprentice }`. Without a master, the candidate falls back to direct Crafter promotion so a faction without any masters can still bootstrap. `apprentice_progress_system` (Economy, daily, after `chief_craft_assignment_system`) advances `progress.ticks` by `TICKS_PER_DAY` per fire, graduates the apprentice to `Profession::Crafter` on `ticks >= target_ticks` (also lifts `Skills[Crafting]` to `APPRENTICE_THRESHOLD` floor, removes `ApprenticeOf` / `ApprenticeProgress` / matching `MentorOf`). Stale mentor links — `mentors_q.get(link.mentor).map(|m| m.apprentice == entity)` returns false (master despawned or `MentorOf` torn off externally) — demote the orphaned apprentice to `Profession::None`, discarding progress; future passes may rebind. Apprentices count toward `current_crafters` in the Crafter target headcount (committed) but stay out of the Farmer / Hunter / Bureaucrat promotion pools (`_ => {}` arm in each system's pre-pass). Apprentice payouts in `job_payout_system` are split: a claimant with `ApprenticeOf { mentor }` receives `share × WAGE_FRACTION_APPRENTICE (0.4)`; the mentor receives `share × WAGE_FRACTION_MENTOR_FEE (0.1)` (silently skipped if mentor is beneficiary or has no `EconomicAgent`); residual `share × 0.5` stays in escrow → refunds to beneficiary on despawn. Both parties get an `Earnings` ring entry + `WagePaid` activity log at the reduced amounts. Deliberate-practice 2× XP: `apprenticeship::xp_with_apprentice_bonus(base, Option<&ApprenticeOf>)` is threaded into the three canonical Crafting-XP sites (`crafting.rs` per-tick + recipe-completion; `corpse.rs` butchering) — apprentices accrue Crafting XP at `APPRENTICE_XP_MULT (2)` rate while a mentor link is live. Healer apprenticeship remains deferred.
- **Skills (`skills.rs`):** `[u32; 8]` clamped to `SKILL_MAX=255` — Farming, Mining, Building, Trading, Combat, Crafting, Social, Medicine. Default 5; `gain_xp()` clamps at `SKILL_MAX`. Companion components `SkillPeaks`, `SkillUseTicks`, `SkillsLastSeen` (all `[u32;8]`) drive the Phase 1 decay model: `skill_peaks_tracker_system` ratchets peaks up and stamps `use_ticks[i] = now` whenever a slot increases (observer over `Changed<Skills>`); `skill_decay_system` runs daily and shrinks each skill toward a peak-derived floor — `peak ≥ SKILL_MASTERY_LINE (80)` → floor `SKILL_MASTERED_FLOOR (30)`; otherwise floor `max(SKILL_FLOOR_BASE (5), peak * 0.30)`. Decay is exponential with `SKILL_DECAY_HALF_LIFE_DAYS=90`.
- **Bucketing:** agents sliced across 20 fixed `BucketSlot`s.
- **LOD:** `Detail / Aggregate / Dormant`. Dormant skip sim. Focus-aware (see Game lifecycle).
- **Memory & gossip (`memory.rs`):** known locations + agent sightings, `u8` freshness decay. Tech awareness gossips between socializing agents via `knowledge::awareness_gossip_system`. `MemoryKind` is a 3-variant `Copy` enum: `AnyEdible`, `Resource(ResourceId)`, `Prey`.
- **Needs:** 6 needs (hunger, sleep, shelter, safety, social, reproduction), `[0,255]`, decay over time. Maslow tiers (`Physiological → Safety → Belonging → Esteem → SelfActualization`) layer on top via `MaslowTier::next_unmet(needs)`.
- **Goal-update cadence (`goal_update_system`):** active agents re-evaluate every **200 ticks (10 s)**; idle agents bypass the gate. Target validation is narrow — only despawned `target_entity` invalidates (Gather/Scavenge/Construct/HaulMaterials). Stage/kind drift on a still-alive target is checked at arrival in the executors (`gather::finish_gather`, `items::finish_scavenge`, `construction`), feeding `MethodOutcome::FailedTarget` into `MethodHistory`.

## Method-design rules

- **Method scoring is *viability* + *distance*, with time-of-day and fatigue scaling for AcquireFood / AcquireGood{wood,stone}.** Methods supply a base utility tier (`UTIL_BASELINE=1.0` / `UTIL_VISIBLE_GROUND=1.5` / `UTIL_CLAIMED_HAUL=2.0` / `UTIL_EXPLORE_FALLBACK=0.3`); the dispatcher subtracts a chebyshev-distance penalty capped at `MAX_DIST_PENALTY=0.30` so inter-tier ranking is preserved. Recent failures bias the next argmax via `MethodHistory` + `score_method_with_history` (`METHOD_FAILURE_PENALTY=0.4`, `METHOD_HISTORY_LEN=6`, `METHOD_HISTORY_TTL_TICKS=600`). A single failure (-0.4) keeps a method as a candidate; two stacked failures (-0.8) push a `UTIL_BASELINE=1.0` method below the Explore fallback (0.3) so terminal-fallback engages naturally. The 30-second TTL keeps bias alive across a typical walk-and-arrive cycle. **Context-aware penalty:** `PlannerCtx.scope` defaults to `ScoringScope::Geometric` (legacy behaviour). `htn_acquire_food_dispatch_system` and the `GatherWood`/`GatherStone` branch of `htn_acquire_good_dispatch_system` swap to `ScoringScope::ContextAware { time_phase, dusk_remaining, fatigue }` via `context_aware_scope(&calendar, needs)`. Inside `dist_penalty` / `full_trip_penalty`, the geometric value is multiplied by a time-of-day factor (Day 1.0 / Dawn 1.10 / Dusk 1.0→2.0 ramp / Night 4.0) and a fatigue factor (`1.0 + fatigue` where `fatigue = 0.6·sleep_norm + 0.4·(1−willpower_norm)`); the cap rises to `MAX_DIST_PENALTY_NIGHT=1.50` only at Night so a 16-tile gather method drops below the 0.3 explore floor and Maslow-driven Sleep wins the next goal flip naturally. Day + zero fatigue matches the geometric baseline by construction, preserving daytime ranking. Phase cuts (`PHASE_DAWN_START=0.00`, `PHASE_DAY_START=0.05`, `PHASE_DUSK_START=0.65`, `PHASE_NIGHT_START=0.85`) live in `world/seasons.rs`.
- **In-hand fast-path for haul dispatchers.** Both `htn_acquire_good_dispatch_system::Haul branch` and `htn_build_claimed_blueprint_dispatch_system` Path B check the agent's `Carrier + EconomicAgent` inventory before issuing `WithdrawMaterial`; if the agent already carries the targeted resource and the bp's slot still needs it, they dispatch `Task::HaulToBlueprint` directly. Strictly scoped to dispatchers running on a held `JobClaim::Haul` (or personal-bp ownership). **Never count agent inventories in posting creation or chief blueprint-candidate scoring** — those layers must read deposited storage only.
- **Farming is `Farm`-goal only.** Both halves are HTN-driven: `PlantFromStorage` plants; `HarvestPlant` harvests. Seed↔plant mapping centralised in `PlantKind::seed_resource()` + `PlantKind::ALL` (`plants.rs`). Adding a seed/plant pair = one `PlantKind::ALL` entry + arm in `seed_resource()`.
- **Faction storage refresh:** `FactionStorage.totals` rebuilt every Economy tick by `compute_faction_storage_system`. HTN dispatchers read these directly when scoring.
- **`MF_UNINTERRUPTIBLE`:** methods set this when their multi-leg chain must survive across goal-dispatch ticks.

## Player command authority (`player_command.rs`)

UI inserts player input as `PlayerCommandEvent` (event bus). The sim drains in `SimulationSet::Input`, attaches a `Commanded { command, status, issued_tick, command_id }` component per actor, then `dispatch_player_command_system` (ParallelB) routes the command into the typed-task pipeline. **All routing happens sim-side** — UI never mutates `PersonAI` / `ActionQueue` / authority markers directly.

- **One event, one component, eighteen variants.** `PlayerCommand`: `Move`, `Gather`, `Mine`, `Build`, `Deconstruct`, `DigDown`, `PickUpItem`, `PickUpCorpse`, `AttackEntity`, `Teach`, `HoldLecture`, `ReadItem`, `EncodeTablet`, `Muster`, `Disband`, `MilitaryMove`, `MilitaryAttack`, `PackCamp`, `PitchCamp { tile, z }`. Pack/Pitch are faction-scoped (chief is the canonical actor) — dispatcher writes a pending op into `nomad::PendingCampOps`, drained by `apply_pack_camp_command_system` / `apply_pitch_camp_command_system` (Sequential, exclusive `&mut World`).
- **Goal forcing replaces filter scatter.** `goal_update_system` reads `Commanded` (via `GoalValidationQueries.commanded_q`); when non-terminal, forces `AgentGoal::FollowingPlayerCommand` and `continue`s. HTN dispatchers gate on goal variants they recognize; `FollowingPlayerCommand` matches none, so they naturally skip. **No `Without<PlayerOrder>` filters anywhere.** Previously 28 of these were scattered across `htn.rs`, `goals.rs`, `tasks.rs`, `terraform.rs`, `trader.rs`. The legacy `PlayerOrder` component and `PlayerOrderKind` enum are gone.
- **Lifecycle is explicit, per-kind.** `player_command_lifecycle_system` (Sequential, after executors) match-arms on `PlayerCommand` to detect terminal state: Move/MilitaryMove = chebyshev arrival; Gather = target plant despawned OR agent idle; Mine = tile no longer Wall/Stone; Build = blueprint despawned; PickUp* = target gone; Attack/MilitaryAttack = foe dead; Muster/Disband = immediate (stamp-and-done). Knowledge orders complete on idle.
- **Reap one tick after terminal.** `reap_terminal_commands_system` strips `Commanded` once status is `Completed | Failed(_) | Superseded`. UI consumers read `RemovedComponents<Commanded>` for HUD feedback.
- **Faction-level commands** (currently `EncodeTablet`) carry an empty `actors` list and apply directly in `drain_player_command_events_system` (writes the `PlayerCraftRequest` resource for `chief_tablet_posting_system` to consume).
- **Schedule placement:** `Input → ParallelA → ParallelB → Sequential → Economy`. The `Input` set is exclusive to player-command drain; commands flush before goal/dispatch reads. The Move race we hit (UI in Update racing FixedUpdate dispatch) is structurally impossible — events deliver before the next FixedUpdate tick.
- **Adding a new order type** = four touches: new `PlayerCommand` variant (`player_command.rs`), `dispatch_one` match arm (same file), `player_command_lifecycle_system` match arm (same file), UI button emits the event. No HTN edits, no schedule edits, no scattered filter changes.
- **UI surface:** the right-click menu (`ui/orders.rs`) uses an internal `MenuAction` enum for button selection that maps to `PlayerCommand` via `menu_action_to_command`. HUD muster + draft-toggle, inspector Read/Lecture/Encode buttons, and military right-click all emit `PlayerCommandEvent`. No UI code touches `PersonAI` / `ActionQueue` directly.

## ActionQueue and typed Task variants (`typed_task.rs`)

`aq.current: Task` is canonical "task running now"; `Task::Idle` default; every Person spawn site bundles `ActionQueue::idle()`. Behind `current` sits a fixed-capacity `queued: [Task; 4]` prefetch ring (private; access via `enqueue` / `pop_next` / `peek_next` / `queued_len` / `queued_is_empty` / `clear_queued`).

- **Producers** (HTN dispatchers, `ui/orders.rs`, `teaching.rs::ReadItem`) route through `aq.dispatch(task)` — enqueues then promotes head into `current` if `current == Idle`.
- **Consumers** (executor exit paths in `gather.rs`, `dig.rs`, `corpse.rs`, `construction.rs`, `items.rs`, `production.rs`, `teaching.rs`, plus `MilitaryMove` arrival in `military.rs`) call `aq.advance()` instead of writing `current = Idle`.
- **External preempts** (`dispatch_player_command_system`, hunter demote, `goal_dispatch_system` stale reset) call `aq.cancel()`, dropping both `current` and the queue.
- Per-tick "pin" sites (lecture/teach pin writes in `teaching.rs`) stay as direct `aq.current = X` writes — idempotent re-assertions, not fresh dispatches.

**Variants:** `Idle`, `WalkTo { tile, z, why }`, `WithdrawGood { filter }`, `WithdrawMaterial { resource_id, qty }`, `WithdrawFood { tile }`, `Equip { slot, resource_id }`, `Construct { blueprint }`, `Gather { tile }`, `Dig { tile }`, `Scavenge { target }`, `Read/Teach/HoldLecture/AttendLecture { tech }`, `PickUpCorpse { corpse }`, `HuntPartyMuster { hearth }`, `Hunt { prey }`, `HaulCorpse { dest }`, `Butcher`, `TameAnimal { target }`, `Sleep { bed }`, `Eat`, `HaulToBlueprint { blueprint }`, `HaulToCraftOrder { order }`, `WorkOnCraftOrder { order }`, `DepositToFactionStorage { resource_id }`, `WalkAndTakeFromMember { target, resource_id, qty }`, `PlayThrow`, `PlayPlant { tile }`, `Explore { kind }`, `Migrate`, plus `Lead/Defend/Raid/RescueAlly/Socialize/Play` (faction/social).

**Rules:**

- Systems mutating typed task **must** include `&mut ActionQueue` alongside `&mut PersonAI`.
- Every executor exit (success/abort/timeout/precondition fail) must `aq.advance()` (or `aq.cancel()` for chain-drop).
- External teardowns clear both `aq` and the legacy `task_id`.
- `military_task_system`, `withdraw_good_task_system`, `withdraw_material_task_system` fall back to `Idle` if `task_id == X` but `aq.current` is the wrong variant — defence in depth.

## HTN domain (`htn.rs`)

- **`AbstractTask` / `AbstractTaskKind`** — high-level goal a method decomposes (`Sleep`, `Eat`, `AcquireFood`, `AcquireGood { resource_id }`, `StockpileFood`, `Scout`, `EquipHuntingSpear`, `ReturnSurplus`, `TameWildHorse`, `PlantFromStorage { resource_id }`, `HarvestPlant`, `ConstructBlueprint`, `Socialize`, `DeliverMaterialToCraftOrder { resource_id }`, `WorkOnCraftOrder`, `HarvestGrainForCraftOrder`, `Play`, `JoinHuntParty`, `EngagePrey`, `DeliverHuntKill`, plus four single-step combat/faction goals).
- **`Method` trait** — `precondition` / `utility` / `expand(ctx) -> Vec<Task>` / `flags` / `tech_gate` / `profession_gate` / `policy_gate` / `name`. `policy_gate` returns `&'static [(ResourceId, RequiredFlag)]` for methods that require a specific `ResourceControlPolicy` flag.
- **`MethodRegistry`** — `Resource`, `AHashMap<AbstractTaskKind, Vec<Box<dyn Method>>>`. Built once in `SimulationPlugin::build` via `register_builtin_methods`. Read-only after init.
- **`MethodFlags`** — plain `u8` bitmask. `MF_UNINTERRUPTIBLE` survives goal flips.
- **`MethodId`** — `u16` newtype with one `pub const` per registered method. Stable identity for `MethodHistory` keying and `PersonAI.active_method`.
- **`MethodHistory`** — per-agent ring (`METHOD_HISTORY_LEN=6`, TTL 600 ticks) of `(MethodId, MethodOutcome, tick)`. Spawned at every Person spawn site.
- **Failure-biased scoring:** multi-method dispatchers score via `score_method_with_history(method, abstract_task, ctx, history, now) = utility - failures * METHOD_FAILURE_PENALTY` (penalty 0.4). Routing/missing-target failures push `FailedRouting` / `FailedTarget`.
- **Cancel paths record failure.** Every executor `aq.cancel()` path that drops a chain due to invalid target / unreachable trailing leg pushes the agent's `active_method` onto `MethodHistory` before cancelling, via `record_target_failure(...)` (target loss) or `record_routing_failure(...)` (chain trailing-leg). Sites: `gather::finish_gather`, `items::item_pickup_system`, `items::finish_scavenge`, `production::finish_withdraw_material`. `MethodId::UNKNOWN` is a sentinel for cancels where `active_method` was already cleared.
- **Goal-flip Abandoned (`record_abandoned_method_system`).** ParallelA, after `goal_update_system`, filtered on `Changed<AgentGoal>`. Pushes `MethodOutcome::Abandoned` against the agent's `active_method` and clears it — *unless* the method is `MF_UNINTERRUPTIBLE`, in which case `goal_dispatch_system`'s preserve-arm matrix keeps the chain alive across the flip. `MethodRegistry::flags_by_id` looks up the method's flags by stable id.
- **Terminal Explore fallback** (`htn_acquire_food_dispatch_system` + `htn_stockpile_food_dispatch_system`). When every registered method's precondition fails, the dispatcher falls through to a synthetic `Task::Explore { kind: AnyEdible }` rather than returning silently. Stamps `MethodId::TERMINAL_EXPLORE` on `ai.active_method`. Routes via the same `pick_explore_tile` + `assign_task_with_routing` path the regular `Task::Explore` arm uses.
- **Reachability-aware deposit pick** (`StorageTileMap::nearest_for_faction_reachable`). Dispatchers picking a deposit/withdraw storage tile filter out tiles whose chunk isn't reachable from the agent's chunk per `ChunkConnectivity`, falling back to the connectivity-blind result when every tile is rejected. Wired into `htn_acquire_food_dispatch_system`, `htn_acquire_good_dispatch_system`, and `htn_stockpile_food_dispatch_system`.
- **Reachability-aware vision pickers** (`CurrentVision::nearest_gather_target` / `nearest_scavenge_target`). Both take an `is_reachable: impl Fn((i32,i32)) -> bool` closure with the same two-pass fallback. Each htn.rs dispatcher builds a `reach_from_agent` closure once and threads it into every visible-target pick — closes the symmetric gap where a Manhattan-closest visible plant or ground item sits across an impassable boundary. The `SharedKnowledge::nearest_in_tier_set` cluster path still uses Manhattan distance only.
- **Chronic-failure release** (`goals::chronic_failure_release_system`). Economy schedule, every `TICKS_PER_DAY/4`. For each agent (any goal except `Survive` / `Sleep`) with ≥3 non-success `MethodHistory` entries within the TTL window: stamps the goal into the `GoalCooldown` ring (`(goal_disc, expires_tick)` × 4 slots; oldest evicted on overflow); also writes the legacy `craft_until_tick` for `should_craft`'s fast path. Then *bifurcates*: a `JobClaim`'d agent has the claim removed + claimant slot released so the chief reposts to a better-positioned worker; an autonomous (no-claim) agent is inserted into `ForceGoalReevaluate` so `goal_update_system` bypasses the 200-tick cadence and flips the goal *next* tick. `Survive` / `Sleep` are exempt — terminal Explore (Phase 3) is the relief valve there. Orthogonal to `job_claim_release_system`'s stuck-idle `fail_count` bump (which fires on `task_id == UNEMPLOYED` for STUCK_FAIL_INTERVAL ticks); this one fires on the *quality* of recent attempts.
- **`ForceGoalReevaluate`** (`Resource`, `AHashSet<Entity>`). Drained by `goal_update_system` on each entity it processes — the agent's `task_id` is reset to UNEMPLOYED, target cleared, then the normal goal-selection logic runs. Ensures the new cooldown bites immediately rather than after the next 200-tick cycle.
- **Success recording:** dispatchers stamp `PersonAI.active_method = Some(method.id())` on successful `aq.dispatch(...)` and clear before pushing `Failed*`. `htn_method_completion_system` (Economy) observes `aq.current == Idle && queued_is_empty && active_method.is_some()` and records `Success`.

### Dispatch systems

All run in `ParallelB`, ordered after `goal_dispatch_system`. Each system serialises on `&mut PersonAI` / `&mut ActionQueue`; Bevy schedules them in parallel where component sets allow. The craft-order + Harvest + Play dispatchers share their own `add_systems` call (Bevy's 20-tuple ceiling). Spear-arming runs *before* food dispatchers so an unarmed hunter prefers fetching their spear over eating.

Each dispatcher: gates on goal + Idle + non-Dormant + `task_id == UNEMPLOYED`, builds `PlannerCtx`, runs argmax across registered methods, routes the head via `assign_task_with_routing`, prefetches the tail on `ActionQueue`. SOLO/unsettled agents skip storage-dependent methods.

**`ClaimTarget.kind: ClaimKind`** (`jobs.rs`) — `None | Specific(ResourceId) | AnyEdible`. Mirrors `MemoryKind`. Stockpile/Haul postings yield `Specific(rid)`; chief `Calories` postings yield `AnyEdible`; Build yields `None`. Food dispatchers gate on `ClaimTarget::is_food()` (true for `AnyEdible` or `Specific(rid)` where `rid.is_edible()`), so adding a new edible to `assets/data/resources/*.ron` automatically participates without code changes.

**Subsistence vs. coordinated stockpile (`htn_stockpile_food_dispatch_system`).** `AgentGoal::GatherFood` is set in two regimes: (1) **subsistence reflex** — `goal_update_system`'s autonomous fallback when `prioritize_food` (food ratio < 1.0); no `JobClaim`. (2) **chief / household / self-post coordination** — agent holds `JobClaim::Stockpile{food}` mapped from a posting via `posting_goal`. The dispatcher serves both — claim is *optional*, not required. The dispatcher's deposit target is household-first: if the agent has `HouseholdMember` and the household sub-faction owns a `FactionStorageTile` (Market preset's `seed_market_households`), the chain ends in the household's larder; else it falls back to the village's. The dispatcher also runs *after* every specialised dispatcher in ParallelB (corpse delivery / hunt engage / hunt join / construction / etc.) and filters out `Carrying` / `MigrationTarget` / `TraderPlan` carriers plus `Profession::{Bureaucrat, Trader}` — subsistence is the lowest-priority autonomous work.

### Chain handoffs

Trailing legs of multi-task chains live in exit helpers on the head executor:

- **`production::finish_withdraw_food`** — primes `task_id = Eat`.
- **`production::finish_withdraw_material`** — releases reservation; routes `HaulToBlueprint` / `HaulToCraftOrder` / `Planter`; primes `Equip` / `PlayThrow` / `Play` in-place.
- **`gather::finish_gather`** (5 exit sites) — routes `DepositToFactionStorage` to nearest storage; routes `HaulToBlueprint` / `HaulToCraftOrder`; primes `Eat` for the Forage chain. Routing failure drops the chain (`aq.cancel()`). Takes a `FinishGatherOutcome::{Completed, TargetInvalid}` parameter: `TargetInvalid` (stale plant, wrong tile kind) calls `aq.cancel()` so the queued tail isn't run with empty hands; `Completed` calls `aq.advance()` and routes the tail as before.
- **`items::finish_scavenge`** — mirror of `finish_gather`; primes `Eat` for the AcquireFood path. Same `FinishGatherOutcome` parameter — GroundItem despawned mid-walk fires `TargetInvalid` so the chain doesn't walk to storage empty-handed.
- **`drop_items_at_destination_system`** — executor for `TaskKind::DepositResource`; dumps hands at `dest_tile`, credits any `JobClaim::Stockpile`, `aq.advance()`s.

### Stale-reset / catch-all (`goal_dispatch_system`)

Runs before HTN dispatchers. Covers only:

- **Stale-task reset** — leftover `task_id != UNEMPLOYED` clears to `(Idle, UNEMPLOYED)` (and `aq.cancel()`) unless the goal legitimately keeps it (preserve-arms enumerate the (goal, task_id) pairs each HTN dispatcher's chain needs to survive). Catch-all at bottom resets `Explore` arrivals.

### Adding a method

New `AbstractTask` variant (if needed) + `Method` impl + register call in `register_builtin_methods`. Method tests in `htn::tests` exercise the trait directly without an `App`. New abstract-task *kinds* also need a per-goal dispatch system (or new branch).

## Memory & gathering (`shared_knowledge.rs`)

3-tier `SharedKnowledge` map is the system of record for static-resource queries.

- **`SharedKnowledge` resource** — per-tier `KnowledgeMap`s keyed by `KnowledgeTier::{Household(u32), Settlement(SettlementId), Faction(u32)}`. Resources stored as `ResourceCluster` influence nodes (center + radius + LRU `representative_tiles[4]` + `estimated_count`); `report_sighting` merges nearby same-(kind,owner) sightings within `CLUSTER_MERGE_RADIUS=8`, `report_depleted` decrements and despawns at zero. `nearest_in_tier_set(TierSet, kind, from, owner_filter, claim_penalty)` walks finest-tier-first via spiral chunk search. `cluster_decay_system` drops clusters older than `CLUSTER_DECAY_TTL_TICKS=25200` once per game-day. **`estimated_count` mirrors rep-slot occupancy** — `report_sighting` recomputes it from `representative_tiles` after every `push_rep`. **`ResourceCluster::nearest_target_tile` returns `Option`** — `None` when every rep slot is empty (no `c.center` fallback). `KnowledgeMap::nearest` filters such clusters out so an exhausted cluster naturally falls out of selection until vision repopulates the LRU.
- **Cluster-saturation skip:** `nearest_with_cluster_filter` and `nearest_in_tier_set_with_cluster_filter` accept a `Fn(&ResourceCluster) -> bool` predicate. `GatherKnowledge::nearest_target_tile` injects `gather_claims.cluster_is_saturated(...)` so the dispatcher walks past clusters whose rep slots are already at `MAX_PARALLEL_GATHERERS_PER_CLUSTER = 3`.
- **Pressure-aware rep selection:** `ResourceCluster::pick_least_pressured_rep(from, penalty)` minimises `chebyshev(rep, from) + penalty(rep)` across the LRU rep slots. Plumbed into `KnowledgeMap::nearest` (during cluster scoring) and `nearest_in_tier_set` (extraction path) so two workers querying the same cluster fan out across rep slots.
- **`ResourceOwner`** — `Public | Person(Entity) | Household(u32) | Settlement(SettlementId) | Faction(u32)`. `is_accessible_to(viewer, household, settlement, faction)` gates harvest without theft semantics.
- **`LandClaim` component** — stamped on `Plant` entities at planting (`production.rs` Planter arm): `HouseholdMember` planter → `Household(id)`; chief Farm posting → `Faction(id)`; otherwise → `Person(self)`. Wild reseed leaves no `LandClaim` → `Public`.
- **`GatherClaims`** (`gather_claims.rs`) — mutex-wrapped per-`(tile, kind)` reservation map. `pressure(tile, now, viewer)` feeds `nearest_in_tier_set`'s `claim_penalty` (×4 weight) so two agents fan out across separate clusters. **Staked at dispatch by all four resource-target dispatchers** right after a successful `Task::Gather` `assign_task_with_routing`, with TTL from `suggested_expiry`. Released at every `gather::finish_gather` exit (via `release_gather_claim` threaded through `GatherRoutingResources`) and on `goal_dispatch_system`'s stale-task `aq.cancel()`. `CurrentVision::nearest_gather_target` *also* consumes the same `pressure × 4` weight. `gather_claim_expiry_system` (Economy, every 150 ticks) sweeps expired entries. Cluster-level mutex primitives: `pressure_across_tiles(rep_tiles, now, viewer)` and `cluster_is_saturated(rep_tiles, now, viewer)`.
- **Vision write-through (`memory.rs::vision_system`)** — every active agent reports plant/item/prey/stone sightings to `SharedKnowledge`'s **finest tier** (Household if a member, else Faction). Settlement and Faction tiers materialise via gossip propagation. **Vision is additive only** — `report_depleted` is never called from vision. Cluster shrinkage happens at gather-arrival via `gather_system::report_depleted` on every `finish_gather` exit. **Empty-arrival paths in `gather_system`** (despawned plant, immature plant, invalid tile) push `MethodOutcome::FailedTarget` into the agent's `MethodHistory` and clear `ai.active_method`, so `score_method_with_history` biases the next dispatch away.
- **Vision-first dispatch (`memory.rs::CurrentVision`)** — alongside the SharedKnowledge write-through, `vision_system` populates a per-agent `CurrentVision { entries: Vec<VisionEntry> }` (cleared at the top of each bucket-active pass). Entries record `{ kind, tile, entity, owner }` for every plant / GroundItem / prey / stone tile the agent saw this pass; food ground items are dual-written under `MemoryKind::AnyEdible` *and* `MemoryKind::Resource(rid)`. The four resource-target dispatchers consult `CurrentVision::nearest_gather_target` / `nearest_scavenge_target` *before* `gk.nearest_target_tile`; memory is only the fallback. Per-agent buffer freshness is bounded by bucket cadence (~20 ticks). The buffer never feeds depletion semantics.
- **Stale-target neighbor-scan retarget** (`gather.rs`): when `gather_system` arrives to find the planned plant despawned or `stage != Mature`, `retarget_neighbor` scans chebyshev `P6B_RETARGET_RADIUS = 2` of `cur_tile` for a same-`MemoryKind` mature plant (filters by `active_gather_claim.kind`). On a hit: routes to the new tile, atomically swaps the cluster claim, rewrites `aq.current`, resets `work_progress = 0`, stamps `last_retarget_tick`. Throttled to one swap per agent per `P6B_RETARGET_COOLDOWN = 40` ticks; after the cooldown the next stale arrival drops back to the legacy `FailedTarget` path.
- **HTN under-foot fast path** (`plants.rs`): `nearest_mature_plant_under_agent(plant_map, plant_query, kind_filter, from, radius=2)` probes `PlantMap` directly for a mature plant. Wired into `htn_acquire_food_dispatch_system` and `htn_stockpile_food_dispatch_system` *before* `current_vision.nearest_gather_target` and `gk.nearest_target_tile`, gated on `gather_claims.pressure(tile, now, actor) == 0`. Catches the agent who just walked onto a wheat tile this tick.
- **LOS (`line_of_sight.rs::has_los`)** — 3D Bresenham with linearly interpolated z. Rays walk at **eye height** = `current_z + 1` (1 voxel above the standing tile). Shared by `fog::fog_update_system` and `vision_system`.
- **Tier gossip propagation (`knowledge.rs::cluster_tier_promotion_system`)** — Economy, every 200 ticks. When a `HouseholdMember` socialises within 3 tiles of a `Profession::Bureaucrat` or `FactionChief` of the same root faction, household-tier clusters promote to the official's `Settlement` tier. Two same-faction officials → settlement-tier clusters bubble to `Faction` tier.
- **Cluster-density-gated chief postings** (`jobs.rs::chief_job_posting_system`): `faction_knows_cluster(...)` walks the faction's `Faction(fid)` tier plus every owned `Settlement(sid)` tier. No known cluster → skip the food / Wood / Stone posting. CraftOrder-driven Stockpile postings (Skin / Tools / Cloth) and Build postings remain ungated.

`AgentMemory` is now ~30 lines: `visited_settlements: [Option<(SettlementId, u8)>; 8]` + `record_settlement` / `known_settlements`. `relationship_decay_system` walks `RelationshipMemory` only — `cluster_decay_system` owns cluster freshness.

Same seed/faction/terrain stays identical; different culture seeds diverge without depending on ad hoc zone jitter.
- **Obstacle clearing (`obstacle.rs`, `clear_obstacle.rs`):** generic Rimworld-style "cut plants / move rocks before building" abstraction. Any entity carrying a `ConstructionObstacle { resolution: ObstacleResolution }` blocks construction at its `(tile, z)`. Two resolutions: `WorkerClear { work_ticks, skill, skill_xp }` — a worker performs `Task::ClearObstacle` adjacent, the entity despawns, yields drop on the ground (no auto-haul). `Relocate` — the system synchronously shoves the entity to the nearest free tile outside the footprint via `relocate_entity_aside` (spiral chebyshev, radius 6). Plants attach `WorkerClear` at spawn; loose rocks (`spawn_chunk_loose_rocks`) attach `Relocate`; every dropped `GroundItem` from `spawn_ground_drop` also attaches `Relocate` so future blueprints over them shove them aside.
  - **Per-blueprint scan:** `populate_pending_clear_system` (Sequential, before `construction_system`) observes `Added<Blueprint>`, calls `obstacle::scan_footprint(&[bp.tile], bp.target_z, …)`, pushes `WorkerClear` hits onto `bp.pending_clear`, and synchronously relocates `Relocate` hits.
  - **Build gate:** `construction_system` checks `bp.obstacles_cleared() (== pending_clear.is_empty())` — workers idle next to dirty footprints until the last entry pops.
  - **Worker pipeline:** `htn_clear_obstacle_dispatch_system` (ParallelB, after `htn_build_claimed_blueprint_dispatch_system`) routes idle Build-goal agents holding a `JobClaim::Build` (Path A) or owning a personal blueprint (Path B) with non-empty `pending_clear` to its first obstacle. `clear_obstacle_task_system` (Sequential, after movement, before construction) accumulates `ai.work_progress` against the obstacle's `work_ticks`, drops yields via `gather::spawn_ground_drop`, despawns the entity (`despawn_plant_internals` for plants; `despawn_recursive` otherwise), pops it off `bp.pending_clear`, awards XP, and `aq.advance()`s.
  - **Multi-tile interior:** `terraform::footprint_completion_system` calls `obstacle::resolve_footprint_sync` over the full `terraform_tiles` rect right before spawning perimeter wall blueprints — catches interior floor tiles that have no individual blueprint of their own (no worker is responsible for them; foundation grading takes them).
  - **Game-start seeded structures** bypass the blueprint pipeline. `clear_obstacle::clear_obstacles_under_seeded_structures` runs `OnEnter(Playing)` after `seed_starting_buildings_system`, walking `StructureIndex` and synchronously clearing every seeded tile.
  - **Late-arriving obstacles** (most importantly: loose rocks scattered by `chunk_streaming_system` into chunks that streamed in *after* the OnEnter pass) are caught by `react_obstacle_under_structure_system` (Sequential, after `sync_indexed_after_move_system`). Reactive on `Added<ConstructionObstacle>` — reads the entity's `Transform` (avoids the one-tick `Indexed` settle race) and, if the tile is already a `StructureIndex` or `BlueprintMap` entry, applies the resolution synchronously. Same pattern handles any future obstacle kind that spawns into already-built territory.
  - **Tree harvest now always despawns** (`PlantKind::Tree::harvest_despawns(_) → true`); bushes still revert to `Harvested` and regrow as before. The `WorkerClear` resolution always despawns regardless of plant kind — bushes inside a footprint don't reseed because the building covers the tile.
- **Game-start seeding (`construction.rs::seed_starting_buildings_system`):** runs `OnEnter(Playing)` after `spawn_population`. Reads `GameStartOptions::{era, seed_buildings}` and spawns era-appropriate fully-built structures around each non-SOLO faction's `home_tile` — bypasses the Blueprint pipeline. Per-era additive layout: Paleo (1 Open Campfire + 4 Beds), Meso (+2 beds), Neo (Hearth tier→Ringed, +Workbench, +Granary, +Loom, beds→8), Chalco (Hearth tier→Lined, +Shrine, +Palisade ring r=5 with east Door), Bronze (+Market, +Barracks, +Monument, walls→Mudbrick). Placement spirals outward from `home`, gating on `chunk_map.is_passable`. The chief planner's per-kind radial counters (`count_beds_near`, `count_campfires_near`, …) register these so it doesn't re-queue them. Nomadic factions branch *before* the era ladder via `seed_nomadic_camp(...)`.
- **`chief_directive_system` concurrency cap:** spawns up to `max(2, members/6)` blueprints per faction (ceiling `MAX_BLUEPRINTS_SAFETY_CAP - 1 = 19`), one new bp per 60-tick window. Pending terraform footprints count toward the cap. Chief `AgentGoal::Lead` is **no longer required** — eating/sleeping chiefs don't pause construction. Organic selected intents are tried first; per-kind in-flight counters still feed the legacy fallback `generate_candidates`. Before spawning, the fallback path rejects any footprint intersecting organic `SettlementBrain.road_tiles`, protecting roads that are planned but not yet carved.
- **Haul posting cleanup contract.** Two cooperating paths drop a `JobProgress::Haul { blueprint, resource_id, .. }` posting once the named slot is satisfied: (1) `construction_system` does eager per-tick cleanup at the deposit moment — when `bp.deposits[i].deposited` first reaches `needed`, the matching Haul posting is removed and every claimant has `JobClaim` + `ClaimTarget` stripped; (2) `chief_job_posting_system` does periodic catch-up (every 60 ticks) using the same logic via `Blueprint::slot_satisfied(resource_id)`. The two-pass cleanup ignores the "skip postings with claimants" short-circuit for the Haul branch only — claimant presence shouldn't block cleanup of a posting whose target is met.
- **CraftOrders + jobs:** `CraftOrder` carries `spawn_tick`; `faction_craft_order_system` despawns orders > `CRAFT_ORDER_TIMEOUT_TICKS=600`. `chief_job_posting_system` (`jobs.rs`) emits one `JobKind::Craft` per faction, picking the recipe with largest `demand-supply` (subject to tech + station — Workbench within 12 of `home_tile`, Loom for loom recipes). **Ingredient gate uses `faction.storage.stock_of()` (deposited stock only), not `resource_supply` (which double-counts member inventories).** When the best recipe is ingredient-blocked, the system pull-posts `Stockpile { resource_id: missing_input }` for each non-Wood/Stone shortfall.
- **Multi-worker postings (`jobs.rs::posting_target_workers`):** layered on top of the per-(kind, rid) workforce-budget cap. Build = 3 workers; Stockpile material = `(target/4).clamp(2, 6)`; Stockpile food = `(target/80).clamp(2, 8)`; Haul = 2; Farm = 3; Craft = 1.
- **Posting target curves:** `GATHER_TARGET_CAP=1500`, `MATERIAL_GATHER_CAP=96`. Food posting target multiplied by `food_seasonal_multiplier` (Spring/Summer 1.0, Autumn 1.5, Winter 1.3) read from `world::seasons::Calendar`.
- **Farmer assignment (`faction.rs::faction_profession_system`):** runs every `FARMER_ASSIGNMENT_CADENCE = TICKS_PER_DAY/4`. Anticipatory promote when per-head food < `FARMER_PROMOTE_PER_HEAD = 32`; demote when per-head food > `FARMER_PROMOTE_PER_HEAD * FARMER_DEMOTE_RATIO (1.6)`. Hysteresis prevents flapping at target.
- **WithdrawMaterial intent:** dispatcher picks the resource. Each HTN dispatcher emitting a `Task::WithdrawMaterial` reads `FactionStorage.totals` + `StorageReservations` to find the nearest tile with effective stock and stamps the resource into the typed task.

## Knowledge & technology

- **Per-person (`knowledge.rs`):** `PersonKnowledge` carries `aware` and `learned` (`u64` bitsets — learned ⊆ aware) plus `learned_at: [u32; 64]`. `complexity(tech)` is era-based (Paleolithic 1 → Bronze Age 5; Cuneiform/Lunar/City-State 6). **No cap on Learned techs.** Learning *speed* slows with current load: `learning_slowdown(stats, k) = 1 + complexity_used / (intelligence × 2)`. Applied to Study (Read/Lecture/Teach) inside `add_study_progress` (divides incoming `amount`, floor 1) and to Passive teaching's per-tick chance roll. Per-action discovery (`try_discover_from_action`) is unaffected. Founders spawn with all-Paleolithic Aware+Learned via `paleolithic_seed`; newborns inherit the OR of both parents' Aware ∪ Learned. `study_progress: AHashMap<TechId, u32>` lives on the same component; `add_study_progress(tech, amount, stats, now)` accumulates scaled ticks and at `study_threshold(tech) = complexity * STUDY_TICKS_PER_COMPLEXITY (3600)` runs `try_learn`.
- **Discovery (per-action):** `discovery_system` consumes `DiscoveryActionEvent`s emitted by `gather_system`, `production_system`, `combat_system`, `social_fill_system`. Eligible techs are those whose prereqs the actor has personally Learned and which the actor isn't already Aware of (one-shot per agent per tech). Roll `base × (1 + INT_mod × 0.1) × (1 + skill_xp / 1000)`, cap 0.5; success grants **Aware** and jump-starts `study_progress` by `complexity × INSIGHT_PROGRESS_PER_COMPLEXITY (1200)` capped strictly below `study_threshold`, so insight never auto-completes. Activity log emits `TechInsight` for the player faction. Mastery still requires Read/Lecture/Teach to push study_progress over the threshold (or passive teaching to land Learned directly).
- **Awareness gossip:** `awareness_gossip_system` (Economy) — adjacent socializing agents OR each other's `aware` bitsets every tick. Also OR-merges `AgentMemory.visited_settlements`.
- **Passive teaching:** `tech_teaching_system` (Economy, after gossip) scans pairs within 3 tiles, finds techs the teacher has Learned that the student is Aware of but not Learned, rolls `0.004 × INT_scale / learning_slowdown(student)`. Highest-complexity teachable tech transfers and lands in Learned unconditionally.
- **Directed teaching (`teaching.rs`):** Three pathways feed `study_progress`. Per-tick credit is the listed base × `INT_scale`, then divided by the student's `learning_slowdown`. **Read** — `Task::Read{tech}` adds `+1 × INT_scale` per tick (player triggers via inspector "Read" button on Aware-only rows with matching tablet/book). **Lecture** — inspector "Lecture" button writes `LectureRequest`; `apply_lecture_request_system` drafts up to 8 same-faction adults within 6 tiles and inserts `Lecturing { ends_tick=now+600, tech, anchor }` + `Attending`; `lecture_tick_system` (Sequential, after movement) credits `+2 × INT_scale` per tick. **1-on-1 teach** — right-click → "Teach"; `apply_teach_order_system` inserts `TeachingPair`/`BeingTaught` with `ends_tick=now+120`; `teach_task_system` credits `+3 × INT_scale` per adjacent tick. All carry `Drafted` while busy.
- **Faction-tech derivation:** `FactionData.techs` is a *chief-Aware* cache — `sync_faction_techs_from_chief_system` (Economy) projects the chief's `aware` bitset every tick. Used today for planning / posting authority (`can_direct_tech`).
- **Community adoption (`technology_adoption.rs`):** `FactionData.tech_adoption: [AdoptionStage; TECH_COUNT]` is the *community-Adopted* layer, derived by `derive_tech_adoption_system` (Economy, every `ADOPTION_DERIVE_CADENCE = TICKS_PER_DAY/4 = 900` ticks). Six stages: `Unknown / Rumored / Demonstrated / Practiced / Adopted / Institutionalized`. Per-tech `AdoptionScale` (Personal / Household / Subsistence / Specialist / MilitaryTransport / Institutional) drives threshold rules — Specialist needs a workshop + recent successful crafts; Institutional needs the chief learned + prereqs adopted + civic building + population scale; Personal/Subsistence land at Adopted under broad-learning shortcuts so seeded founder bands work. `RecentTechUse` is a sparse per-(faction, tech) ring of last-N use timestamps (cap 8, TTL 60 days). Helpers split the gating surface: `can_direct_tech(faction, tech)` = chief-Aware (planning); `community_has_adopted(faction, tech)` (civic / tier / material gates — also exposed as `FactionData::community_has(tech)`); `worker_can_perform(person, tech)` = per-person Learned. `community_adoption_bitset(faction)` packs the Adopted set into a `FactionTechs` snapshot so existing `best_X_for(&FactionTechs)` helpers keep their signatures. `current_era(&community_adoption_bitset(faction))` is the era projection used by civic and layout decisions — Era stays a *label*, all real gating happens per-tech via `community_has_adopted`. Per-person execution gates remain `has_learned`: HTN `tech_gate`, `JobKind::Craft` claim gating, `faction_hunter_assignment_system` (`HUNTING_SPEAR`), `mount_check_system` (`HORSEBACK_RIDING`).
- **Decay:** `derive_tech_adoption_system` is symmetric — when conditions stop being met (specialists die, civic building demolished, no recent use), the stage walks down. Downgrades are throttled to one stage per game-day via `stage_changed_at_tick`; upgrades land immediately. `step_down` walks `Institutionalized → Adopted → Practiced → Demonstrated → Rumored → Unknown`. Upgrades to `Adopted` / `Institutionalized` emit `ActivityEntryKind::TechAdopted` / `TechInstitutionalized` for the player faction; downgrades stay silent so a brief Adopted → Practiced flicker doesn't spam the log.
- **Founder seeding (`PersonKnowledge::seeded_realistic_through_era`):** the chief learns Personal+Household+Subsistence+Specialist+Institutional techs of era ≤ E; every ~8th member is a Specialist (adds Specialist+MilitaryTransport); the rest carry only Personal+Household+Subsistence as Learned. *Everyone* remains Aware of the full era, so awareness gossip + study can still spread mastery. The legacy `seeded_through_era` (every tech Learned) is retained for tests / fixtures.
- **Tablets and books (`Item.tech_payload: Option<TechId>`):** `Good::ClayTablet` (`OneHand`, 1500 g) and `Good::Book` (`Small`, 600 g). Crafted via recipes gated on `CUNEIFORM_WRITING`, Workbench station. Reading does not consume them.
- **Tablet posting (`chief_tablet_posting_system`):** every `CHIEF_TABLET_POSTING_INTERVAL=3600` ticks per faction. Reads chief's `learned`, picks the highest-complexity bit the chief Learned that <50% of adults are Aware of (skipping techs already covered by a live tablet posting), posts a `JobKind::Craft` with `tech_payload=Some(chosen)`. Player override: `PlayerCraftRequest` (inspector "Encode") — consumed every tick, `JobSource::Player`/priority 180.
- **Tech tree (`technology.rs`):** 44 techs (incl. `PORTABLE_DWELLINGS`), 5 eras, prereq DAG, per-tech `triggers: &[TechTrigger]`. `TechBonus` adds yield/storage/combat bonuses; bonuses aggregate via `faction.techs` (chief-aware).
- **Tech panel UI (`ui/tech_panel.rs`):** Per-tech icon now reflects `AdoptionStage` not chief-Aware: ◆ Institutionalized (bright green) / ✓ Adopted (green) / ● Practiced / ◐ Demonstrated / ◉ Rumored / ◎ Discoverable / ○ Locked. Hover surfaces both the stage and chief Learned/Aware state so the operator can see the gap between "the chief knows it" and "the village has it."

## Hunting pipeline (`corpse.rs` + the three-phase HTN pipeline)

- Wolf/Deer no longer drop Meat/Skin on death. `combat.rs::death_system` strips AI/needs/species and inserts `Corpse { species, fresh_until_tick }`, indexed in `CorpseMap`. `corpse_decay_system` (Economy) despawns at `CORPSE_FRESHNESS_TICKS=600`.
- **Chief hunt orders (`HuntOrder` on `FactionData`):** `chief_hunt_order_system` posts daily (`TICKS_PER_DAY=3600`, staggered by `fid`). Scans `SpatialIndex` within `HUNT_SCAN_RADIUS=40`, posts `Hunt { species, area_tile, target_party_size = 4(Wolf)/2(Deer), mustered, deployed_tick }` or `Scout`. Mid-day invalidation sweep clears stale orders.
- **HuntFood pipeline:** three HTN abstract tasks (hunter-only, `HUNTING_SPEAR`-gated, `MF_UNINTERRUPTIBLE`): `JoinHuntParty` (Muster + Travel) under chief `HuntOrder::Hunt`; `EngagePrey` (Hunt + PickUpCorpse) at the area; `DeliverHuntKill` (HaulCorpse + Butcher) gated on `Carrying`. Butcher drops `species_yield()` Meat+Skin, despawns corpse.
- **Weapon precondition:** `HuntPreyMethod` / `MusterAtHearthMethod` / `TravelToHuntAreaMethod` all gate on `PlannerCtx.agent_has_weapon` (true when a `weapon` resource sits in `Equipment[MainHand]`, `Carrier` hands, or `EconomicAgent.inventory`). The engage / join-party dispatchers compute the field by reading `&EconomicAgent + &Carrier + Option<&Equipment>`. `PickUpFreshCorpseMethod` deliberately skips the gate (no combat — agent just carries the kill). An unarmed hunter under `HuntOrder::Hunt` therefore can't dispatch `Task::Hunt` / `Task::HuntPartyMuster` and falls through to `htn_equip_hunting_spear_dispatch_system`.
- **Scout:** `htn_scout_dispatch_system` + `ScoutForPreyMethod` emits `Task::Explore { kind: Prey }` while chief holds `HuntOrder::Scout`; chief flips to Hunt the moment `vision_system` writes a prey sighting.
- **EquipHuntingSpear:** **goal-agnostic** — runs for any unarmed Hunter (no goal filter) so a hunter mid-`Lead` / `Defend` / `Socialize` still fetches their spear when `HuntOrder::Hunt` is live. Gates: Hunter profession + Learned `HUNTING_SPEAR` + no weapon in any slot + faction storage has weapon stock. Chain preservation across goal flips keys on `ai.reserved_resource == Some(weapon())` for the `WithdrawMaterial` leg and `Task::Equip { resource_id == weapon() }` for the `Equip` leg, so the chain survives external goal-update cycles without explicit per-goal arms.
- **Live chase (`combat::hunt_chase_system`):** Sequential schedule, after `movement::sync_indexed_after_move_system` and before `combat::combat_system`. Reads `aq.current = Task::Hunt { prey }`, looks up the prey's `Transform + Health`, and acts:
  - Prey despawned → `record_target_failure` + `aq.cancel()` (the next dispatch tick re-evaluates with the recency bias).
  - Prey dead → no-op; `combat_system::death_system` handles `aq.advance()` in the same tick.
  - Prey moved (`prey_tile != ai.dest_tile`) and Chebyshev distance ≤ `HUNT_LEASH_RADIUS (30)` → update `ai.dest_tile` / `ai.target_tile` / `ai.target_z` to the live prey tile; if state was `Working`, drop back to `Seeking` (so `movement_system`'s `Following`-arm replan picks up the goal mismatch); clear `PathFollow.segment_path` + status → `Idle` to force a fresh A* request.
  - Prey beyond leash → cancel with `FailedTarget` so the dispatcher can pick a closer prey or transition to Scout.
- `corpse_follow_system` (Sequential, after `movement_system`) snaps corpse Transform to carrier. `respond_to_distress_system` recruits any same-faction Hunter within `HUNTER_RESPOND_RANGE=50` regardless of LOS, removing `Carrying`.
- **`Carrying(Entity)`** — marker for "carrying corpse E"; inserted at pickup arrival, removed at butcher / decay / rescue / muster / hunter-demote.

## Plant lifecycle (`plants.rs`)

Calendar-driven, season-edge-triggered. Replaces the previous per-frame tick clock — season modulates the *rate* of growth, not whether transitions can fire. `Plant.growth: u16` is a stage-time accumulator; on each season change `plant_lifecycle_system` adds `season_growth(kind, prev_season)` and runs at most one stage transition per plant.

- **Per-season growth rates (`season_growth`):** Winter is dormant for all kinds (zero growth, lethal for Grain). Per kind: Grain Spring/Summer/Autumn = 4/5/2; BerryBush 5/4/2; Tree 4/5/3.
- **Stage thresholds (`stage_threshold`):** Grain Seed/Seedling/Mature = 3/5/2 (annual; no Harvested stage). BerryBush 5/30/4 with Harvested mirroring Seedling at 30 → ~3 calendar years to first fruit. Tree 12/48/18 with Harvested mirroring Seedling → ~4 calendar years from seed to maturity.
- **Sprout chance:** Seed → Seedling rolls `fastrand::f32() < 0.20`; failure despawns the entity. Combined with §scatter, this culls exponential spread.
- **Mature → fruiting:** at the transition, a single dice roll fires (`PlantKind::Grain` 20% / `BerryBush` 10% / `Tree` 5%) for one Seed in chebyshev radius (Grain/Berry r=2, Tree r=3) onto a `Grass | Farmland` tile. Post-roll Grain enters Overripe, BerryBush reverts to Harvested, Tree reverts to Mature.
- **Winter mortality:** at Winter onset, every Grain plant despawns. Mature grain performs the scatter dice first (the "wheat goes to seed in winter" path). Immature grain just dies.
- **System scheduling:** `plant_lifecycle_system` runs in `FixedUpdate` `SimulationSet::Sequential`, ordered after `advance_calendar_system`. Edge-triggered via `Local<Option<Season>>` — fires only on season change (≤4 calls per game year).
- **No fertility multiplier on growth speed.** Removed when migrating off `growth_ticks`; can return as a per-season bonus or yield modifier later.
- **`deer_graze_system`** still resets a grazed bush to `Harvested` with `growth = 0`; nothing else in the file changed externally.

## Animal spawn distribution (`animals.rs::spawn_animals`)

Initial-condition only. Per-species `SocialPattern` picks a `(group_min..=group_max, cluster_radius)`: `HERD` (Deer/Horse/Cow — 8–15, r=5), `PACK` (Wolf/Pig/Rabbit/Fox — 3–6, r=3), `SOLITARY` (Cat — 1, r=0). `cluster_spawn_tiles` pops shuffled centers from the species' biome pool (`forest_tiles` / `grass_tiles`) and lays members on tiles inside the biome `AHashSet` within `cluster_radius` of the center. Total counts (`WOLF_COUNT`, `DEER_COUNT`, …) and per-species component bundles are unchanged. Runtime cohesion is not modeled — `animal_movement_system` wanders independently.

## Nomadic mode (`nomad.rs`, `nomad_pool.rs`, `wild_herd.rs`, `pack_deploy.rs`, `sedentary_collapse.rs`)

See root `CLAUDE.md` for the high-level survey. Detailed system list:

- **`Lifestyle::{Settled (default), Nomadic}`** on `FactionData`. Households inherit parent village's lifestyle via `spawn_household` alongside `land_policy`.
- **`Deployable { packed_form, refund_pct, refund_resource, refund_qty }`** (`pack_deploy.rs`) on every nomadic structure. `Deployable::fully_packable(rid)` for full-carry shelter (Bedroll/Yurt); `Deployable::refund_only(pct, resource, qty)` for sticks-and-leaves teardown (Tent uses `refund_only(0.5, wood, 6)` → 3 wood drops as `GroundItem` on migration teardown). `compute_refund_drop()` returns `floor(qty × pct)`.
- **Build sites:** `BuildSiteKind::{Bedroll, Tent, Yurt}`. Bedroll (1 skin + 2 wood, no gate) finalizes as `(Bed { tier: Crude }, Deployable, StructureLabel("Bedroll"))` — existing sleep dispatch finds it via `BedMap`. Tent (6 wood + 3 skin). Yurt (8 wood + 6 skin, gated on `PORTABLE_DWELLINGS`, Workbench station) packs into `packed_yurt` good (`bulk: two_hand`, `weight_g: 80000`).
- **Camp seeding** (`seed_nomadic_camp`): hearths via `paleolithic_hearth_positions`, then `seed_bedrolls_around_hearth` (radial 2..=5, one per founder), `seed_tents_around_hearth` (outer 5..=7, ~1 per 4 founders), `seed_yurts_around_hearth` (Neolithic+, inner 3..=5, capped 2). `auto_found_default_settlements_system` short-circuits on nomadic; `spawn_population` skips the per-faction `FactionStorageTile` spawn for nomads.
- **Camp lifecycle (`FactionData.camp_state` + `migration_phase` + `nomad_autopilot`):** every nomadic faction carries `CampState::{Pitched, Packed { since_tick }}` and the AI-autopilot phase machine `MigrationPhase::{Idle, Surveying { started_tick, scouts, quadrants }, PendingCommit { target, chosen_tick }, Walking { target }}`. AI factions (`nomad_autopilot == true`, default for AI archetypes) run through the full Surveying → PendingCommit → atomic commit → Walking → Idle pipeline. Player factions (`nomad_autopilot == false`, flipped by `spawn_population` for `Lifestyle::Nomadic`) skip the FSM entirely — free-form migration is **Pack Camp → wander with regular Move orders → Pitch Camp Here**. No route/waypoint/Start-Migration ceremony.
- **Migration** (`nomad.rs`):
  - `nomad_migration_system` (Economy, daily, exclusive `&mut World`): per nomadic faction whose `MigrationPhase == Idle` and cooldown elapsed, scores local food clusters within `NOMAD_FORAGE_RADIUS = 24`. If `< members × 3`, picks 2-3 lowest-need members (skip Drafted, skip chief), stamps `AgentGoal::Scout` + `ScoutAssignment { quadrant, target_tile, assigned_tick }`, and transitions `MigrationPhase = Surveying { started_tick, scouts, quadrants }`. Quadrant targets sit at chebyshev `SURVEY_SCOUT_RADIUS = 100` from `home_tile` so scouts seed faction-tier `SharedKnowledge` far beyond the band's normal vision before the chief commits.
  - `nomad_survey_dispatch_system` (ParallelB) routes idle `AgentGoal::Scout` + `ScoutAssignment` agents toward their target via `Task::Explore { kind: AnyEdible }`; vision write-through populates clusters as they walk.
  - `nomad_survey_completion_system` (Economy, daily, exclusive `&mut World`): when a Surveying faction's `now - started_tick >= SURVEY_WINDOW_TICKS = TICKS_PER_DAY*4`, runs `pick_migration_target` over the now-enlarged knowledge, writes `pending_migration`, transitions to `PendingCommit`, strips Scout state from surveyors, and drops them into `ForceGoalReevaluate`.
  - `pick_migration_target` (Phase D): scoring is food + herd + water + biome-season + danger + recency + **continuous distance penalty** (`-d * DIST_WEIGHT`, `DIST_WEIGHT = 0.4`). The legacy 25-60 hard annulus is dropped — `NOMAD_MIN_TARGET_DIST = 8` (just rejects literally-on-camp candidates) and `NOMAD_MAX_TARGET_DIST = 200` (generous safety cap). Scouted clusters from far quadrants now compete on score, not distance.
  - `nomad_migration_commit_system` (Sequential, every tick, exclusive `&mut World`) drains `pending_migration`. Steps: extract `pack_camp_assets(world, &[(fid, anchor, radius)])` shared with player Pack — does redistribute_essentials → pack pass → despawn + refund pass — then `seed_nomadic_camp` at target → bulk-stamp `MigrationTarget` (with `bounce_count: 0`) on every band member → set goal to `MigrateToCamp` → flip `camp_state = Pitched`, `migration_phase = Walking { target }` → insert `FollowingBand { faction, last_redirect_tick }` on every owned `Tamed` animal so the redirect survives Dormant LOD via `following_band_animal_redirect_system` (ParallelA, every `TICKS_PER_DAY/4`).
  - `pack_camp_assets(world, packs)` takes per-pack `(fid, anchor, radius)` triples; radius derived from `seed_nomadic_camp_extent(members, era)` (8 + members/6 + 4, clamped 12..=22) so large bands sweep their actual footprint. Reused by `apply_pack_camp_command_system`.
  - `MigrationTarget { tile, started_tick, last_dispatched_tick, bounce_count }` Component + `WalkReason::Migration` + `TaskKind::Migrate`. `bounce_count` (cap 2) tracks connectivity reroutes to the band centroid before the marker is released.
  - `nomad_migration_dispatch_system` (ParallelB, `ParamSet`-split for centroid pre-snapshot): drops the legacy `Without<Drafted>` filter so drafted hunters migrate with the band. On connectivity dead-end, snapshots `(fid → reachable peer tiles)` and reroutes `target.tile = median peer`, incrementing `bounce_count`; only releases when `bounce_count >= 2` AND no reachable peer exists.
  - `nomad_migration_arrival_system` (Sequential, after `movement_system`) strips the marker on chebyshev arrival within `MIGRATE_ARRIVAL_RADIUS = 4`, `MIGRATE_TIMEOUT_TICKS = TICKS_PER_DAY*3`, or `MIGRATE_STALL_TICKS = TICKS_PER_DAY/2` Idle dispatch inactivity. Drops the goal back to `GatherFood` and inserts the agent into `ForceGoalReevaluate` so Survive-tier needs preempt naturally next tick (no more hardcoded GatherFood). Tally remaining migrators per faction; when zero remain in `Walking`, flip `migration_phase = Idle`.
  - `goal_update_system` carries `Option<&MigrationTarget>` *and* `Option<&ScoutAssignment>` (bundled in a sub-tuple to fit the 16-query ceiling). Both have high-priority preempt branches; Survive-tier needs / under-raid / rescue still preempt above. `goal_dispatch_system` preserve-arms: `(MigrateToCamp, Migrate)` and `(Scout, Explore)`.
- **Player Pack / Pitch — free-form migration (`PlayerCommand::PackCamp` / `PitchCamp { tile, z }`):** chief-actor faction-scoped commands written to `PendingCampOps`. Dispatcher validates (mobile faction, correct camp state, target passable + reachable + ≥ `MIN_PITCH_DISTANCE = 0` from current centroid — the player can pitch literally on the tile they packed from). **`apply_pack_camp_command_system` is observable HTN labor**: it first stamps non-drafted band members with `PackingDuty`, preempting stale autonomous task chains, releasing storage/gather reservations, and dropping `JobClaim` ownership so every eligible worker is cleanly `UNEMPLOYED`; then it enumerates every `Deployable` structure within `seed_nomadic_camp_extent` of `home_tile`, picks the nearest unused band member per structure, and dispatches `Task::UnpitchStructure { structure }` via `assign_task_with_routing`. Workers walk to each shelter, accumulate `work_progress` to `UNPITCH_WORK_TICKS = 40`, then `unpitch_structure_task_system` (`nomad_pack_labor.rs`, Sequential after movement) despawns the entity and drops its `packed_form` / `packed_bundles` / refund as `GroundItem`s at the tile. `camp_state` flips to `Packed { since_tick }` immediately so the goal gate stops settled-life work and members eat/sleep/socialize between dismantle jobs. The AI fast path (`pack_camp_assets_atomic`) is still used by `nomad_migration_commit_system` for dormant LOD bands. `apply_pitch_camp_command_system` keeps the synchronous `seed_nomadic_camp` call — the player chose the spot, the camp comes up immediately. - **Manual scout dispatch + persisted candidates (`PlayerCommand::SendScout { direction, range }`):** chief-actor command queues a `PendingManualScout`; `apply_manual_scout_command_system` picks the least-needy non-Drafted non-chief member and stamps `ScoutAssignment { kind: ScoutKind::PlayerManual { faction_id }, … }` with `target_tile` at `home + direction*range`. For the player-flow band, the intent is purely cosmetic / a hint to scout-completion scoring — free-form Pack/Pitch ignores it.

Candidates carry per-tile `reasons: Vec<CandidateReason>` (`FreshWater / Pasture / Herd / Wolves / PoorShelter / LongCarry / SnowRisk / EnemyFaction / Sanitation`) computed from component-score thresholds.

- **Wild herds (`wild_herd.rs`):** `WildHerdRegistry` resource holding per-herd `(id, species, aggregate_count, leader_tile, range_center, bloomed, members, flee_until_tick, last_birthed_tick)`. `seed_wild_herds_system` (Startup, after `spawn_animals`) places `WILD_HERD_COUNT = 3` herds (2 Horse + 1 Cow) at random grassland tiles with `WILD_HERD_AGGREGATE = 120` head each. `wild_herd_migration_system` (Economy, daily): (a) **predator flee** — query `&Transform, With<Wolf>`, if any wolf within `HERD_FLEE_RADIUS = 8` of leader, set `flee_until_tick = now + TICKS_PER_DAY` and bias drift opposite the average wolf bearing (×3); (b) **water seek** — `water_within(chunk_map, tile, 4)`; if absent, `nearest_water` probes `HERD_WATER_PROBE_RADIUS = 12` and steps toward the result; (c) **camp avoidance** — for every nomadic faction's `home_tile` within `HERD_CAMP_AVOID_RADIUS = 10`, push drift away by ×4; (d) **seasonal birth** — non-Winter season + `last_birthed_tick + TICKS_PER_SEASON ≤ now` adds `WILD_HERD_BIRTH_PER_SEASON = 12` to `aggregate_count`, capped at `WILD_HERD_AGGREGATE_CAP = 200`. Order: flee dominates. Standalone `water_within` / `nearest_water` helpers are catalog-free, callable from tests. `wild_herd_bloom_system` (Sequential, every tick) bloops/collapses based on camera-focus chebyshev: enter `BLOOM_RADIUS_TILES = 32` → spawn up to `BLOOM_VISIBLE_CAP = 60` individual `Horse`/`Cow` entities (carrying a `WildHerdMember { herd_id }` marker) clustered within radius 4 of the leader; surplus stays in `aggregate_count`. Leave `COLLAPSE_RADIUS_TILES = 48` → despawn surviving members, restore aggregate. `members` filtered against `Query<&WildHerdMember>` at collapse so killed members stay dead and a hunted herd shrinks visibly.
- **Sedentarization** (`nomad_sedentarize_system`, Economy, daily, after `nomad_migration_system`): emits `SettlementLifecycleEvent::SwitchArchetype` for any nomadic faction with: (a) `member_count ≥ NOMAD_SEDENTARIZE_MIN_MEMBERS = 12`; (b) no `pending_migration`; (c) stable for ≥ `NOMAD_SEDENTARIZE_TICKS = TICKS_PER_SEASON * 4` (one game-year). On flip, `auto_found_default_settlements_system` picks up the faction next tick.
- **Reverse collapse (`sedentary_collapse.rs`):** Economy daily. For each settled top-level faction, samples three triggers: (1) `member_count < SEDENTARY_COLLAPSE_MIN_MEMBERS = 6`, (2) sustained food deficit (`food_total < members * 10`), (3) shelter loss (`bed_count_within_OLD_CAMP_RADIUS < members/3`). Failing samples bump `FactionData.collapse_streak` by `TICKS_PER_DAY`; healthy samples reset to zero. At `COLLAPSE_TRIGGER_TICKS = TICKS_PER_SEASON` emits `SettlementLifecycleEvent::SwitchArchetype { new_archetype_key: nomadic_variant_of(...) }`. `handle_switch_archetype` for nomadic destinations calls `reseed_nomadic_camp_for(world, faction_id, at_tile)`. Inspector debug panel shows `⚠ Collapse streak: N ticks` in orange for any settled faction whose streak is non-zero.
- **Activity log:** `ActivityEntryKind::CampMoved { from, to }` and `ActivityEntryKind::Sedentarized { camp }`. `nomad_migration_commit_system` and `nomad_sedentarize_system` emit them.
- **Inspector polish:** `ui/debug_panel.rs` faction section shows `Lifestyle:` + a "Camp tile" / "Home tile" label switch + `Last migration: tick N`. Hover panel surfaces "Packs as: {good_name} on migration" for fully-packable shelters or "Teardown: drops {qty} {resource} on migration ({pct}% refund)" for refund-only Tents. Inspector also shows `⇒ Migrating to (x, y)` in yellow when `pending_migration` is set.

## Pluralist Economy

Per-resource policy flags + per-settlement markets + sub-faction households + Bureaucrat/Trader professions + P2P currency + escrow + U_bid scoring + tribute + craft contracts. Currency invariant: `EconomicAgent.currency` + `FactionData.treasury` + `Settlement.treasury` + `JobEscrow.amount` is conserved across every operation.

### Settlements (`settlement.rs`)

The **economic** unit (market + treasury + market_tile), distinct from `SettlementPlan` (the layout of zones around a hearth). One faction can own many settlements; a megachunk can host many competing settlements.

- `Settlement` (Component): `id`, `owner_faction`, `market_tile`, `founding_tick`, `name`, `treasury: f32`, `market: SettlementMarket`, `peak_population`.
- `SettlementId(u32)` newtype; `SettlementMap` (Resource) with `by_id` / `by_megachunk` / `by_faction` indices.
- `auto_found_default_settlements_system` (FixedUpdate, before `settlement_planner_system`): spawns one Settlement per non-SOLO faction at its `home_tile` if missing.

### Currency + escrow

- **`pay(world, from, to, amount) -> bool`** (`economy/transactions.rs`): atomic agent-to-agent transfer. Only sanctioned way to move currency between agents.
- **`FactionData.treasury: f32`** — faction-level wealth pool, distinct from per-settlement treasuries.
- **`JobEscrow { amount, beneficiary }`** Bevy component on a sidecar entity per funded posting. Lifecycle: producer debits wallet + spawns sidecar; on payout (Phase 0 wage payout: `job_payout_system`, Economy schedule, exclusive `&mut World`, after `job_claim_release_system`) `JobEscrowIndex(AHashMap<JobId, Entity>)` resolves the escrow, `escrow.amount` is split across claimants (direct credit to `EconomicAgent.currency`; *no* `pay()` call — funds are already debited and held in escrow), `escrow.amount` zeroes, sidecar despawns; on cancellation/expiry/bench-invalid (`completed = false` on `JobCompletedEvent`), the payout system despawns the escrow with amount intact and the `on_job_escrow_remove` hook refunds `beneficiary`. **All `aq.cancel()` sites stay untouched** — refund piggybacks on entity despawn. `JobCompletedEvent` carries `claimants: Vec<Entity>`, `completed: bool`, and `target_rid: Option<ResourceId>` (Phase 3 — derived per emission site from `JobProgress::target_rid()`). Workers paid receive an `Earnings(VecDeque<EarningEntry>)` ring component (cap 16; entries carry `(job_kind, target_rid, amount, tick)`) plus an `ActivityEntryKind::WagePaid` log entry.
- **Phase 4a chief-funded postings (`wage-aware-labor-market-v2`):** `chief_post_funding_system` (Economy, exclusive `&mut World`, after `chief_job_posting_system`) scans the `JobBoard` for chief-source postings with `reward == 0.0`. For factions with a non-empty `economic_policy` map (Mixed/Market), it computes `chief_wage_for(progress)` (Stockpile/Haul/Craft → `trade_base_value(rid) * qty * CHIEF_MARGIN (0.5)`, Haul discounted ×0.5; Calories/Build/Planting → flat per-day formulas bounded by `CHIEF_BUILD_WAGE_CAP=30`), debits the faction `treasury`, spawns a `JobEscrow { amount, beneficiary: chief_entity }`, registers it in `JobEscrowIndex`, and sets the posting's `reward`. Unaffordable wages leave the posting at `reward = 0` (no escrow, chief still has the worker on a free communal contract). Subsistence factions (empty `economic_policy`) skip funding entirely so the prior reward-0 behaviour is preserved.
- **Phase 3 wage signal (`wage-aware-labor-market-v2`):** every `FactionData` carries `wage_signal: AHashMap<(JobKind, Option<ResourceId>), WageEMA { ema_per_day, last_update_tick, samples }>`. `faction_wage_signal_system` (Economy, daily — runs when `clock.tick % TICKS_PER_DAY == 0`, after `job_payout_system`) walks every member's `Earnings` ring, sums last-day payouts per `(village, kind, rid)` triple via `FactionRegistry::root_faction`, then folds samples into the village's EMA with `α = 1 − 0.5^(1/5) ≈ 0.129` (5-day half-life). Fresh keys seed straight from the sample; pre-existing keys with no fresh sample decay toward zero so an outage shows up in the signal. Per-agent cross-faction perception lives on `PerceivedFactionWages { by_key: AHashMap<(fid, kind, rid), (ema, observed_tick)> }` (cap 32, oldest-first eviction). `wage_gossip_system` (Economy, after `awareness_gossip_system`) piggybacks `AgentGoal::Socialize` within `WAGE_GOSSIP_RADIUS = 3` chebyshev: each socialiser snapshots up to `WAGE_GOSSIP_TOP_K = 4` of their most-recent observations (own faction's `wage_signal` + cached perceptions), neighbours merge entries skipping their own faction, applying an `exp(-age/TICKS_PER_DAY)` staleness penalty. Same-faction wages always read `wage_signal` directly — perceived wages exist only for cross-faction comparisons (Phase 4+ EV scoring, migration).
- **Hook registration:** `JobsPlugin::build` calls `register_component_hooks::<JobEscrow>().on_remove(on_job_escrow_remove)`.

### Per-resource policy (`economy/policy.rs`)

`ResourceControlPolicy` flags: `chief_allocates_labor`, `private_actors_allowed`, `state_sells_at_market`, `prices_fixed_by_state`, `fixed_price`. `Default` = all-communist; `capitalist()` preset.

`FactionData.economic_policy: AHashMap<ResourceId, ResourceControlPolicy>` — empty by default. `Method::policy_gate` returns `&'static [(ResourceId, RequiredFlag)]`; `method_passes_policy_gate(method, faction)` returns true iff every gate entry is satisfied. SOLO agents reject any non-empty gate.

### Sub-factions / households

A household is a `FactionData` with `parent_faction = Some(village_id)`. Reuses the entire faction primitive — storage, treasury, chief-equivalent (`household_head`), member tracking.

- **`FactionData.parent_faction: Option<u32>`** — `None` = top-level village, `Some(id)` = sub-faction.
- **`FactionData.household_head: Option<Entity>`** — analogue to `chief_entity`.
- **`FactionData.children_factions: Vec<u32>`** — reverse pointer.
- **`FactionRegistry::spawn_household(parent, home_tile, head, &catalog)`** — creates the sub-faction, wires parent/child links, sets `household_head`. **Policy inheritance:** if the parent village's `economic_policy` map is non-empty the household is stamped with `ResourceControlPolicy::capitalist()` on every catalog resource; if the parent map is empty (default-communist village) the household keeps an empty map. This lets households exist structurally under communist tribes without becoming private-contract-firing micro-firms.
- **`FactionRegistry::root_faction(id)`** — walks `parent_faction` to the village.
- **Formation:** `CoSleepTracker.bond_strength: u16` accumulates ticks of cosleep with the *same* partner. At `HOUSEHOLD_BOND_THRESHOLD` (one game-week) `household_formation_system` (Economy) spawns the household. Both members get `HouseholdMember { household_id }`.
- **Spawn-time seeding (Market preset only):** `spawn_population` (`person.rs`) calls `seed_market_households` after the per-faction member loop. Every adult founds a one-person household with its own plot tile, a dedicated `FactionStorageTile`, and `HOUSEHOLD_SEED_TREASURY (15.0)`. Subsistence/Mixed presets skip this.
- **Inheritance:** `pregnancy_system` threads the mother's `Option<&HouseholdMember>` to the newborn.
- **`HouseholdMember` is a marker only — not a faction migration.** Parents keep `FactionMember.faction_id` pointing at the village.

### Bureaucrat profession

Officials physically employed by the settlement treasury — demote when treasury can't fund them.

- **`Profession::Bureaucrat`** + `FactionData.state_funds_public_works: bool` (default false) governance flag.
- **`FactionData.bureaucrat_treasury_empty_streak: u32`** — running tally; **decoupled from bureaucrat count** so a demote-to-zero can't immediately re-promote until funds arrive.
- **`chief_bureaucrat_appointment_system`** (Economy, every `BUREAUCRAT_ASSIGNMENT_CADENCE = TICKS_PER_DAY/4`): mirrors `faction_hunter_assignment_system`. Target = `max(1, member_count * BUREAUCRAT_MIN_RATIO)`; ranks by Social skill. When `bureaucrat_treasury_empty_streak >= BUREAUCRAT_QUIT_DAYS * TICKS_PER_DAY`, target=0 → all demote.
- **`bureaucrat_salary_tick_system`** (Economy, every `TICKS_PER_DAY/24`): debits the faction's first settlement treasury, credits each bureaucrat by `BUREAUCRAT_DAILY_WAGE/24`. Treasury bottoms at 0.
- **`bureaucrat_admin_dispatch_system`** (`htn.rs`, ParallelB after combat dispatcher): for any Bureaucrat with `task_id == UNEMPLOYED + Idle + non-Dormant`, dispatches `Task::Lead { dest = settlement.market_tile }` via `assign_task_with_routing`. Direct dispatch (no HTN method).

### Chief postings gated on policy

`JobPosting` carries `poster_class: PosterClass` (`Chief / Bureaucrat / HouseholdHead / Individual`), `reward: f32`, and `settlement_id: Option<SettlementId>`.

Per-resource gates in `chief_job_posting_system`:

| Branch | Gate |
|---|---|
| Stockpile Calories (food) | `policy_for(Fruit).chief_allocates_labor` |
| Stockpile Wood/Stone | `policy_for(target_rid).chief_allocates_labor` |
| Stockpile (CraftOrder demand) | per `target_rid` |
| Haul (per-blueprint deposit) | `policy_for(slot.resource_id).chief_allocates_labor` |
| Build | `!faction.state_funds_public_works` |
| Craft | `policy_for(recipe.output_resource).chief_allocates_labor` |
| Farm | `policy_for(Grain).chief_allocates_labor` |

Default factions have an empty policy map → all chief postings fire. Capitalist factions opt out selectively.

**Workforce-budget policy gate:** `compute_workforce_budget` (`projects.rs`) consults `chief_will_post_for_slot(faction, slot)` per slot. Slots whose corresponding chief posting is policy-disabled collapse to 0 share, and their proportional cut of `usable = 1 − FREE_FLOOR` rerouted to `free`.

**Household income skim:** `split_market_earnings_with_household(world, agent, earned)` (`economy/transactions.rs`) redirects `caps.income.household_skim_pct` of market earnings to the household treasury when the agent has `HouseholdMember`.

**Household-poster path:** `household_contract_posting_system` (Economy, every `HOUSEHOLD_POSTING_CADENCE = TICKS_PER_DAY`) walks every household with `treasury >= HOUSEHOLD_MIN_TREASURY_FOR_POSTING (10.0)` and posts one paid craft contract via `post_craft_contract_from_treasury`. Recipe picked by `pick_household_recipe(MaslowTier::next_unmet(head_needs), village_techs)`: Belonging-tier head + LOOM_WEAVING-aware village → recipe 4 (Cloth); else → recipe 0 (Tools). `poster_class=HouseholdHead`, `reward = HOUSEHOLD_CONTRACT_REWARD (5.0)`, lands on the **village's** job board.

### Per-settlement markets

Production trade routes through the agent's faction's first settlement market when one exists; SOLO/unsettled fall back to the global `Market`.

- **`SettlementMarket`** (`economy/market.rs`): `calculate_price`, `sell_item`, `try_buy_item`, `clear_flow`, `set_stock` (trader fast path).
- **`market_sell_system` / `market_buy_system`**: take `Res<SettlementMap>` + `Query<&mut Settlement>` + `&FactionMember`. Disjoint borrows.
- **`settlement_price_update_system`** (Economy, alongside global `price_update_system`): walks every Settlement, ticks `update_prices`, clears bid counters. Bid-driven discovery — see `economy/CLAUDE.md` for the price formula. No synthetic demand injection.
- **`ui/economy_panel.rs`** prefers the player faction's first settlement market.

### Maslow needs + esteem/self-actualization triggers

Two additive needs on `Needs`: `esteem: f32` (Tier 4) and `self_actualization: f32` (Tier 5). Both inverted polarity (0 = unfulfilled).

`MaslowTier::next_unmet(needs)` returns the lowest-numbered tier below threshold. Strictly *additive* — does not replace `goal_update_system`'s goal selection.

- **`esteem_driven_posting_system`** (Economy, every `ESTEEM_POSTING_CADENCE = TICKS_PER_DAY`): walks agents whose `next_unmet == Esteem` AND `currency >= ESTEEM_POSTING_MIN_CURRENCY (50.0)`; posts a Torch contract via `post_craft_contract` with `reward = ESTEEM_CONTRACT_REWARD (8.0)`, bumps `esteem` by `ESTEEM_POSTING_GAIN (30.0)`. Hungry/unsafe/lonely agents never reach the gate.
- **`self_actualization_teaching_system`** (`teaching.rs`, Economy, every `SELF_ACTUALIZATION_CADENCE = TICKS_PER_DAY`): walks agents whose `next_unmet == SelfActualization` with at least one Learned tech; picks the highest-complexity Learned tech and writes `LectureRequest`. Bumps `self_actualization` by `SELF_ACTUALIZATION_LECTURE_GAIN (30.0)`.

### U_bid scoring at job-claim layer

`job_claim_system` (`jobs.rs`) branches on `posting.reward`:

- **Paid (`reward > 0.0`):** `U_bid = E(R) + priority_bonus + affinity_bonus - C_action - C_opportunity` where `E(R) = posting.reward * wealth_modifier(agent.currency) * disposition.earn_income_multiplier()` (Phase 6: `disposition` multiplier `= 1.0 + entrepreneurial/255` from the `Disposition` Component — falls back to `1.5` (the median-128 baseline) when the component is missing); `wealth_modifier(c) = 1.0 + 0.5 / (c + 50)` bounded at 1.0; `priority_bonus = posting.priority * 0.01` (Phase 6 regression fix: chief postings post at `priority = 200`, household at `180`, individual at `100`, so chief gets a +2.0 cushion that prevents `CHIEF_MARGIN = 0.5`-half-value chief postings from being strictly outcompeted by full-value private contracts at equal reward); `C_action = euclidean(agent_tile, work_tile) * BID_DIST_DISCOUNT`; `C_opportunity = 0.0` (stub).
- **Unpaid (`reward == 0.0`, chief / legacy):** legacy `priority + skill + bias - distance`.

### Trader profession (autonomous arbitrage)

`Profession::Trader`. `trader_buy_at_settlement` / `trader_sell_at_settlement` are atomic primitives in `economy/transactions.rs`. `SettlementMarket::set_stock(id, qty)` is the fast-path setter so the helpers don't double-mutate currency.

- **`TraderPlan { phase, buy_settlement, sell_settlement, resource_id, qty }`** component. `TraderPhase::TravelingToBuy | TravelingToSell`.
- **`trader_market_step_system`** (Economy, exclusive `&mut World`, after `self_actualization_teaching_system`, **`run_if(tick % 20 == 0)`**): on arrival at the phase's market tile, calls buy/sell helpers and advances or clears the plan. Seeds new plans on idle traders by scanning `AgentMemory.visited_settlements` pairs for the best Cloth gap exceeding `TRADER_MIN_GAP=0.25`. Currency floor `TRADER_MIN_CAPITAL=30.0`. `pick_arbitrage` snapshots `(price, stock, treasury)` per visited settlement once.
- **`trader_route_dispatch_system`** (ParallelB, after `bureaucrat_admin_dispatch_system`): for plan-bearing traders not at their phase target, dispatches `Task::Lead { dest }`.
- V1 scope: Cloth-only arbitrage + `TRADER_TRADE_QTY=5`.

### Tribute

Treasury-to-treasury transfer between factions — agent currency untouched.

- **`FactionData.dominance_over: Vec<u32>`** + **`subordinate_to: Option<u32>`**.
- **`FactionRegistry::set_dominance(dominant, subordinate)`** — idempotent.
- **`tribute_payment_system`** (Economy, after `bureaucrat_salary_tick_system`, every `TRIBUTE_CADENCE = TICKS_PER_DAY`): walks every faction with `subordinate_to.is_some()`; transfers `min(TRIBUTE_PER_DAY, subordinate.treasury)` to overlord. Destitute subordinates pay 0 — no debt.

### P2P craft contracts

- **`post_craft_contract(world, poster, faction_id, recipe, qty, reward, deadline)`** (`jobs.rs`): atomic — debits poster's currency, pushes a `JobPosting` with `poster_class=Individual` + contract terms, spawns a `JobEscrow` sidecar. Refuses on insufficient funds / invalid recipe / qty=0 / non-positive reward.
- Lifecycle: completion = caller zeros `escrow.amount` then despawns. Cancellation/expiry = caller despawns with amount intact; `on_job_escrow_remove` refunds the poster.
- **U_bid integration:** `reward > 0` fires the U_bid branch — wealthy poster's contract outscores an equidistant chief Craft posting.

## Land ownership (`land.rs`)

Plot-based ownership layer over the compatibility `SettlementPlan` projection generated from `SettlementBrain` parcels/districts. Plots start `Tenure::StateOwned` held by the settlement's owning faction; in Mixed/Market presets the listing system publishes them, households acquire what they can afford, the rent-collection system bills monthly with eviction after two misses, and sharecropping plots split harvest yield between tenant and landlord at gather time. Settlement-expansion plot diff (Phase 8) deferred.

- **`Plot`** (Component): `id, settlement_id, faction_id, rect, z, zone_kind, tenure, holder, base_value, last_valued_tick, missed_payments, frontage_edge, access_tile, parent_plot`. `Tenure` ∈ `{StateOwned, Leased, Sharecropping, Freehold}`. `TenureHolder` ∈ `{State, Household}`.
- **`PlotIndex`** (Resource): `by_id`, `by_settlement: u32 → Vec<PlotId>`, `by_tile: (i32,i32) → PlotId` (surface-only), `by_faction_hash: u32 → u64` for idempotency, `next_id`. `plot_at(x, y)` is the hot lookup.
- **`LandListings`** (Resource): `for_sale` / `for_lease` Vec stubs.
- **`carve_plots_system`** (FixedUpdate, Economy, after `settlement_planner_system`, before `chief_directive_system`): per faction, if `plan.culture_hash != by_faction_hash[fid]` — tear down existing plots and re-carve, valuing each new plot via `compute_plot_value`. Multi-settlement factions currently bind every plot to `SettlementMap::first_for_faction(fid)`. Plot sizes: Residential 6×6, Crafting/Storage 4×4, Agricultural 10×10; Civic/Sacred/Market/Defense kept whole-zone. `subdivide()` folds trailing slivers (`< stride/2`) into the previous slice.
- **`compute_plot_value`** (`PLOT_BASE_VALUE = 50.0`): `BASE * zone_mul * (centre_factor + home_factor) * terrain_factor`. Distance factors anchor on `Settlement.market_tile` + `FactionData.home_tile`; zone multipliers 0.6–1.5; terrain samples fertility at centre + four corners (Agricultural reweights harder). Falls back to neutral 0.5 terrain factor when the plot lies in unloaded chunks.
- **`LandPolicy`** (`../economy/policy.rs`): faction-level governance flags. Set by `land_policy_for(EconomyPreset)` in `spawn_population` and inherited by households via `spawn_household`. Subsistence = all-false; Mixed = rent + sharecrop; Market = adds sale + freehold. `default_lease_period_days = 30`, `rent_yield_pct = 0.04` (≈ 50 %/year), `default_share_to_landlord = 0.30` for Mixed/Market.
- **`tile_buildable_by`**: rejects placements when the tile sits on a plot held by another faction or another household. Wired into `construction.rs::chief_directive_system` after candidate selection — civic blueprints pass `requesting_household=None`. Wild tiles always pass. Pure-logic core in `holder_permits_build` for testing.
- **`land_listing_system`** (Economy, every `TICKS_PER_DAY/4`): publishes Sale + Lease listings for `StateOwned` plots when the faction's `LandPolicy` permits. Civic-class zones (Civic / Sacred / Defense / Market) are skipped. Cap `TARGET_LISTINGS_PER_FACTION = 8`. Sale asking = `base_value`; lease asking = `base_value × rent_yield_pct` per month, floored at `MIN_MONTHLY_RENT`. Skips `parent_plot.is_some()` plots.
- **`household_land_acquisition_system`** (Economy, every `TICKS_PER_DAY`): household sub-factions with no plot held and `treasury ≥ HOUSEHOLD_MIN_TREASURY_FOR_LEASE` browse listings owned by their parent village. Affordability bounds: lease ≤ 40 % of treasury, buy ≤ 70 %. Preference: cheapest affordable Sale → cheapest affordable Lease → Sharecrop. Transaction is a direct faction-to-faction treasury transfer (currency-conservative). Plot tenure flips, holder flips to `Household { faction_id: household_id }`. Also claims the nearest unowned same-village Agricultural plot within 12 chebyshev tiles as a child (mirrored tenure, parent-funded rent).
- **`rent_collection_system`** (Economy, every 30 game-days): for each Leased plot whose `paid_through_tick` has expired, attempts the household→landlord faction-treasury transfer. Success: advance `paid_through_tick`, reset `missed_payments`. Failure: increment `missed_payments`, still advance the cycle. On `missed_payments >= EVICTION_MISS_THRESHOLD (2)`: tenure → `StateOwned`, holder → `State { faction_id: original_landlord }`. Emits `PlotEvictedEvent { plot_entity, plot_id, plot_rect, plot_z, landlord_faction, policy }` carrying the landlord's `caps.land.eviction_policy`. Skips `parent_plot.is_some()` plots.
- **`evicted_plot_cleanup_system`** (Economy, after `rent_collection_system`): drains the queue and acts on the policy. `LeaveStructures` (default) and `RevertToState` no-op; `Demolish` walks `StructureIndex.0` over every tile in `plot_rect`, computes a refund via `Deployable::compute_refund_drop`, spawns the refund as a `GroundItem`, and `despawn_recursive`s the structure. Mudbrick walls / non-`Deployable` buildings despawn cleanly without drops.
- **Sharecropping**: `state_sharecrops` flag triggers `ListingKind::Sharecrop` entries for `Agricultural` plots. Zero upfront cost; tenure becomes `Sharecropping { share_to_landlord, paid_through_tick }`. **Harvest hook in `gather.rs`**: `lookup_sharecrop_split` checks the harvest tile's plot. On a sharecrop plot the landlord's share (`split_sharecrop_yield(qty, share_to_landlord)`, rounding floor-in-tenant-favour) is spawned as a `GroundItem` at the landlord's nearest `FactionStorageTile` via `spawn_or_merge_ground_item`; the tenant's remainder flows through `route_yield`. The `SharecropResources` SystemParam in `land.rs` keeps `gather_system` under Bevy's 16-param ceiling.
- Inspector hover (`../ui/hover.rs`) surfaces plot info — id, zone, rect, tenure, holder, value, parent-plot link.

## Game lifecycle and regions

- **`GameState`:** `SpawnSelect` (default) | `Playing`. Globe gen at `WorldPlugin::build`. Spawn-select UI in `SpawnSelect`; spawn systems on `OnEnter(Playing)`. Update-stage systems gated `in_state(Playing)`; FixedUpdate sim systems are not gated.
- **`PendingSpawn(Option<(i32, i32)>)`:** player's chosen mega-chunk; `None` = globe centre (sandbox).
- **Sandbox bypass:** `SandboxPlugin` flips state to `Playing` at Startup.
- **`SettledRegions` (`region.rs`):** `AHashMap<RegionId, SettledRegion>` + `by_megachunk` reverse index. `SettledRegion { megachunk, founding_tick, name, camera_bookmark, player_owned }`. `settle()` idempotent.
- **`MegaChunkCoord`:** `from_chunk/from_tile/center_tile/chunk_range`. `MEGACHUNK_SIZE_CHUNKS=16`, independent of climate cells.
- **Multi-focus chunk streaming (`SimulationFocus`):** `Vec<FocusPoint>` rebuilt each tick from camera + every settled region centre. Chunk DATA loads for any focus disc (camera `LOAD_RADIUS=12`, region `REGION_LOAD_RADIUS=6`); SPRITES + plants + loose rocks only inside camera focus.
- **Focus-aware LOD (`update_lod_levels_system`):** camera distance produces base LOD; entities within 8 chunks of any non-camera focus are promoted from Dormant to Aggregate.
- **Edge-walk expansion (`region::detect_edge_crossing_system`, Economy):** each tick checks every player-faction Person's mega-chunk; if unsettled, calls `settle()`, names "Outpost N", bookmarks position, emits `ActivityLogEvent::RegionSettled`.
