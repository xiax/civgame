# Simulation (`src/simulation/`)

Agent AI, factions, knowledge, hunting, and the typed-task pipeline. See the root `CLAUDE.md` for the cross-cutting `SimulationSet` ordering and global conventions.

## Behavioural test fixture (`test_fixture.rs`)

Headless `App` harness for asserting AI behaviour without rendering / UI / globe generation. Build a `TestSim::new(seed)`, scaffold terrain with `flat_world(radius, z, kind)`, spawn agents via `spawn_person(faction, tile, |b| b.hunger(...).add_inventory(...))`, optionally place storage tiles + ground items, then drive the schedule with `tick()` / `tick_n(n)`.

- Time is deterministic via `TimeUpdateStrategy::ManualDuration` — every `app.update()` advances `Time` by exactly one fixed-tick (1/20 s).
- Game state stays in `SpawnSelect` so `OnEnter(Playing)` systems (which spawn 200 people on the real globe) never fire; FixedUpdate sim systems run regardless of state.
- A camera entity is spawned at world origin so `update_lod_levels_system` doesn't drop every test agent to `Dormant`.
- Baseline regression tests live in the `baseline_behaviour` mod and pin observable behaviour of the legacy plan/task pipeline.

## Agent AI (Goals → Plans → Tasks)

- **Goals (`goals.rs`):** High-level objectives (`Survive`, `Gather`, `Defend`, `Farm`, `Build`, `Socialize`, ...) driven by Needs + Faction state.
- **Plans (`plan/`):** Multi-step sequences scored by static linear function `dot(state, plan.state_weights) + plan.bias` plus manual bonuses (persistence, ally influence, dist-weighted memory penalty). `PlanScoringMethod::Weighted` is normal; `Random` is ε-greedy. Module split: `plan/mod.rs` (types, `ActivePlan`/`PlanHistory`/`KnownPlans`, `resolve_target`, `plan_execution_system`), `plan/registry.rs` (step+plan tables), `plan/state.rs` (`build_state_vec` + `count_visible_*`).
- **`PlanId` / `StepId` (`plan/mod.rs`):** Newtype wrappers around `u16`/`u8` with a named `pub const` per registered plan/step (e.g. `PlanId::HUNT_FOOD`, `StepId::CHOP_FOREST`). Bare `PlanId(n)` / `StepId(n)` constructors are public for migration / serialisation only — every code path that writes a plan or step ID should use the named const so typos surface at compile time. `PersonAI.last_plan_id` is still `u16` because the sentinel `UNEMPLOYED = u16::MAX` predates the typed wrappers; comparisons round-trip via `.raw()`.
- **PlanFlags (`PlanDef.flags`):** `PF_EXPLORE`, `PF_SCAVENGE`, `PF_TARGETS_FOOD/WOOD/STONE`, `PF_DROP_FOOD_ON_TIMEOUT`, `PF_UNINTERRUPTIBLE`. Candidate filter reads flags, not plan-id matches. `PF_UNINTERRUPTIBLE` plans (e.g. 7 BuildBlueprint, 13–16 craft pipeline, 29 HaulFromStorageAndBuild, 33 ClaimedHaul, 34 ClaimedBuild) survive a goal flip — they end only via completion, timeout, target invalidation, precondition fail, or external preempt.
- **Tasks (`tasks.rs`):** The agent's *current* action — `TaskKind` enum (Gather, Construct, WithdrawMaterial, Hunt, Butcher, Equip, ...). Transient; managed by `plan_execution_system`. `Idle` between tasks.
- **Professions (`person.rs::Profession`):** `None | Farmer | Hunter`. Persistent role. Farmer auto-assigned by `faction_profession_system` when food < 100. Hunter is chief-driven for every faction via `faction_hunter_assignment_system` (Economy, every `TICKS_PER_DAY/4`): target headcount = `max(1, adults*0.20)` × martial × prey-density, capped at adults/2; demotes do full teardown. Plans gate on `PlanDef.requires_profession`.
- **Skills (`skills.rs`):** `[u8; 8]` — Farming, Mining, Building, Trading, Combat, Crafting, Social, Medicine. Default 5; `gain_xp()` saturating.
- **Bucketing:** Agents sliced across 20 fixed `BucketSlot`s to spread CPU.
- **LOD:** `Detail / Aggregate / Dormant`. Dormant entities skip sim entirely. Focus-aware (see Game lifecycle below).
- **Memory & gossip (`memory.rs`):** Known locations + agent sightings, `u8` freshness decay, shared via `plan::plan_gossip_system`.
- **Needs:** 6 needs (hunger, sleep, shelter, safety, social, reproduction), `[0,255]`, decay over time, feed goal selection.

## Plan-design rules of thumb

- **Plan scoring is *viability*, not motivation.** Inside a need-driven goal the triggering need is constant across candidates, so weighting it again is circular noise. Plan weights answer "would this succeed and produce value?" via `SI_HAS_*` (inventory), `SI_VIS_*` (visibility), `SI_MEM_*` (memory), `SI_SKILL_*`, `SI_STORAGE_*`. Need-slots 0–5 are populated for inspector readout but not currently weighted (`#[allow(dead_code)]`). Bias is for "this is the cheap-and-immediate right answer."
- **Source vs good visibility (`STATE_DIM=42`):** Sources (slots 35–37: `SI_VIS_PLANT_FOOD`, `SI_VIS_TREE`, `SI_VIS_STONE_TILE`) feed Forage/Gather. Goods (slots 38–40: ground wood/stone/food) feed Scavenge. Never collapse the two — gather/farm gate on source-vis OR memory; `PF_SCAVENGE` plans gate on the matching ground slot; `PF_EXPLORE` plans invert (need both source-vis AND ground-vis to be zero).
- **Explore is per-resource and self-cancelling:** `ExploreForFood/Wood/Stone` (35/36/37) carry `memory_target_kind`. `explore_satisfaction_system` (after `vision_system`, before `plan_execution_system`) ends the plan with `PlanOutcome::Success` the moment matching memory is recorded — workers don't run out the 5000-tick timeout.
- **Farming is `Farm`-goal only.** Plans 1 (FarmFood), 4 (PlantFromStorage), 66 (PlantBerryFromStorage) never appear under Survive/GatherFood. Seed↔plant mapping is centralised in `PlantKind::seed_good()` + `PlantKind::ALL` (`plants.rs`); the Planter executor walks `PlantKind::ALL` and consumes one seed from carrier-or-inventory via `consume_one_good` (`production.rs`). Adding a new seed/plant pair = new `Good` variant + new `PlantKind` variant + arm in `seed_good()` + arm in `Good::is_seed()`. `StepPreconditions::requires_good` checks inventory + carrier so harvested seeds (which land in hands) satisfy plant-step gating.
- **Faction storage stocks in state vector (slots 29–32, 34, 41):** `SI_STORAGE_FOOD/WOOD/STONE/GRAIN_SEED/BERRY_SEED`, normalised against `STORAGE_SATURATE=20`. Refreshed by `compute_faction_storage_system` (Economy). Lets withdraw/haul plans score on actual stock; lets producers self-throttle when storage is full.

## Typed Task variants (`typed_task.rs`)

The agent's typed task lives on its own `ActionQueue` component — `aq.current: Task` is the canonical "task running now," `Task::Idle` is default, every Person spawn site bundles `ActionQueue::idle()`. The Phase 3 `PersonAI.task` field has been retired.

Phase 4b-i added a fixed-capacity `queued: [Task; 4]` prefetch ring behind `current` (private fields; access via `enqueue` / `pop_next` / `peek_next` / `queued_len` / `queued_is_empty` / `clear_queued`). Phase 4b-ii wired the ring into the live runtime:

1. **Producers** (`plan_execution_system`, the player-order handlers in `ui/orders.rs`, the `ReadItem` handler in `teaching.rs`) route through `aq.dispatch(task)` instead of writing `current` directly. `dispatch` enqueues then immediately promotes the head into `current` if `current == Idle` — single-task dispatches are behaviourally identical to the old direct write, but a future multi-task expansion can pre-stage a chain.
2. **Consumers** (every executor exit path in `gather.rs`, `dig.rs`, `corpse.rs`, `construction.rs`, `items.rs`, `production.rs`, `teaching.rs`, plus the MilitaryMove arrival in `military.rs`) call `aq.advance()` instead of `aq.current = Task::Idle`. `advance` pops the head of the queue, or sets `current = Idle` when the queue is empty.
3. **External preempts** (`apply_muster_hunters_system`, `faction_hunter_assignment_system` demote, `goal_dispatch_system` stale reset) call `aq.cancel()`, dropping both `current` and the prefetched queue so chained follow-ups can't outlive the plan that produced them.

Per-tick "pin" sites (the lecture / teach pin writes in `teaching.rs:369,531,579`) deliberately stay as direct `aq.current = X` writes — they're idempotent re-assertions, not fresh dispatches; routing them through `dispatch` would pile duplicates onto the queue every tick.

The ring is still empty in steady state because no producer pushes more than one task per dispatch yet, but the consumer wiring is already in place. Phase 5 method bodies that pre-decompose a chain (e.g. `WithdrawMaterial → WalkTo → DepositGoods`) will see them flow through `current` automatically.

Current variants (one entry per task family):

- `Idle` — between tasks.
- `WalkTo { tile, z, why: WalkReason }` — pure movement; `WalkReason` tags the dispatch context (today only `MilitaryMove`).
- `WithdrawGood { filter: WithdrawGoodFilter }` — `Specific(Good)` or `AnyEntertainment`.
- `WithdrawMaterial { good, qty }` — withdraw N of a specific material from storage.
- `Equip { slot: EquipmentSlot, good: Good }`.
- `Construct { blueprint: Entity }` — covers Construct + ConstructBed (the `task_id` discriminant survives for reward scaling).
- `Gather { tile }`, `Dig { tile }`, `Scavenge { target: Entity }`.
- `Read { tech }`, `Teach { tech }`, `HoldLecture { tech }`, `AttendLecture { tech }` — all in `teaching.rs`.
- `PickUpCorpse { corpse: Entity }` — first leg of hunt → carry → butcher.
- `Sleep { bed: Option<Entity> }` — produced by `SleepMethod` and dispatched by `htn_dispatch_system` (Phase 5a-ii). Bookkeeping only today: the Sleep "executor" is a state transition (`AiState::Sleeping`) rather than a per-tick task system, so no consumer reads the typed variant yet. `bed = Some(_)` means the agent is routing toward (or sleeping in) a claimed `HomeBed`; `bed = None` covers both the "head to faction home" branch and the solo "sleep in place" fallback. Cleared via the standard `aq.cancel()` stale-reset path in `goal_dispatch_system` when the goal flips off `Sleep`.
- `Eat` — parameterless, in-place. Produced by `EatFromInventoryMethod` and dispatched by `htn_eat_dispatch_system` (Phase 5b-ii). The Eat executor (`production::eat_task_system`) inspects inventory + hands itself (smallest-cover-then-largest selection across every edible the agent is carrying), so the variant carries no parameters. The legacy `EatFromInventory` plan (PlanId 25) was deleted in 5b-ii — the only path that produces `Task::Eat` for an in-place "eat what you're carrying" dispatch is now the HTN registry. The Eat-as-final-step path inside Forage / ScavengeFood / WithdrawAndEat plans still flows through `plan_execution_system` because those plans haven't migrated yet.
- `WithdrawFood { tile }` — pull one edible off the named faction-storage tile. Single-tick withdraw, no per-tick work accumulation. Produced by `WithdrawFromStorageMethod` and dispatched by `htn_acquire_food_dispatch_system` (Phase 5b-iii-ii) as the head of an `AcquireFood → WithdrawFood → Eat` chain. The legacy `WithdrawAndEat` plan (PlanId 9) was deleted in 5b-iii-ii — the only path that produces `TaskKind::WithdrawFood` is now the HTN registry. The chained `Eat` rides the `ActionQueue` prefetch ring; on `withdraw_food_task_system`'s exit `aq.advance()` promotes it into `current` and the executor primes the legacy channel (`task_id = TaskKind::Eat`, `state = Working`, `work_progress = 0`) so `eat_task_system` picks up next tick without re-entering dispatch. **This is the first multi-task chain in the runtime that exercises the prefetch ring at production time.**
- `HaulToBlueprint { blueprint }` — carry the agent's hand contents to the named `Blueprint` and drop them into its deposit slots. Produced by `WithdrawAndHaulToBlueprintMethod` and dispatched by `htn_acquire_good_dispatch_system` (Phase 5c-ii-b) as the trailing leg of an `AcquireGood → WithdrawMaterial → HaulToBlueprint` chain for `JobClaim::Haul` agents. The legacy `ClaimedHaul` plan (PlanId 33) was deleted in 5c-ii-b — the only path that produces this haul chain is now the HTN registry. The chain handoff lives in `withdraw_material_task_system`'s `finish_withdraw_material` exit helper: it `release_reservation`s the storage tile, `aq.advance()`s, and (when the new current is `HaulToBlueprint { blueprint }`) looks up the blueprint's tile via `bp_query` and routes the agent there with `TaskKind::HaulMaterials` / `target_entity = Some(blueprint)`. The haul leg piggybacks on `construction_system`'s existing hauler branch (`is_hauler = task == HaulMaterials as u16`) — that path already deposits-on-arrival and credits the `JobClaim::Haul` via `record_progress_filtered`, so no new per-tick task system is needed. **This is the second multi-task chain in production and the first whose trailing leg requires its own routing decision.**
- `Explore { kind: MemoryKind }` — walk to a random reachable tile near the agent's faction home, hoping to record a `MemoryKind::{kind}` sighting along the way. Produced by `ExploreForFoodMethod` (under `AcquireFood`) and `ExploreForMaterialMethod` (under `AcquireGood`) as the lone-task fallback expansion when no concrete target is in ctx — the HTN analogue of the legacy `ExploreForFood` / `ExploreForWood` / `ExploreForStone` plans (PlanId 35/36/37). The "executor" is the legacy `TaskKind::Explore` path: HTN dispatchers use the `pick_explore_tile` helper (mirrors `StepTarget::ExploreTile`'s 8-roll connectivity check) to pick a random reachable destination near the faction home, route via `assign_task_with_routing(... TaskKind::Explore, None ...)`, and `aq.dispatch(Task::Explore { kind })`. `vision_system` records matching memory en route. Termination: `goal_dispatch_system`'s catch-all at the bottom flips the agent's `task_id` back to `UNEMPLOYED` and calls `aq.advance()` once `state == Working` (i.e., on arrival), letting the next HTN dispatch tick re-evaluate with the populated ctx. **Live at 5c-ii-d-iv-ii**: PlanId 36/37 deleted, PlanId 35 retargeted to GatherFood-only (HTN owns the Survive case).
- `DepositToFactionStorage { good }` — carry the agent's hand contents to the nearest faction storage tile and drop them. Produced by `GatherFromKnownMethod` (Phase 5c-ii-c-i — scaffolding) as the trailing leg of an `AcquireGood → Gather → DepositToFactionStorage` chain. The "executor" is the legacy `TaskKind::DepositResource` path (`faction_dump_at_storage_system`), which is parameterless: it dumps everything in hands at the current `dest_tile`. The `good` payload here is recorded for chain-integrity inspection (the gather-exit handoff in 5c-ii-c-ii will be able to assert "this chain is depositing Wood, did the Gather step actually leave Wood in our hands?") and to keep the AcquireGood family symmetric with `WithdrawMaterial { good, .. }` and `HaulToBlueprint { blueprint }` — every variant in the family documents what the agent is *for*. **Scaffolding only at 5c-ii-c-i**: the variant is produced in unit tests but no dispatcher consumes the typed channel yet; the legacy `GatherWood` (PlanId 2) and `GatherStone` (PlanId 3) plans remain authoritative.
- `WalkReason::Gather` — `WalkTo` reason added at 5c-ii-c-i. No producer emits it yet; reserved for the eventual `GatherFromKnownMethod` dispatch path that may want a separate `WalkTo` step ahead of `Gather` (today the `Gather { tile }` leg is routed by `assign_task_with_routing`, mirroring the 5b-iii-ii / 5c-ii-b precedent of "no separate `WalkTo` task in the chain — the dispatcher routes the head and the executor's exit routes the tail"). Kept as a placeholder so the `WalkReason` enum has a Gather discriminant if a future dispatcher decides to dispatch routing as a discrete first task.

**Rules of the road:**

- Dispatcher (`plan_execution_system` + ad-hoc dispatch sites like `apply_*` order systems) populates `aq.current` alongside the legacy `task_id` field.
- Systems that mutate the typed task **must** include `&mut ActionQueue` in their query alongside `&mut PersonAI`. In `plan_execution_system` the component lives in `OptionalQuery` as `Option<&mut ActionQueue>` (the 15-tuple ceiling on `AgentQuery` is full); the executor `expect`s it, since every spawn bundle adds one — a missing component is a bundle-invariant violation.
- Executor reads its parameters from the typed variant (with optional fallback to a legacy field where one survives, e.g. `Construct` falls back to `target_entity` for the HaulMaterials path).
- Every executor exit path (success, abort, timeout, precondition fail) must clear `aq.current = Task::Idle` to prevent stale params leaking to the next task.
- External teardowns (`apply_muster_hunters_system`, `faction_hunter_assignment_system` demote, `goal_dispatch_system` stale reset) must also clear the typed task whenever they clear `task_id`.
- Inconsistent-state guard: `military_task_system`, `withdraw_good_task_system`, and `withdraw_material_task_system` fall back to `Idle` if `task_id == X` but `aq.current` is the wrong variant — defence in depth against a forgetful dispatcher. Companion regression tests live in `test_fixture::baseline_behaviour`.
- Test helper: `test_fixture::person_task(&app, entity) -> Task` reads `ActionQueue.current`. Use it instead of `person_ai(...).task` (which no longer exists).

## HTN domain (`htn.rs`)

Phase 5 introduces an HTN-style decomposition path that runs alongside the legacy plan registry. Four abstract tasks are dispatcher-live: `Sleep` (Phase 5a-ii), `Eat` (Phase 5b-ii), `AcquireFood` (Phase 5b-iii-ii), and `AcquireGood { good }` (Phase 5c-ii-b for `AgentGoal::Haul` + `JobClaim::Haul`; Phase 5c-ii-c-ii for `AgentGoal::GatherWood` / `GatherStone` + `MemoryKind::Wood`/`Stone`). The legacy `EatFromInventory` (PlanId 25), `WithdrawAndEat` (PlanId 9), `ClaimedHaul` (PlanId 33), `GatherWood` (PlanId 2), `GatherStone` (PlanId 3), and the legacy `goal_dispatch_system` Sleep arm have all been deleted; the HTN registry is authoritative for those five kinds plus Sleep.

- **`AbstractTask` / `AbstractTaskKind`** — the high-level goal a method decomposes. `Sleep`, `Eat`, and `AcquireFood` are dispatcher-live; `AcquireGood { good }` is the first parameterised variant — the good payload threads through to the methods so one method body serves every material (versus `AcquireFood`'s implicit-food-category shape).
- **`Method` trait** — `precondition` / `utility` / `expand(ctx) -> Vec<Task>` / `flags` / `tech_gate` / `profession_gate` / `name`. Methods read only the `PlannerCtx` fields they care about; the ctx is a borrowed snapshot built per-decision, not a long-lived component (post-Phase-6 shape). New ctx fields land on demand: 5b-i added `edible_count: u32` and `hunger: f32` for `EatFromInventoryMethod`; 5b-iii-i added `nearest_storage_tile: Option<(i32, i32)>` and `faction_food_stock: u32` for `WithdrawFromStorageMethod`; 5c-i added `material_storage_tile: Option<(i32, i32)>` and `material_stock_for_target: u32` for `WithdrawMaterialFromStorageMethod`. Sleep-only sites leave them at zero / `None`.
- **`MethodRegistry`** — `Resource`, `AHashMap<AbstractTaskKind, Vec<Box<dyn Method>>>`. Built once in `SimulationPlugin::build` via `register_builtin_methods`. Read-only after init so dispatch systems can borrow immutably in parallel.
- **`SleepMethod`** — registered under `AbstractTaskKind::Sleep`. Mirrors the three-branch Sleep logic the legacy `goal_dispatch_system` arm encoded: own-bed (live claim) → faction-home → in-place. All three branches expand to a single `Task::Sleep { bed }` because routing/state-transition is downstream of the typed task.
- **`EatFromInventoryMethod`** — registered under `AbstractTaskKind::Eat`. Replaces the deleted `EatFromInventory` plan (PlanId 25). Precondition is `edible_count > 0 && hunger >= EAT_TRIGGER_HUNGER`; expansion is a single `Task::Eat`. Consumed by `htn_eat_dispatch_system` (Phase 5b-ii).
- **`WithdrawFromStorageMethod`** — registered under `AbstractTaskKind::AcquireFood`. Precondition is `faction_food_stock > 0 && nearest_storage_tile.is_some() && hunger >= EAT_TRIGGER_HUNGER`; expansion is a two-task chain `[Task::WithdrawFood { tile }, Task::Eat]`, which is the first method body in the registry that produces more than one task. Consumed by `htn_acquire_food_dispatch_system` (Phase 5b-iii-ii). Replaced the deleted `WithdrawAndEat` plan (PlanId 9). Sibling `ScavengeFoodFromGroundMethod` arrived in 5c-ii-d-iii-i (utility 1.5 outranks this method's 1.0 when both are applicable).
- **`ScavengeFoodFromGroundMethod`** — registered under `AbstractTaskKind::AcquireFood`. Precondition gates on `scavenge_target_entity.is_some() && scavenge_target_tile.is_some() && hunger >= EAT_TRIGGER_HUNGER` (paired-field requirement matching `ScavengeFromGroundMethod`, plus the AcquireFood hunger gate symmetric with `WithdrawFromStorageMethod`); rejects mis-routed abstract-task variants. Utility `1.5` (bias-on-visibility, parity with `ScavengeFromGroundMethod`'s 1.5 under AcquireGood). Expansion is `[Task::Scavenge { target }, Task::Eat]` — the **first AcquireFood chain that doesn't end in storage withdraw** (the legacy `ScavengeFood` plan's deposit-then-re-walk-back-to-withdraw pattern was wasted motion when the agent is already hungry; under HTN the agent picks up the food and eats it on the spot). Reuses the existing `scavenge_target_entity` / `scavenge_target_tile` ctx fields populated by 5c-ii-d-ii-a's gather/scavenge dispatcher branch — same field shape, different abstract task. **Live at 5c-ii-d-iii-ii for the Survive case:** `htn_acquire_food_dispatch_system` scans `SpatialIndex` within `VIEW_RADIUS=15` for visible loose edible `GroundItem`s (filtering on `is_edible()`, excluding faction storage tiles), populates `scavenge_target_*` per decision, and dispatches the chain. When both this method's ctx fields and `WithdrawFromStorageMethod`'s `nearest_storage_tile` + `faction_food_stock` are populated, the registry argmax picks scavenge (1.5 > 1.0). The chain handoff lives in `item_pickup_system`'s `finish_scavenge` exit helper: when the prefetch ring promotes `Task::Eat`, the helper primes the legacy `task_id = TaskKind::Eat` channel directly so `eat_task_system` picks up next tick (mirrors `production::finish_withdraw_food`'s Eat handoff for the `[WithdrawFood, Eat]` chain). The dispatcher passes `target_entity = Some(target)` to `assign_task_with_routing` so `goal_update_system`'s Scavenge target validation (`goals.rs:286-293`) doesn't flag the task invalid — the AcquireGood scavenge branch can pass `None` because GatherWood/Stone goals always carry a `JobClaim::Stockpile` that bypasses `goal_update_system` (`goals.rs:237`), but the Survive-driven AcquireFood path has no such bypass. **Legacy `ScavengeFood` plan (PlanId 6) retargeted from `SURVIVE_AND_GATHER_FOOD_GOALS` to `GATHER_FOOD_GOALS`** — the chief-driven storage-fill path still uses the legacy `[CollectFood, DepositGoods]` plan because under GatherFood (no hunger gate) the agent should deposit, not eat in place. Migrating GatherFood needs a `ScavengeFoodForStorageMethod` whose expansion is `[Scavenge, DepositToFactionStorage { food_good }]` plus a `scavenge_food_good` ctx field threading the picked-up good through; deferred until the wiring substrate stabilises.
- **`WithdrawMaterialFromStorageMethod`** — registered under `AbstractTaskKind::AcquireGood`. Precondition gates on `material_stock_for_target > 0 && material_storage_tile.is_some()` and rejects mis-routed abstract-task variants defensively. Expansion is a single `Task::WithdrawMaterial { good, qty: 1 }` — the material analogue of `WithdrawFromStorageMethod` but without an automatic terminal step (consuming the material is its own decomposition; chaining belongs there). The `qty: 1` unit-acquisition contract keeps the method orthogonal to claim-based bulk patterns; `FulfillClaim` (post-5c) is where larger qty wiring lands. Utility `1.0`. No dispatcher consumes the bare-withdraw path today (5c-ii-c will wire it under stale gather goals when storage is well-stocked); 5c-ii-b's haul dispatcher always picks `WithdrawAndHaulToBlueprintMethod` because `claimed_blueprint.is_some()` is in the haul method's precondition but not in this one's, and the haul utility is `2.0`.
- **`WithdrawAndHaulToBlueprintMethod`** — registered under `AbstractTaskKind::AcquireGood`. Precondition gates on `material_stock_for_target > 0 && material_storage_tile.is_some() && claimed_blueprint.is_some()` and rejects mis-routed abstract-task variants. Utility `2.0` (strictly higher than the bare-withdraw method's `1.0`). `MF_UNINTERRUPTIBLE` flag mirrors the legacy `PF_UNINTERRUPTIBLE`. Expansion is a two-task chain `[Task::WithdrawMaterial { good, qty: 1 }, Task::HaulToBlueprint { blueprint }]` — the second multi-task chain in the registry after `WithdrawFromStorageMethod`'s `[WithdrawFood, Eat]`, and the first whose trailing leg requires its own routing decision. Replaced the deleted `ClaimedHaul` plan (PlanId 33). Consumed by `htn_acquire_good_dispatch_system` (Phase 5c-ii-b).
- **`GatherFromKnownMethod`** — registered under `AbstractTaskKind::AcquireGood`. Precondition gates on `gather_target_tile.is_some()` (a known harvest tile from `AgentMemory::best_for(MemoryKind::Wood|Stone)` — tree for Wood, stone tile for Stone) and rejects mis-routed abstract-task variants defensively. Utility `1.0` (parity with the bare-withdraw method; the haul method's `2.0` keeps the hauler context exclusive via its `claimed_blueprint` precondition). Expansion is a two-task chain `[Task::Gather { tile }, Task::DepositToFactionStorage { good }]` — the third multi-task chain in the registry. Replaced the deleted `GatherWood` (PlanId 2) and `GatherStone` (PlanId 3) plans (each a 2-step `[Gather, DepositGoods]`). Consumed by `htn_acquire_good_dispatch_system`'s gather branch (Phase 5c-ii-c-ii), which maps `AgentGoal::GatherWood`/`GatherStone` to `(Good, MemoryKind)` pairs and reads `AgentMemory::best_for` to seed `gather_target_tile`. The chain handoff lives in `gather.rs`'s `finish_gather` exit helper — every `gather_system` exit path calls it instead of the legacy `aq.advance() + Idle reset`; when the prefetch ring promotes `Task::DepositToFactionStorage`, `finish_gather` looks up the nearest faction storage tile via `StorageTileMap::nearest_for_faction` and routes the agent there with `TaskKind::DepositResource`. From there `drop_items_at_destination_system` is the executor — it dumps everything in hands at `dest_tile`, credits any `JobClaim::Stockpile` companion, and (Phase 5c-ii-c-ii) calls `aq.advance()` so the typed channel returns to Idle.
- **`ScavengeFromGroundMethod`** — registered under `AbstractTaskKind::AcquireGood`. Precondition gates on `scavenge_target_entity.is_some() && scavenge_target_tile.is_some()` (a known loose `GroundItem` of the target good — fallen wood, surface stone, dropped fruit) and rejects mis-routed abstract-task variants defensively. **Utility `1.5`** (Phase 5c-ii-d-ii-a bias-on-visibility): a loose item is strictly cheaper to acquire than a tree-to-chop, so a worker who can see one scavenges it instead. Sits between `GatherFromKnownMethod`'s `1.0` and `WithdrawAndHaulToBlueprintMethod`'s `2.0`. - **Distance-weighted utility (Phase 5c-ii-d-v).** Methods whose ctx carries a target tile (`nearest_storage_tile`, `material_storage_tile`, `gather_target_tile`, `scavenge_target_tile`) subtract `chebyshev_dist(ctx.tile, target) * DIST_DISCOUNT_PER_TILE` from their base utility, capped at `MAX_DIST_PENALTY`. Within a single method, closer targets outscore farther ones, so the dispatcher's "nearest applicable" semantics survive method-level argmax instead of being hand-coded into each ctx-population site.

**`htn_dispatch_system`** (ParallelB, ordered `.after(tasks::goal_dispatch_system)`) consumes the registry. For each non-Drafted, non-PlayerOrder agent with `AgentGoal::Sleep`:

1. Short-circuit the in-progress states — already `Sleeping`, just-arrived `Working` on the Sleep tile (flip to `Sleeping`), or in flight (`Seeking`/`Routing`) toward one.
2. Build a `PlannerCtx { tile, faction_id, faction_home, home_bed, home_bed_tile }` from the agent's `Transform`, `FactionMember`, `HomeBed`, the bed's `Transform` lookup (via `bed_query`), and `FactionRegistry::home_tile`. A bed entity that's despawned or unloaded silently degrades `home_bed_tile` to `None`, which the SleepMethod treats as "no live claim."
3. Argmax over `methods_for(AbstractTaskKind::Sleep)` filtered by `precondition`. f32 has no total order, so `partial_cmp(...).unwrap_or(Equal)` ties break on declaration order.
4. Match on the expansion's first `Task::Sleep { bed }`:
   - `Some(bed_entity)` + live `home_bed_tile` → `assign_task_with_routing(... TaskKind::Sleep, Some(bed_entity) ...)` and `aq.dispatch(Task::Sleep { bed: Some(_) })`.
   - `None` + faction home outside the 5-tile disc → route to home, `aq.dispatch(Task::Sleep { bed: None })`.
   - `None` + at-home / solo / no-home → set `AiState::Sleeping`, `task_id = TaskKind::Sleep`, `aq.dispatch(Task::Sleep { bed: None })`.
5. Push any remaining tasks (none today; SleepMethod returns a single-element vec) onto the prefetch ring via `aq.enqueue`. This path exists so multi-step Sleep expansions ("drink water → sleep") flow without code change later.

`goal_dispatch_system` retains the no-plan stale-reset block (so a goal flip off Sleep still triggers `aq.cancel()` + `task_id = UNEMPLOYED`, and the `AgentGoal::Survive if task_id == TaskKind::Eat` arm preserves an in-flight Eat task across plan boundaries) and the Explore-cleanup catch-all. It no longer matches on `AgentGoal::Sleep`. The pathfinder/`FactionRegistry`/`HomeBed`/`Bed`/`EconomicAgent` resources moved with the Sleep arm — `goal_dispatch_system` now only borrows `&mut PersonAI`, `&mut ActionQueue`, `&AgentGoal`, `&LodLevel`, `Option<&ActivePlan>`.

**`htn_eat_dispatch_system`** (ParallelB, ordered `.after(htn_dispatch_system)`) handles the in-place Eat case. For each non-Drafted, non-PlayerOrder agent with `AgentGoal::Survive`:

1. Skip if the agent has an `ActivePlan` (let Forage / ScavengeFood / WithdrawAndEat run their own Eat-as-final-step) or is already on a non-Idle task. Same gate `plan_execution_system` uses to start a fresh plan.
2. Build a `PlannerCtx { tile, faction_id, edible_count, hunger, .. }` — the Sleep-only fields stay at their `None` defaults; `edible_count` comes from `total_edible(agent, carrier)` (inventory + both hands), `hunger` from `Needs.hunger`.
3. Cheap pre-filter: skip the registry walk when `edible_count == 0` or `hunger < EAT_TRIGGER_HUNGER` (matches the method's own precondition).
4. Argmax over `methods_for(AbstractTaskKind::Eat)` filtered by `precondition`. Today only `EatFromInventoryMethod`.
5. On `Task::Eat`: prime the legacy channel (`state = Working`, `task_id = Eat`, `work_progress = 0`) so `eat_task_system` can start accumulating ticks, and `aq.dispatch(Task::Eat)` mirrors the typed channel. Any further tasks land on the prefetch ring.

The split into per-goal dispatch systems (Sleep, Eat, AcquireFood) is pragmatic: each has its own component-set requirements (Eat needs `EconomicAgent`/`Carrier`/`Needs`/`Option<&ActivePlan>`; AcquireFood adds the four pathfinder resources + `StorageTileMap` + `FactionRegistry`), and all three serialise on `&mut PersonAI` / `&mut ActionQueue` anyway, so the split costs no parallelism. Future goals will follow the same per-goal-system pattern rather than ballooning a single dispatcher's query.

**`htn_acquire_food_dispatch_system`** (ParallelB, ordered `.after(htn_eat_dispatch_system)`) handles the "no food on hand, faction storage has food, agent is hungry" branch of `AgentGoal::Survive`. For each non-Drafted, non-PlayerOrder Survive agent without an `ActivePlan` and an idle task slot:

1. Pre-filter: skip SOLO agents (no faction storage), skip agents with `total_edible(...) > 0` (defer to `htn_eat_dispatch_system` for in-place eat), and skip when `hunger < EAT_TRIGGER_HUNGER`. The food-on-hand split lets `EatFromInventoryMethod` and `WithdrawFromStorageMethod` keep symmetric preconditions without a hand-tuned tie-breaker.
2. Build a `PlannerCtx { tile, faction_id, faction_home, hunger, nearest_storage_tile, faction_food_stock, .. }` — `nearest_storage_tile` from `StorageTileMap::nearest_for_faction`, `faction_food_stock` from `FactionRegistry::food_stock` floored to u32. Sleep / Eat ctx fields stay at their zero / `None` defaults.
3. Argmax over `methods_for(AbstractTaskKind::AcquireFood)` filtered by `precondition`. Today only `WithdrawFromStorageMethod`.
4. On `Task::WithdrawFood { tile }`: `assign_task_with_routing(...)` to the storage tile (Bevy's `dispatched` flag is checked — if routing fails, e.g. no reachable adjacent standable, the chain is abandoned and the next tick re-evaluates), then `aq.dispatch(Task::WithdrawFood { tile })`.
5. Push the trailing `Task::Eat` (and any future tail) onto the prefetch ring via `aq.enqueue`.

The withdraw → eat handoff lives in `withdraw_food_task_system`'s `finish_withdraw_food` exit helper: it calls `aq.advance()` (promoting the queued `Task::Eat` into `aq.current`), and when the new current is `Task::Eat` it primes the legacy channel (`state = Working`, `task_id = Eat`, `work_progress = 0`) so `eat_task_system` picks up immediately on the next tick. Other (non-Eat / Idle) follow-ups fall back to a clean `(Idle, UNEMPLOYED)` slot; an unexpected variant in the chain triggers `aq.cancel()` to drop the entire ring rather than strand the agent. **This is the first multi-task chain that the runtime actually exercises**: until now every `expand()` returned a single-element vec and `aq.enqueue` was a no-op in production.

**`htn_acquire_good_dispatch_system`** (ParallelB, ordered `.after(htn_acquire_food_dispatch_system)`) handles `AgentGoal::Haul` for agents with a live `JobClaim::Haul` + `ClaimTarget` companion. For each non-Drafted, non-PlayerOrder Haul-goal agent without an `ActivePlan` and an idle task slot:

1. Read `target.good` and `target.blueprint` from the `ClaimTarget`. Skip if either is `None` (defensive against partially-populated targets).
2. Cheap-reject when the faction has no stock of the target good via `FactionData.storage.stock_of(good)`.
3. Walk `StorageTileMap.by_faction[fid]` to find the nearest tile with effective stock (after `StorageReservations`) of the target good. `StorageTileMap::nearest_for_faction` ignores good-specificity, so the dispatcher does an explicit per-tile scan via `SpatialIndex` + `Query<&GroundItem>`.
4. Build a `PlannerCtx { material_storage_tile, material_stock_for_target, claimed_blueprint, .. }` and argmax over `methods_for(AcquireGood)`. Today: `WithdrawAndHaulToBlueprintMethod` (utility 2.0, gated on `claimed_blueprint.is_some()`) wins over the bare `WithdrawMaterialFromStorageMethod` (utility 1.0).
5. Read the expansion's two-task chain `[WithdrawMaterial, HaulToBlueprint]`, route the head via `assign_task_with_routing(... TaskKind::WithdrawMaterial, None ...)` to the storage tile, add a `StorageReservations` entry (mirroring `plan_execution_system`'s site at `plan/mod.rs:2724`), `aq.dispatch` the typed task, and `aq.enqueue` the trailing `Task::HaulToBlueprint { blueprint }`.

The withdraw → haul handoff lives in `withdraw_material_task_system`'s `finish_withdraw_material` exit helper (analogous to `finish_withdraw_food`): it `release_reservation`s the storage tile, `aq.advance()`s, and (when the new current is `Task::HaulToBlueprint { blueprint }`) looks up the blueprint's tile via `bp_query` and routes the agent there with `TaskKind::HaulMaterials, Some(blueprint)`. The haul leg piggybacks on `construction_system`'s existing hauler branch — no new per-tick task system was needed. `goal_dispatch_system`'s no-plan stale-reset gate was extended for `AgentGoal::Haul + (WithdrawMaterial | HaulMaterials)` so the multi-tick haul chain isn't reset every tick by the absence of an `ActivePlan`. The legacy `ClaimedHaul` plan (PlanId 33) was deleted in 5c-ii-b.

**Phase 5c-ii-c-ii — Gather branch.** The same dispatcher also handles `AgentGoal::GatherWood` / `AgentGoal::GatherStone`. For each non-Drafted, non-PlayerOrder gather-goal agent without an `ActivePlan` and an idle task slot:

1. Map the goal to a `(Good, MemoryKind)` pair (`GatherWood → (Wood, MemoryKind::Wood)`, `GatherStone → (Stone, MemoryKind::Stone)`).
2. Read `AgentMemory::best_for(memory_kind)` for the gather target tile. Skip when memory is empty — the legacy `ExploreForWood`/`ExploreForStone` plans (PlanId 36/37) are still authoritative for the no-knowledge case via `plan_execution_system`.
3. Build a `PlannerCtx { gather_target_tile: Some(tile), .. }` (leaving `material_storage_tile`, `claimed_blueprint`, etc. at `None` so only `GatherFromKnownMethod`'s precondition fires).
4. Argmax over `methods_for(AcquireGood)`; the gather method (utility 1.0) is the only applicable method on this ctx shape.
5. Read the expansion's two-task chain `[Gather { tile }, DepositToFactionStorage { good }]`, route the head via `assign_task_with_routing(... TaskKind::Gather, None, ...)` to the gather tile, `aq.dispatch` it, and `aq.enqueue` the trailing `DepositToFactionStorage`.

The gather → deposit handoff lives in `gather.rs`'s `finish_gather` exit helper. Every `gather_system` exit path (5 sites — despawned plant, immature plant, completed stone tile, invalid target, hands at haul cap) calls it instead of the legacy inline `aq.advance() + Idle reset`. When the prefetch ring promotes `Task::DepositToFactionStorage`, `finish_gather` looks up the nearest faction storage tile via `StorageTileMap::nearest_for_faction` and routes the agent there with `TaskKind::DepositResource`. On routing failure (no storage / unreachable / SOLO defensive) the chain is dropped via `aq.cancel()` — the agent stays Idle with full hands and re-evaluates next tick. The deposit leg runs on `drop_items_at_destination_system` which already handles `task_id == TaskKind::DepositResource` end-to-end; 5c-ii-c-ii added `&mut ActionQueue` to its query and an `aq.advance()` at the exit so the typed channel returns to Idle (previously the deposit step was always plan-driven, with no `Task::*` peer in flight). The 4 routing resources (`StorageTileMap` + `ChunkGraph` + `ChunkRouter` + `ChunkConnectivity`) are bundled into a `GatherRoutingResources` `SystemParam` to keep `gather_system` under Bevy's 16-tuple `IntoSystem` ceiling. `goal_dispatch_system`'s no-plan stale-reset gate was extended for `AgentGoal::GatherWood`/`GatherStone + (Gather | DepositResource)` so the multi-tick chain isn't reset every tick by the absence of an `ActivePlan`. The legacy `GatherWood` (PlanId 2) and `GatherStone` (PlanId 3) plans were deleted in 5c-ii-c-ii. The `Explore` (36/37) and `Scavenge` (38/39) plans for these goals are still plan-driven — they handle the `gather_target_tile == None` case (no memory) and the `vis_ground_wood/stone > 0` case (loose items on the ground) respectively.

Adding a new method = new `AbstractTask` variant (if needed) + new `Method` impl + register call in `register_builtin_methods`. Method tests live in `htn::tests` and exercise the trait directly without an `App` — that pattern keeps registry-shape regressions cheap to pin. Adding a new abstract-task *kind* also requires a per-goal dispatch system (or a new branch in an existing one) that builds the right `PlannerCtx` shape and routes the expansion's head into the legacy `task_id` channel.

**`Carrying(Entity)` component (in `corpse.rs`):** Marker for "this person is carrying corpse E." Inserted at pickup arrival, removed at butcher / decay / rescue / muster / hunter-demote. `corpse_follow_system` keys on `(&Carrying, &Transform)`.

## Faction systems

- **Factions (`faction.rs`):** `FactionTechs` bitset is a derived projection of the chief's `PersonKnowledge.aware`; rebuilt every Economy tick by `sync_faction_techs_from_chief_system`. `FactionCenter`, bonding, storage rollup, raids. `SOLO=0` = ungrouped. `StorageTileMap` indexes storage tiles per faction. `FactionStorage::totals` refreshed every Economy tick.
- **Construction (`construction.rs`):** `BlueprintMap`, `WallMap`, `BedMap`. `faction_blueprint_system` decides what to build; `construction_system` consumes resources and finalizes tiles/entities. `generate_candidates` paces hearths by era (Paleo/Meso `(members+5)/6` gated on crescent saturation + bed deficit; Neolithic `(members+7)/8` gated on each hearth having ≥8 beds in 2..6 crescent ring; Chalcolithic+ single civic hearth). Defensive walls (`PalisadeSegment`) are Chalcolithic+ only. `best_wall_material` ladder: Palisade < WattleDaub (`PERM_SETTLEMENT`) < Mudbrick (`FIRED_POTTERY`) < Stone (`COPPER_WORKING`).
- **CraftOrders + jobs:** `CraftOrder` carries `spawn_tick`; `faction_craft_order_system` despawns orders > `CRAFT_ORDER_TIMEOUT_TICKS=600`. `chief_job_posting_system` (`jobs.rs`) emits one `JobKind::Craft` per faction, picking the recipe with largest `demand-supply` (subject to tech + station availability — Workbench within 12 tiles of `home_tile`, Loom for loom recipes; Loom recipes pass `bench: None` because `job_claim_release_system` only validates Workbench). `resource_demand_system` (`faction.rs`) populates demand for crafted outputs as a fraction of `member_count`.
- **WithdrawMaterial intent:** Storage tiles hold many goods, so the *resolver* picks. All `TaskKind::WithdrawMaterial` steps share `StepTarget::WithdrawForFactionNeed { need, selector }` (`need`: Blueprint/CraftOrder/HaulClaim; `selector`: MostDeficient/Specific). One resolver, parameterised. Per-good deliver-plans removed; plan 15 `DeliverFromStorageToCraftOrder` covers everything.
- **StorageReservations resource:** `(tile, good) → reserved_qty`, mutex-wrapped (parallel resolver). Each successful `WithdrawMaterial` dispatch increments and stashes `(reserved_tile, reserved_good, reserved_qty)` on `PersonAI`; `release_reservation` decrements on every teardown path. Resolver subtracts reservations from raw `GroundItem.qty` so two agents can't commit to the same one-unit stack.
- **Activity log (`ui/activity_log.rs`):** Bottom-right egui panel; `ActivityLogEvent { tick, actor, faction_id, kind }` with kinds `Constructed`, `Crafted`, `TechDiscovered`, `RegionSettled`. Filtered to player faction; capped at 16 entries.

## Knowledge & technology

- **Per-person knowledge (`knowledge.rs`):** `PersonKnowledge` carries two parallel `u64` bitsets — `aware` (heard of, free, gossiped during Socialize) and `learned` (mastered, costs `complexity()` points, subset of `aware`) plus `learned_at: [u32; 64]` for LRU eviction. `complexity(tech)` is era-based (Paleolithic 1 → Bronze Age 5; Cuneiform/Lunar/City-State bumped to 6). Capacity = `intelligence × 2` (`stats::knowledge_capacity`). Adding a Learned tech beyond capacity demotes the LRU entry back to Aware-only — no per-tick decay. Founders spawn with all-Paleolithic Aware+Learned via `PersonKnowledge::paleolithic_seed`; newborns inherit the OR of both parents' Aware ∪ Learned (free awareness only — no inherited mastery). Progressive study lives on the same component: `study_progress: AHashMap<TechId, u32>`. `add_study_progress(tech, amount, capacity, now)` grants awareness on the first tick, accumulates ticks, and on `study_threshold(tech) = complexity * STUDY_TICKS_PER_COMPLEXITY (3600)` runs `try_learn` (LRU eviction may apply) then clears the entry.
- **Discovery (per-action):** `discovery_system` consumes `DiscoveryActionEvent`s emitted by `gather_system`, `production_system`, `combat_system`, `social_fill_system`. Eligible techs are those whose prerequisites the actor has personally Learned ("next-level adjacent"). Roll `base × (1 + INT_mod × 0.1) × (1 + skill_xp / 1000)`, capped at 0.5; on success the tech lands directly in Learned.
- **Awareness gossip:** `plan_gossip_system` extension — adjacent socializing agents OR each other's `aware` bitsets every Economy tick. Free, fast.
- **Passive teaching:** `tech_teaching_system` (Economy, after gossip) scans pairs of socializing agents within 3 tiles, finds techs the teacher has Learned that the student is Aware of but not Learned, and rolls a low chance (`0.004 × INT_scale`). On success the highest-complexity teachable tech transfers to the student's Learned set (subject to capacity).
- **Directed teaching (`teaching.rs`):** Three pathways feed `study_progress`. (1) **Read** — `Task::Read{tech}`. `read_task_system` adds `+1 × INT_scale` per tick. Player triggers via inspector "Read <tech>" button on Aware-only rows that have a matching tablet/book in inventory; `apply_player_knowledge_orders_system` consumes `PlayerOrderKind::ReadItem` and inserts `Drafted`. (2) **Lecture** — inspector "Lecture" button on a Learned-tech row writes `LectureRequest(Some((lecturer, tech)))`. `apply_lecture_request_system` (Economy) drafts up to 8 same-faction adults within 6 tiles, drops their `ActivePlan`, and inserts `Lecturing { ends_tick=now+600, tech, anchor }` + `Attending`. `lecture_tick_system` (Sequential, after movement) credits `+2 × INT_scale` per tick to each attending student and releases on threshold/timeout/distance. (3) **1-on-1 teach** — right-click a friendly person → "Teach". `PlayerOrderKind::Teach(student)` routes the teacher adjacent; `apply_teach_order_system` then inserts `TeachingPair`/`BeingTaught` with `ends_tick=now+120`. `teach_task_system` credits `+3 × INT_scale` per adjacent tick and emits `ActivityEntryKind::Taught` on success. All three pathways carry `Drafted` while busy.
- **Faction-tech derivation:** `FactionData.techs` is a *cache* — `sync_faction_techs_from_chief_system` (Economy) projects the chief's `aware` bitset onto it every tick. Most read sites ask "can the faction direct this?" → chief-aware suffices. Per-person execution gates use `has_learned` instead:
  - Plan candidate filter (`plan/mod.rs:plan_execution_system`).
  - `job_claim_system` (`jobs.rs`) gating `JobKind::Craft` claims on `recipe.tech_gate`.
  - `faction_hunter_assignment_system` only promotes candidates with Learned `HUNTING_SPEAR`.
  - `mount_check_system` requires per-person Learned `HORSEBACK_RIDING`.
- **Tablets and books (`Item.tech_payload: Option<TechId>`):** `Good::ClayTablet` (id 20, `OneHand`, 1500 g) and `Good::Book` (id 21, `Small`, 600 g). Crafted via recipes 10/11 in `CRAFT_RECIPES` (gated on `CUNEIFORM_WRITING`, Workbench station). `CraftOrder.tech_payload` and `JobProgress::Crafting.tech_payload` thread the chosen TechId through to `craft_order_system`'s completion stamp where it lands on the produced `Item`. Reading does not consume them. Item equality partitions tablets-of-tech-A from tablets-of-tech-B in storage automatically.
- **Tablet posting (`chief_tablet_posting_system`, `jobs.rs`):** Runs every `CHIEF_TABLET_POSTING_INTERVAL=3600` ticks per faction. Reads chief's `learned`, tallies adult-awareness across all faction members, picks the highest-complexity bit the chief Learned that <50% of adults are Aware of (skipping techs already covered by a live tablet posting), and posts a `JobKind::Craft` with `recipe=10` and `tech_payload=Some(chosen)`. Player override: `PlayerCraftRequest` resource — written by inspector "Encode" button or `PlayerOrderKind::EncodeTablet`; consumed every tick (regardless of cadence) and posted with `JobSource::Player`/priority 180.
- **Tech tree (`technology.rs`):** 43 techs, 5 eras, prereq DAG, per-tech `triggers: &[TechTrigger]` mapping `ActivityKind → per_unit_chance`. `TechBonus` adds yield/storage/combat bonuses; bonuses still aggregate via `faction.techs` (chief-aware).

## Game lifecycle and regions

- **`GameState`:** `SpawnSelect` (default) | `Playing`. Globe gen at `WorldPlugin::build`. Spawn-select UI in `SpawnSelect`; spawn systems on `OnEnter(Playing)`. Update-stage systems gated `in_state(Playing)`; FixedUpdate sim systems are not gated (queries empty until entities spawn).
- **`PendingSpawn(Option<(i32, i32)>)`:** Player's chosen mega-chunk; `None` = globe centre (sandbox).
- **Sandbox bypass:** `SandboxPlugin` flips state to `Playing` at Startup, skipping spawn-select.
- **`SettledRegions` (`region.rs`):** `AHashMap<RegionId, SettledRegion>` + `by_megachunk` reverse index. `SettledRegion { megachunk, founding_tick, name, camera_bookmark, player_owned }`. `settle()` idempotent.
- **`MegaChunkCoord`:** `from_chunk/from_tile/center_tile/chunk_range`. `MEGACHUNK_SIZE_CHUNKS=16`, independent of climate cells.
- **Multi-focus chunk streaming (`SimulationFocus`):** `Vec<FocusPoint>` rebuilt each tick from camera + every settled region centre. Chunk DATA loads for any focus disc (camera `LOAD_RADIUS=12`, region `REGION_LOAD_RADIUS=6`); SPRITES + plants + loose rocks only inside the camera focus.
- **Focus-aware LOD (`update_lod_levels_system`):** Camera distance produces base LOD; entities within 8 chunks of any non-camera focus are promoted from Dormant to Aggregate so off-screen agents keep ticking.
- **Edge-walk expansion (`region::detect_edge_crossing_system`, Economy):** Each tick checks every player-faction Person's mega-chunk; if unsettled, calls `settle()`, names "Outpost N", bookmarks position, emits `ActivityLogEvent::RegionSettled`.
