# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
cargo run                  # Run the game (normal mode)
cargo run -- --sandbox     # Run sandbox mode (small 5√ó5 chunk map, one of every entity type)
cargo build --release      # Optimized build (thin LTO, single codegen unit)
cargo check                # Fast type check without compiling
cargo test                 # Run tests
```

## Architecture

CivGame is a Dwarf Fortress-style civilization simulation built on **Bevy 0.15** (ECS). Six plugins compose the game:

| Plugin | Directory | Responsibility |
|--------|-----------|----------------|
| `WorldPlugin` | `src/world/` | Procedural terrain (Perlin noise, Z-levels -16..+15), chunk streaming (32√ó32 tiles, 64√ó64 world), biomes, calendar/seasons, `SpatialIndex` |
| `SimulationPlugin` | `src/simulation/` | Agent AI, needs, combat, reproduction, factions, technology, memory/gossip, plant growth, raids |
| `EconomyPlugin` | `src/economy/` | Markets, goods, prices, transactions |
| `PathfindingPlugin` | `src/pathfinding/` | Flow-field caching, chunk-level navigation graph |
| `RenderingPlugin` | `src/rendering/` | 16√ó16 pixel art tiles, camera (WASD/scroll), chunk streaming visuals, entity sprites |
| `UiPlugin` | `src/ui/` | All UI via `bevy_egui`: inspector, HUD, economy panel, world map, right-click context menu, activity log |

### Simulation scheduling (`SimulationSet`)

Systems must be assigned to the correct set ‚Äî misplacement causes subtle bugs:

```
ParallelA ‚Üí ParallelB ‚Üí Sequential ‚Üí Economy
```

- **ParallelA** ‚Äî read-heavy systems (needs ticks, mood, LOD, goal updates, animal sensing)
- **ParallelB** ‚Äî `goal_dispatch_system` (Sleep-only fallback ‚Äî every other goal is plan-driven; survives only because the bed/camp fallback chain wasn't worth porting)
- **Sequential** ‚Äî mutating systems with tight ordering: `gather` ‚Üí `dig` / `construction` ‚Üí `movement` ‚Üí `combat` ‚Üí `production` ‚Üí `plan_execution`
- **Economy** ‚Äî post-simulation pass: gossip, faction storage rollup, reproduction, raids, technology, market price updates

### Key simulation systems

- **Person agents:** Goal-driven via `PersonAI` component.
  - **Goals (`goals.rs`):** High-level objectives (e.g., `Survive`, `Gather`, `Defend`) driven by Needs and Faction state.
  - **Plans (`plan/`):** Multi-step sequences (e.g., "Find Food" ‚Üí "Harvest") scored by a static linear function ‚Äî `dot(state, plan.state_weights) + plan.bias` plus manual bonuses (persistence, ally influence, dist-weighted memory penalty). `PlanScoringMethod::Weighted` for normal agents; `Random` for Œµ-greedy exploration. Built from a `StepRegistry` + `PlanRegistry` populated at startup. Module split: `plan/mod.rs` holds types, `ActivePlan`/`PlanHistory`/`KnownPlans`, `resolve_target`, and `plan_execution_system`; `plan/registry.rs` holds the step + plan data tables; `plan/state.rs` holds `build_state_vec` + `count_visible_*`. Each `PlanDef` carries `flags: PlanFlags` (`PF_EXPLORE`, `PF_SCAVENGE`, `PF_TARGETS_FOOD/WOOD/STONE`, `PF_DROP_FOOD_ON_TIMEOUT`, `PF_UNINTERRUPTIBLE`) ‚Äî the candidate filter reads flags rather than per-plan-id matches. `PF_UNINTERRUPTIBLE` (set on plans 7 `BuildBlueprint`, 13-16 craft-order pipeline, 29 `HaulFromStorageAndBuild`, 33 `ClaimedHaul`, 34 `ClaimedBuild`) skips the goal-validity preemption branch in `plan_execution_system`: a worker mid-faction-task can have their `AgentGoal` flip to `Survive` from hunger/sleep/safety spikes without the plan being aborted ‚Äî the plan ends only via completion, timeout, target invalidation, precondition failure, or external preempt (rescue, withdraw resolver fail).
  - **Explore is per-resource:** `ExploreForFood` (id 35) / `ExploreForWood` (36) / `ExploreForStone` (37) each carry `memory_target_kind: Some(...)`. The candidate filter inverts for these IDs: an Explore plan is only available when the agent has neither memory nor visibility of its target. `explore_satisfaction_system` (runs after `vision_system`, before `plan_execution_system`) removes the `ActivePlan` and logs `PlanOutcome::Success` the moment memory of the target kind is recorded ‚Äî so workers stop wandering as soon as they spot what they were looking for, instead of running the 5000-tick timeout.
  - **Tasks (`tasks.rs`):** The **current active task** an agent is performing ‚Äî `TaskKind` enum: `Idle, Gather, Trader, Raid, Defend, Planter, Hunter, Scavenge, Construct, ConstructBed, DepositResource, Socialize, Reproduce, Explore, Dig, Sleep, Eat, WithdrawFood, TameAnimal, Craft, Deconstruct, Lead, Terraform, HaulMaterials, MilitaryMove, MilitaryAttack, Play, WithdrawMaterial, WithdrawGood, PlayPlant, PlayThrow, HaulToCraftOrder, WorkOnCraftOrder, PickUpCorpse, HaulCorpse, Butcher, Equip`. Tasks are transient and managed by `plan_execution_system` (the dispatch path for every goal except `Sleep`); `goal_dispatch_system` survives only as the Sleep fallback. An agent is `Idle` when between tasks. `task_interacts_from_adjacent()` flags tasks where the agent works from an adjacent tile rather than stepping onto the target.
  - **Professions (`person.rs::Profession`):** Persistent role ‚Äî `None | Farmer | Hunter`. Farmer is auto-assigned by `faction_profession_system` when food stock falls below 100. Hunter is **chief-driven for every faction** (no longer player-only ‚Äî `HunterTargetCount` is gone). `faction_hunter_assignment_system` (faction.rs, Economy set, after `chief_hunt_order_system`) runs once every `TICKS_PER_DAY/4` (~15 game-hours) and reconciles each faction's hunter roster against a target headcount: `floor = max(1, adults * 0.20)` when `HUNTING_SPEAR` is unlocked, scaled by `martial_scale = 0.5 + martial/255` (0.5√ó‚Äì1.5√ó) and `density_scale = clamp(nearby_prey / adults, 1, 2)` (1√ó‚Äì2√ó), capped at `adults/2`. Promotes highest-Combat `None` adults; demotes lowest-Combat surplus Hunters with full teardown (`ActivePlan` + `release_reservation` + `carried_corpse` clear). Profession gating in plans uses `PlanDef.requires_profession` (candidate filter checks it before scoring).
  - **Skills (`skills.rs`):** Per-agent `[u8; 8]` array ‚Äî `Farming, Mining, Building, Trading, Combat, Crafting, Social, Medicine`. Default 5; `gain_xp()` is saturating.
  - **Bucketing:** Agents are bucketed across 20 fixed time slots (`BucketSlot`) to spread CPU load across frames.
- **Needs:** 6 tracked needs (hunger, sleep, shelter, safety, social, reproduction) clamped [0, 255] that decay over time and feed goal selection.
- **Factions:** Groups share a technology bitset (`u64`, up to 64 techs, Prehistoric ‚Üí Bronze Age), camp entity (`FactionCenter`), bonding scores, storage rollup (`compute_faction_storage_system`), and raid logic. `SOLO` faction ID = 0 means ungrouped. `StorageTileMap` indexes storage tiles per faction.
- **Construction (`construction.rs`):** `BlueprintMap`, `WallMap`, `BedMap` resources track build state. `faction_blueprint_system` decides what to build; `construction_system` consumes resources and finalizes tiles/entities.
- **Activity log (`ui/activity_log.rs`):** Bottom-right egui panel listing recent construction/crafting completions and technology discoveries. `ActivityLogEvent { tick, actor, faction_id, kind }` ‚Äî `kind` is `Constructed { site, tile, result_entity }`, `Crafted { name }`, or `TechDiscovered { tech_name, era_name }`. `construction_system` and `craft_order_system` emit with the lead worker as actor and their `FactionMember.faction_id`; `tech_discovery_system` emits with `actor: Entity::PLACEHOLDER` for the player faction only. `ResultLink` is `Built { entity, snapshot }`, `HeldByActor`, or `NoTarget` (tech rows). `activity_log_ingest_system` filters by `ev.faction_id == player_faction.faction_id` (no longer via FactionMember lookup); snapshots actor transform; capped at 16 entries. Panel shows a "‚òÖ" prefix for tech rows (no actor link), clickable actor/thing links for construction/craft rows, and a game-date stamp `"Spr D3"` reconstructed from `entry.tick / TICKS_PER_DAY` (no Calendar resource needed). Adding a new event type only requires a new `ActivityEntryKind` variant + emit site ‚Äî the panel renders `NoTarget` kinds as plain labels.
- **Hearth pacing by era (`generate_candidates` in `construction.rs`):** Era is read via `technology::current_era(techs)` (highest era of any unlocked tech). Paleolithic/Mesolithic ‚Äî `paleolithic_hearth_count` upper bound `(members+5)/6`, gated on crescent saturation across all hearths AND `bed_deficit_pre > 0`; bands fill out hearth #1's crescent before #2 opens. Neolithic ‚Äî `(members+7)/8` upper bound, gated on every existing hearth having ‚â• 8 beds in its 2..6 crescent ring (one extended-family household per hearth); helper `count_beds_in_crescent` does the count. Chalcolithic+ ‚Äî single civic-zone hearth (hearth-per-house remains future work ‚Äî requires a Household component).
- **Defensive walls are Chalcolithic+:** `generate_candidates` skips the `BuildIntent::PalisadeSegment` defense branch unless `current_era(techs) >= Chalcolithic`. Real Neolithic farming villages were typically unwalled; defensive perimeters belong to city-state society. The `best_wall_material` ladder now requires `COPPER_WORKING` for `WallMaterial::Stone` (was incorrectly gated on Paleolithic `FLINT_KNAPPING`); pre-Chalcolithic huts use `Mudbrick` (with `FIRED_POTTERY`), `WattleDaub` (with `PERM_SETTLEMENT`), or `Palisade`. The era gate only suppresses defensive perimeters ‚Äî `Hut` / `Longhouse` walls still place freely using `wall_mat`.
- **Carrying (`carry.rs`):** `Carrier` holds two hand slots, separate from `EconomicAgent.inventory`. Goods have a `Bulk` class ‚Äî `TwoHand` (Wood, Stone, Iron) requires both hands empty to pick up; `OneHand` / `Small` need one free hand. `enforce_hand_state_system` runs before gather/dig and consults `gather_target_yield_bulk` to upgrade the hand requirement to "both empty" when the harvest yield is `TwoHand` (otherwise `route_yield` would silently spill the wood/stone to ground). `is_at_haul_cap` returns `true` for any TwoHand stack with `qty > 0` since no more can be added regardless of qty. `Carrier::pickup_capacity(item)` is a read-only mirror of `try_pick_up` ‚Äî resolvers use it to size a commit before dispatch (matches the executor's bulk arms exactly: TwoHand returns 0 if any hand is occupied; OneHand fills at most one hand per call).
- **Storage withdraw routes through hands first (`production.rs::withdraw_material_task_system`):** Goods withdrawn from faction storage are offered to `Carrier::try_pick_up` first, then `EconomicAgent::add_good` for any leftover, and the GroundItem is decremented only by the amount actually accepted. Stone (5000g) weighs as much as the entire inventory cap, so without the hand-first path even a single seed in inventory caused the executor to silently destroy units. The companion fix in `resolve_withdraw_for_faction_need` (`plan/mod.rs`) sums `carrier.pickup_capacity(item) + (inv_room / unit_w)` instead of an inventory-only `.max(1)` floor ‚Äî if combined capacity is 0 the resolver returns `None` so it can pick a different tile/good.
- **Eating from hands (`production.rs::eat_task_system`):** Foraged Fruit/Meat/Grain are `Bulk::Small` and routed through `Carrier::try_pick_up` first, so freshly-gathered food usually lives in hands, not `EconomicAgent.inventory`. The eat executor and the `requires_any_edible` precondition both call `production::total_edible(agent, carrier)` and decrement from whichever store the picked edible came from (`agent.inventory[idx].1 -= 1` or `Carrier::remove_good`). Without this an agent holding fruit in hand would abort `TaskKind::Eat` immediately and starve.
- **Source vs good visibility (`STATE_DIM`=41):** Each visibility slot answers exactly one yes/no question, and source visibility never shares a slot with good visibility. Sources (slots 35-37): `SI_VIS_PLANT_FOOD` (mature edible plants), `SI_VIS_TREE` (mature trees), `SI_VIS_STONE_TILE` (Stone tiles) ‚Äî feed `Forage`/`Gather` plans, which can only act on a source. Goods (slots 38-40): `SI_VIS_GROUND_WOOD`, `SI_VIS_GROUND_STONE`, `SI_VIS_GROUND_FOOD` ‚Äî feed `Scavenge*` plans, which can only pick up loose `GroundItem`s. Counters: `count_visible_plant_food`/`count_visible_trees`/`count_visible_stone_tiles` (sources) and `count_visible_ground_*` (goods). The candidate filter in `plan_execution_system` mirrors the split: gather/farm plans gate on source-vis OR memory; plans with `PF_SCAVENGE` gate on the matching ground slot; plans with `PF_EXPLORE` invert the gate (require both source-vis AND ground-vis to be zero). The resource a Scavenge/Explore plan targets is named via the matching `PF_TARGETS_FOOD/WOOD/STONE` flag. `Deliver*ToCraftOrder` plans for wood/stone are storage-driven (not vis-driven) ‚Äî see WithdrawMaterial below.
- **Faction storage stocks (`STATE_DIM`=41, slots 29-32):** `SI_STORAGE_FOOD` (29) / `SI_STORAGE_WOOD` (30) / `SI_STORAGE_STONE` (31) / `SI_STORAGE_SEED` (32). Per-good total in the agent's faction storage from `FactionRegistry.factions[id].storage`, normalised against `STORAGE_SATURATE = 20` and clamped to [0, 1]. SOLO members and unfounded factions get zeros. Lets withdraw/haul/deliver-from-storage plans (9 `WithdrawAndEat`, 15 `DeliverFromStorageToCraftOrder`, 29 `HaulFromStorageAndBuild`, 4 `PlantFromStorage`) score on actual stockpile presence, and lets producer plans (0 `ForageFood`, 1 `FarmFood`, 2 `GatherWood`, 3 `GatherStone`, 5 `HuntFood`, 6/38/39 `Scavenge*`, 7 `BuildBlueprint`) self-throttle via a negative weight when storage is full. `compute_faction_storage_system` (faction.rs) refreshes `FactionStorage::totals` every Economy tick ‚Äî the state vector reads it without recomputing.
- **Farming plans are Farm-goal‚Äìonly:** Plans 1 (`FarmFood`) and 4 (`PlantFromStorage`) both serve `AgentGoal::Farm` exclusively ‚Äî they are not available under `Survive` or `GatherFood`. Farming is a directed job, not a spontaneous survival behaviour. `FarmFood` (steps: `FarmFarmland` ‚Üí `DepositGoods`, `memory_target_kind: Food`) harvests mature Grain when the agent has food/grain memory. `PlantFromStorage` (steps: `WithdrawSeed` (step 33, from faction storage) ‚Üí `PlantSeed` (step 4, nearest Grass tile), `memory_target_kind: None`) withdraws a Seed from storage and plants it; scored on `SI_STORAGE_SEED` so it competes only when seeds are stockpiled. Together they cover the full farming loop: plant seeds from storage when grain is scarce, harvest mature grain otherwise.
- **Plan scoring philosophy ‚Äî viability over motivation:** Inside a need-driven goal (Survive, Build, Socialize) the triggering need is constant across all candidate plans, so weighting it again is circular noise. Plan weights answer "if I picked this, would it succeed and produce value?" via inventory presence (`SI_HAS_*`), visibility (`SI_VIS_*`), memory (`SI_MEM_*`), skills (`SI_SKILL_*`), and storage stocks (`SI_STORAGE_*`). The need-slots 0-5 are still populated by `build_state_vec` for inspector readout and future cross-goal discriminators, but no plan currently weights them; they carry `#[allow(dead_code)]` in `plan/mod.rs`. Exception: `SI_WILLPOWER_DISTRESS` on play plans is the legitimate discriminator (PlayGoal triggers on social need, distress varies independently). Bias is for "this plan is the cheap-and-immediate right answer" (e.g. `EatFromInventory` bias 0.5 + `HAS_FOOD` 2.0 dominates Survive when food is in hand).
- **CraftOrder timeout:** `CraftOrder` carries `spawn_tick: u64`; `faction_craft_order_system`'s 60-tick pass despawns any order older than `CRAFT_ORDER_TIMEOUT_TICKS = 600` (~30 s at 20 Hz) and prunes `CraftOrderMap` entries pointing at already-dead entities. Workers attached to a timed-out order self-clear next tick ‚Äî `craft_order_system` resets `ai.task_id` when `order_query.get()` fails, and `plan_execution_system`'s "stale reservation" safety net releases any lingering `StorageReservations` entry. Without this sweep a stuck order (materials never arrive, station inaccessible) permanently consumes a slot of the per-faction `CRAFT_ORDERS_PER_FACTION_*` cap.
- **Craft-job posting demand signal:** `chief_job_posting_system` (`jobs.rs`) emits one `JobKind::Craft` per faction at a time, picking the recipe in `CRAFT_RECIPES` whose output good has the largest `demand - supply` (subject to tech gate and station availability ‚Äî Workbench within 12 tiles of `home_tile` for Workbench-required recipes, Loom for Loom-required, none for stationless). `resource_demand_system` (`faction.rs`) populates demand for each crafted output (Tools, Weapon, Cloth, Luxury, Shield, Armor) as a fraction of `member_count`. Without these inserts no Craft job is ever posted, no `CraftOrder` spawns, and plans 13‚Äì16 sit dormant ‚Äî preserve them when refactoring. Loom recipes pass `bench: None` because `job_claim_release_system` only validates Workbench entities; passing a Loom entity would get the posting dropped on the next release sweep.
- **WithdrawMaterial intent:** Storage tiles hold many goods at once, so the resolver ‚Äî not the executor ‚Äî decides what to take. All `TaskKind::WithdrawMaterial` steps share one `StepTarget::WithdrawForFactionNeed { need, selector }` variant that funnels through `resolve_withdraw_for_faction_need`: `need` is `Blueprint` / `CraftOrder` / `HaulClaim` (where demand comes from), `selector` is `MostDeficient` / `Specific(Good)` (which good to commit to). Steps 32 (`FetchMaterialFromStorage`, Blueprint+MostDeficient), 40 (`FetchCraftOrderMaterialFromStorage`, CraftOrder+MostDeficient), and 41 (`WithdrawClaimedHaulMaterial`, HaulClaim) parameterise the same resolver ‚Äî adding a new "fetch X for Y" is just a new `StepDef` with different params. Steps 46/47 and the per-good plans 11/12 (`DeliverWoodToCraftOrder`, `DeliverStoneToCraftOrder`) are gone; plan 15 (`DeliverFromStorageToCraftOrder`) handles every material because step 40's `MostDeficient` selector picks whichever good open orders need most.
- **Storage reservations (`StorageReservations` resource):** Tile-scoped `(tile, good) ‚Üí reserved_qty` map. Each successful `WithdrawMaterial` dispatch increments the entry by the committed qty and stashes `(reserved_tile, reserved_good, reserved_qty)` on `PersonAI`; `release_reservation` decrements and clears the fields on every teardown path (executor exit, plan abort/preempt, drafted/playerorder swap). The resolver subtracts reservations from raw `GroundItem.qty` when computing effective stock, so two agents in the same tick can no longer commit to the same one-unit stack ‚Äî the second resolver sees zero effective stock and either picks another tile/good or returns `None`. Wrapped in a `Mutex` because `plan_execution_system` runs per-agent in parallel.
- **WithdrawMaterial executor reliability:** `withdraw_material_task_system` (production.rs) on entry drops any held stack whose good doesn't match `withdraw_good` (using `spawn_or_merge_ground_item` at the agent's current tile, not the storage tile) so the worker arrives with hands aligned for the deposit step. After the spatial scan it computes `taken = promised - remaining`; if `taken == 0` (the reserved stack was emptied between dispatch and arrival) the executor pushes `PlanOutcome::FailedNoTarget` to `PlanHistory` and removes `ActivePlan`, preventing the silent "haul-with-empty-hands" trip the next `HaulTo*` step would otherwise produce. Every exit path calls `release_reservation`.
- **Scavenge plans:** `ScavengeFood` (plan 6), `ScavengeWood` (plan 38), `ScavengeStone` (plan 39) all share the shape `[Collect*, DepositGoods]` ‚Äî walk to the nearest matching `GroundItem`, pick it up, deposit at faction storage. They clean up loose materials left by `harvest_ground_drops`, prior spills, and combat drops. `ScavengeWood`/`Stone` score on `SI_VIS_GROUND_WOOD`/`SI_VIS_GROUND_STONE` (weight 1.5) and outscore their `Gather*` siblings only when ground litter is actually present.
- **Loose rock worldgen (`chunk_streaming.rs::spawn_chunk_loose_rocks`):** Called in `chunk_streaming_system` alongside `spawn_chunk_plants` whenever a chunk is first loaded. Scatters `Good::Stone` `GroundItem` stacks (qty 1‚Äì3) across ~35% of `TileKind::Stone` surface tiles using the same deterministic hash pattern as plants (`ROCK_HASH_SEED = 0xDEAD_C0DE`). No `SpatialIndex` needed at spawn time ‚Äî `update_spatial_index_system` picks them up the next frame. Rocks are immediately scavengeable by `ScavengeStone` (plan 39).
- **GroundItem sprites (`entity_sprites.rs::spawn_ground_item_sprites`):** Reactive Update system ‚Äî attaches a `GroundItemVisual` child sprite to newly spawned `GroundItem` entities. Currently handles `Good::Stone` ‚Üí sprite key `"resource_loose_rock"` (a 16√ó16 two-rock pixel-art design in `sprite_library.rs`). Uses `FogPersistent` so rocks remain dimly visible in explored tiles. Other goods remain invisible for now; add a match arm to extend coverage.
- **Terrain deformation (`dig.rs`):** `dig_system` mines surface or wall tiles, yields stone, awards Mining XP, and emits `TileChangedEvent`. Combined with `CameraViewZ`, the player can view and dig underground.
- **LOD:** Entities have `Detail / Aggregate / Dormant` levels by camera distance. Dormant entities skip simulation entirely (every per-agent system checks this first).
- **Memory & gossip:** Agents store known locations and agent sightings; share them through `PlanRegistry` gossip with `u8` freshness decay (`memory.rs`, `plan::plan_gossip_system`).
- **Hunting pipeline (`corpse.rs` + plans 5/64/65):** Animals (Wolf/Deer) no longer drop `Meat`/`Skin` on death ‚Äî `combat.rs::death_system` strips their AI/needs/species markers and inserts a `Corpse { species, fresh_until_tick }` component, indexed in `CorpseMap`. `corpse_decay_system` (Economy) despawns expired corpses (`CORPSE_FRESHNESS_TICKS = 600`).
  - **Chief hunt orders (`faction.rs::HuntOrder` on `FactionData::hunt_order`):** Per-faction directive set by `chief_hunt_order_system` once per game-day (`TICKS_PER_DAY = 3600`, staggered across the day by `fid`). The chief scans `SpatialIndex` within `HUNT_SCAN_RADIUS = 40` of `home_tile` for living Wolves/Deer, picks the species with highest count, and posts `HuntOrder::Hunt { species, area_tile = centroid, target_party_size = 4 if Wolf else 2, mustered, deployed_tick, posted_tick }`. With no prey nearby, posts `HuntOrder::Scout`. A mid-day invalidation sweep (`TICKS_PER_DAY/4` cadence, offset phase) clears stale orders that never deployed within `HUNT_PARTY_TIMEOUT = TICKS_PER_DAY/4` or whose target area has been emptied. The chief also writes `nearby_prey_count` for the headcount system to read.
  - **Plan 5 `HuntFood`** is hunter-only (`requires_profession: Hunter`, `tech_gate: HUNTING_SPEAR`, `PF_UNINTERRUPTIBLE`), gated in the candidate filter on `hunt_order == Some(Hunt(_))`. Steps: 57 (`MusterAtHearth` task=`HuntPartyMuster`, walks to nearest hearth to the chief's `area_tile` via `StepTarget::HearthForHunt`, then `wait_for_party_task_system` registers the hunter into `mustered` and blocks until `mustered.len() >= target_party_size` or `deployed_tick` is set ‚Äî first hunter to cross the threshold writes `deployed_tick`, releasing all waiters) ‚Üí 58 (`TravelToHuntArea` task=Explore, target=`HuntArea`) ‚Üí 5 (Hunt prey) ‚Üí 53 (PickUpCorpse, sets `PersonAI.carried_corpse`) ‚Üí 54 (HaulCorpse to nearest hearth via `CampfireMap`, fallback `FactionCenter`) ‚Üí 55 (Butcher, work_progress builds to `BUTCHER_TICKS = 60`, then drops `species_yield()` Meat+Skin at agent's tile, despawns corpse, awards Crafting XP, emits `ActivityLogEvent::Crafted { name: "Butchered Wolf/Deer" }`).
  - **Plan 65 `ScoutForPrey`** is hunter-only and gated on `hunt_order == Some(Scout(_))`. Single step 59 (task=Explore, target=`StepTarget::ScoutForPrey` ‚Äî random reachable tile within ¬±96 of home, like `ExploreTile`); `vision_system` writes `MemoryKind::Prey` along the way, which the chief picks up at the next decision boundary to flip to `Hunt`. Not `PF_UNINTERRUPTIBLE` ‚Äî survival pressures can preempt scouting.
  - **Plan 64 `AcquireHuntingSpear`** is uncoupled from `hunt_order` so a fresh hunter can arm before the next muster. Chains step 52 (`WithdrawGood(Good::Weapon)` from storage) ‚Üí step 56 (`Equip` MainHand); plan self-deselects via `StepPreconditions::forbids_good(Good::Weapon)`, which also counts equipped slots.
  - When no `hunt_order` is active, hunters fall through to the normal plan competition (gather, haul, scavenge) and stay flexible labour.
  - `corpse_follow_system` (Sequential, after `movement_system`) snaps the corpse Transform to the carrier's translation each tick. `respond_to_distress_system` (sound.rs) augments candidate gathering: any `Profession::Hunter` in the victim's faction within `HUNTER_RESPOND_RANGE = 50` tiles is recruited regardless of LOS/audibility, and drops `carried_corpse` so combat takes priority.
- **Equipment unification (`economy/item.rs`, `simulation/items.rs`, `simulation/combat.rs`):** Combat-relevant stats live on `Item` itself (`weapon_stats: Option<WeaponStats>`, `armor_stats: Option<ArmorStats>`), populated by `Item::new_manufactured` from `(good, material, quality)`. `Equipment::items` is `HashMap<EquipmentSlot, Item>` (value-typed ‚Äî no entity layer); combat reads `eq.items.get(&MainHand).and_then(|it| it.weapon_stats)` and per-body-part armor reduction from any covering slot. `ArmorStats::covered_parts` is a 4-bit bitmask over `armor_coverage::{HEAD, TORSO, ARMS, LEGS}` so `Item` stays `Copy`; `BodyPart::coverage_bit()` does the lookup. The new `TaskKind::Equip` + `StepDef` 56 (`EquipMainHandWeapon`, `StepTarget::EquipItem { slot, good }`) drives `equip_task_system` (Sequential, after `item_pickup_system`, before `plan_execution_system`): an instant in-place transfer that pulls the highest-multiplier matching `Item` from inventory (preferred) or `Carrier`, places it in the slot, and bounces any displaced occupant back to inventory or to ground via `spawn_or_merge_ground_item_full`. Slot is shipped through PersonAI as `equip_slot: u8` (sentinel `EQUIP_SLOT_NONE = 0xFF`) and the good through the existing `craft_recipe_id`. `StepPreconditions::forbids_good` now also checks `Equipment::has_good`, so `AcquireHuntingSpear` self-deselects after the equip step. `death_system` spills equipped items as `GroundItem`s alongside inventory + hand stacks, preserving stats so a looter re-equips a real weapon. Manufactured stats survive a storage round-trip: `spawn_or_merge_ground_item_full` (used by `faction.rs` deposit and `equip_task_system` overflow) merges by full `Item` equality (not by good), and `withdraw_good_task_system` calls `agent.add_item(gi.item, 1)` rather than reconstructing as a commodity. The original `spawn_or_merge_ground_item(good, qty)` is kept as a wrapper for combat drops, scavenge spills, and other commodity paths.
- **Muster button (`military.rs`):** HUD "Muster" button sets `MusterHuntersRequest.pending = true`; `apply_muster_hunters_system` (Economy) consumes the pulse and inserts `Drafted` on every player-faction `Profession::Hunter`, clearing `ActivePlan`/reservations/carried corpse and resetting `PersonAI` to Idle. The player issues a rally point separately via right-click on a tile ‚Äî that path already registers `HotspotKind::RallyPoint` and assigns `MilitaryMove`, so muster is the "go to military mode" half and right-click is the "go here" half. Orthogonal to the chief's hunting muster (which uses `HuntOrder::Hunt::mustered` on `FactionData`, not the `Drafted` component) ‚Äî the player can still mobilize hunters for combat without disturbing chief-issued hunt orders.
- **Right-click context menu (`ui/orders.rs`):** Three-section popup for non-drafted player-faction workers. Section 1 ‚Äî tile actions (Move, Mine, Gather, Dig Down, Deconstruct, Build ‚ñ∏). Section 2 ‚Äî entities on the tile (`‚îÄ‚îÄ Entities ‚îÄ‚îÄ`): name + health label, `[Attack]` button for non-friendly entities with health, `[Pick up corpse]` for `Corpse` entities. Section 3 ‚Äî ground item stacks (`‚îÄ‚îÄ Items ‚îÄ‚îÄ`): per-stack `[Pick up: Nx Good]` button. Sections 2 & 3 are shown only when relevant entities/items exist. Populated each right-click by iterating `SpatialIndex` at the target tile; `TileDisplayQueries` and `RoutingResources` are `SystemParam` bundles used to stay under Bevy's 16-parameter limit. `PlayerOrderKind` has three entity-targeted variants: `PickUpItem(Entity)` (routes to `TaskKind::Scavenge`, also sets `TargetItem`), `AttackEntity(Entity)` (routes to `TaskKind::MilitaryAttack`), `PickUpCorpse(Entity)` (routes to `TaskKind::PickUpCorpse`).
- **Inspector inventory management (`ui/inspector.rs`):** The inspector panel has two new sections. "Currency & Inventory" section now shows `[Drop 1]`/`[Drop All]` buttons per inventory item and `[Drop]` per hand slot. "Equipment" section (default open) shows all 6 `EquipmentSlot`s with `[Unequip]` buttons for occupied slots and a "Equip from inventory/hands" subsection listing equippable items with per-slot `[Item ‚Üí SlotName]` buttons. Actions are queued via `PendingInspectorAction` resource (decouples read-only panel rendering from component mutation) and executed by `inspector_action_system` (runs in Update after `inspector_panel_system`). `valid_equip_slots(good)` in `items.rs` maps goods to their valid slots.

### World and rendering

- **Terrain noise (`terrain.rs::surface_v`):** 4-octave FBM ‚Äî continental macro layer at freq 0.005 (weight 0.35), base octave at 0.02 (0.40), then 2√ó and 4√ó harmonics (0.18, 0.07). The result is reshaped via a signed power curve `sign(c) * |c|^0.65` (centered around 0.5) so peaks/basins push toward Z extremes instead of clustering at mid-elevation. Lowering the base frequency or the power exponent makes features bigger / more dramatic respectively.
- **Globe noise (`globe.rs::generate_globe`):** Elevation uses 4 octaves: continental at 0.012 (weight 0.30), then 0.03/0.06/0.12 (0.42, 0.20, 0.08). Rainfall uses 2 octaves at 0.03 / 0.09 (0.70 / 0.30). Halved frequencies (vs the older 0.06 base) give continent-scale biome patches across the 64√ó32 globe grid.
- **Cached globe (`world.bin`):** `globe.rs::load_or_generate` deserializes this file if present and skips regeneration. Delete `world.bin` after changing globe-level noise to see the effect; tile-level (`terrain.rs`) changes are always live.
- **`TileChangedEvent` pipeline:** Mutations to `ChunkMap` emit `TileChangedEvent`; `refresh_changed_tiles_system` (PostUpdate) rebuilds the affected tile sprites. Use this whenever code edits a tile in place.
- **`CameraViewZ`:** Player-controlled view Z-level ‚Äî defaults to `i32::MAX` (surface). Lower it to peer underground; `update_tile_z_view_system` re-skins all tile sprites accordingly.
- **`sprite_library.rs`:** Procedural pixel-art sprites built from a 32-color warm earth-tone palette via `ascii_to_image` + char-substitution templates. Loaded at `Startup` by `setup_sprite_library`. New sprites should reuse the palette and substitution helpers rather than introducing new color systems.
- **PNG textures** in `assets/textures/` (e.g., `gatherer_s_a.png`, `wolf_anim_s_a.png`) are toggled via `entity_sprites::toggle_art_mode` as an alternative to the procedural sprites.

### Geology

- **Surface vs subsurface:** `terrain.rs::surface_kind_fn` chooses the surface tile (Water/Grass/Farmland/Forest/Stone) by biome thresholds. Below the surface, `proc_tile` runs a layered model: cave noise first, then biome-thick **topsoil** (Dirt), then **ore vein lookup**, falling back to bedrock (`TileKind::Wall`). Cave cavities take precedence over ore ‚Äî anything carved by cave noise drops to Air/Dirt and the would-be ore is lost.
- **Topsoil depth (`topsoil_depth(biome)`):** Mountain 1, Desert/Tundra 2, Grassland 3, Taiga/Tropical/Temperate 4, Ocean 0. Mountains expose bedrock and ore almost immediately; forests have a deep dirt cap.
- **Ore veins (`ORE_BANDS` in terrain.rs):** Six 3D Perlin fields seeded `WORLD_SEED + 2..=7`, one per ore. Each band has a depth range (tiles below surface), threshold, and frequency. Ordered shallow ‚Üí deep: Coal (1..6, 0.45), Copper (2..8, 0.50), Tin (5..12, 0.55), Iron (6..14, 0.52), Silver (10..18, 0.60), Gold (14..32, 0.65). Higher threshold = rarer. Bands skip the noise sample entirely when depth is out-of-range, keeping subsurface lookups cheap. Ore is encoded as `TileKind::Ore` + `TileData.ore: u8` (`OreKind` enum); `Wall` and `Stone` carry `OreKind::None`.
- **Two tile-read paths (footgun):** `chunk.rs::tile_at_local` (cache-only) returns `Wall` for any uncarved subsurface tile ‚Äî passability/LOS-correct but **material-wrong**. For ore-accurate reads (mining yields, inspector display) call `terrain::tile_at_3d` instead. `carve_tile` does this internally to compute per-block drops.
- **Mining yields (`carve.rs::carve_tile`):** Returns `Vec<(Good, u32)>` ‚Äî per-block drops based on the actual tile read via `tile_at_3d`. `Wall|Stone ‚Üí (Stone, 2)`, `Ore ‚Üí (ore_yield_good(ore), 2)` where `ore_yield_good` maps `OreKind ‚Üí Good` (Coal/Iron/Copper/Tin/Gold/Silver). Floor is still rewritten to `Dirt` after carving for walkability. `gather.rs` and `dig.rs` route the drops through `route_yield`/`Carrier::try_pick_up`; `gather.rs` credits the matching `ActivityKind::*Mining` per drop and applies faction multipliers per-good.
- **`StoneProfile.bonus_yields` removed:** Surface Stone tiles no longer roll random Coal/Iron drops; ore is now strictly geological. Players (and the AI) must dig down to find ore. `COPPER_WORKING` triggers on `CopperMining`; `TIN_PROSPECTING` triggers on `TinMining` ‚Äî both also still trigger on `StoneMining`.
- **`ActivityKind`:** `ACTIVITY_COUNT = 13`. New variants: `CopperMining = 9, TinMining = 10, GoldMining = 11, SilverMining = 12`. Update `ACTIVITY_KINDS`/`ACTIVITY_NAMES` (debug_panel.rs) and `activity_name` (tech_panel.rs) when adding more.
- **Rendering:** `TileMaterials.materials` and `FogTileMaterials.materials` are keyed by `(TileKind, OreKind, z_bucket)`. Non-ore tiles use `OreKind::None`; ore tiles fan out into per-ore color handles via `RENDERABLE_ORES` in `chunk_streaming.rs`. `color_map.rs::ore_tile_color` defines the per-ore sRGB. `resolve_render_tile` returns `(TileKind, OreKind, z, Visibility)`.
- **No save-format change:** `WorldGen` is reconstructed each launch from `WORLD_SEED` constants; chunk caches still hold only surface data; `world.bin` (`globe.rs`) is the globe biome grid only. Determinism is preserved by per-ore seed offsets.

### Spatial and tile conventions

- World tiles: `(i16, i16)` pairs; convert with `tile_to_world()` / inverse.
- Chunk coords: `ChunkCoord::from_world()` using `div_euclid()`.
- Z-levels: `i8`, range `Z_MIN` (-16) to `Z_MAX` (+15).
- Fixed update loop: **20 Hz** (`Time::<Fixed>::from_hz(20.0)` in `main.rs`).
- After mutating tiles, write `TileChangedEvent { tile: (x, y) }` so the renderer refreshes.

## Constraints

- **ECS discipline:** All logic lives in Systems; Components hold data only. No object-oriented inheritance.
- **UI:** Use `bevy_egui` for all panels. Avoid `bevy_ui` except for specific rendering overlays.
- **Hashing/randomness:** Use `ahash::AHashMap` (not `std::HashMap`). Use `fastrand` in hot paths, `rand` for initialization.
- **No new crates** without explicit user permission.
- **Error handling:** Avoid `unwrap()` in core systems; use `match` / `if let`.
- **Mutable aliasing:** Be mindful of Bevy query mutable aliasing ‚Äî test empirically when touching systems with overlapping component queries.
